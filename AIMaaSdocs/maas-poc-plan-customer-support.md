> 用途：面向「智能客服」场景，在 6–8 周内验证基于阿里云 Qwen + 百炼（Model Studio）的 AI 客服方案在业务指标、体验与工程可行性上的价值，可直接用作对外 POC 计划文档骨架。

## 1. POC 目标与范围

- 业务目标（可度量）：
  - 在选定业务线（如售后客服 / 会员服务）的 Web/App 渠道上，将自助解决率从约 30–40% 提升至 50–60%；
  - 在不增加一线人力的前提下，将平均处理时长（AHT）降低 10–20%；
  - 将质检覆盖率从 < 3% 提升到 ≥ 50% 自动质检覆盖，并将严重风险话术漏检率控制在 < 5%；
  - 提升用户满意度（CSAT）或「问题一次解决率」约 5–10 个百分点。
- 技术目标：
  - 验证在现有客服系统 / 工单系统 / CRM / 订单系统基础上，通过阿里云 Model Studio 能否以可接受的延迟（P95 < 2.5s）和成本稳定运行；
  - 验证 Qwen + Qwen Embedding 与现有 FAQ/文档/工单数据结合的效果，评估不同模型规格（如 Qwen3.5-Omni vs 经济型）的性价比差异；
  - 验证通过百炼知识库、插件和应用开发能力能否覆盖 FAQ 自助问答、订单查询、升级单生成和自动质检 4 类核心能力。
- 范围与不做的事：
  - 本次 POC 仅覆盖 1–2 个业务线、1–2 个主要渠道（建议先选 Web + App），不追求全渠道上线；
  - 不在生产环境直接执行高风险写操作（如退款、关单等），相关场景只演示「生成操作建议 / 工单草稿」；
  - 不做大规模历史数据治理，只针对 POC 所需的有限文档和代表性对话进行整理与导入。

## 2. 场景与用例选择

- 从现有客服场景文档与历史数据中选出 3–5 个高价值、高频用例：
  - 必须完成用例：
    - U1：FAQ / 业务规则自助问答（如「积分规则」「活动政策」「物流时效」等）；
    - U2：订单 / 账号 / 设备状态查询（如「订单进度」「物流状态」「设备在线状态」）；
    - U3：复杂问题的升级单生成与会话总结（包含标签、优先级、建议处理方案）；
  - 可选用例：
    - U4：自动质检与话术评分（识别敏感话术、流程缺失、合规风险）；
    - U5：基于历史对话为新客服生成辅导建议 / QA 知识卡片。
- 对每个用例给出输入 / 输出示例与成功标准：
  - U1 输入：用户自然语言问题 + 用户/租户基本属性；
  - U1 输出：结构化答案（结论 + 关键条款引用），成功标准：
    - 答案准确度 ≥ 85%（人工评估）；
    - 引用到正确版本的政策 / 文档；
  - U2 输出：当前状态 + 下步建议 + 关键字段解释，成功标准：
    - 工具调用成功率 ≥ 98%；
    - 无越权查看他人订单 / 账号信息；
  - U3 输出：结构化升级单（问题类型、紧急程度、影响范围、已尝试动作、推荐接手人），成功标准：
    - 升级单可直接被二线使用，无需大量补充；
  - U4 输出：每条对话的质检标签与评分，成功标准：
    - 与人工质检的结论一致率 ≥ 80%。

## 3. 技术方案概览（POC 版）

- 模型与 MaaS：
  - 主要对话模型：Qwen3.5-Omni 或同档位 Qwen3 系列模型；
  - Embedding 模型：Qwen Embedding（通用文本），用于 FAQ/政策文档/工单摘要的向量检索；
  - 如涉及多语言 / 图片理解，可在部分用例中试验多模态模型（如 Qwen-VL）。
- 工程链路（尽量简化）：
  - 接入层：从现有客服系统（或简化版 Web Demo）引流部分会话到 AI 客服；
  - 百炼层：
    - 使用 Model Studio「知识库（RAG）」托管 FAQ、政策、产品文档；
    - 使用「应用开发」配置客服应用，包含 RouterAgent / FAQAgent / TaskAgent / EscalationAgent / QCAgent 等；
    - 通过插件/组件集成工单系统、CRM、订单系统查询接口（只读为主）；
  - 业务系统：现有客服系统负责坐席界面、会话路由和人工接管，AI 结果通过消息接口回填。
- 环境：
  - 推荐部署在「测试租户 / 沙箱环境」，或在生产系统中启用「灰度租户 + 白名单用户」；
  - API 调用通过统一网关转发到阿里云 Model Studio，以便统计、限流和审计。

## 4. Eval 集与评分体系

- Eval 集构建：
  - 从历史对话中抽取 200–500 条代表性请求，覆盖 FAQ、订单查询、投诉升级等类型；
  - 人工整理期望输出（标准答案 / 标准处理步骤），剔除包含敏感隐私/合规风险的内容；
  - 针对质检用例，从历史抽样 100–200 条已被人工质检的对话，作为对照集。
- 评分方式：
  - 自动评分：
    - 规则引擎（如检查是否引用正确政策编号、是否调用了正确工具）；
    - Judge LLM（可使用较小的 Qwen 模型）对答案进行 1–5 分的维度打分（准确、完整、礼貌、合规等）；
  - 人工抽检：
    - 产品 / 业务 / 质检人员对 10–20% 样本进行人工复核，衡量 LLM 评分的可靠性；
  - 质检用例：
    - 对比自动质检与人工质检结论的一致率（通过/不通过 + 关键标签）。
- 指标示例：
  - 业务效果：
    - 任务完成率（给出的答案被判定为「可直接使用」的比例）；
    - 一次解决率（是否需要追问 / 升级）；
  - AI 质量：
    - 答案准确度、引用正确度、工具调用成功率；
    - 高风险回答（涉及退款、赔偿等）的合规率；
  - 系统指标：
    - 首 token 延迟、整体响应时间、错误率、单位请求成本。

## 5. POC 步骤与时间规划

- 阶段 1：方案细化与数据准备（1–2 周）
  - 与业务方确认 POC 业务线、渠道、目标指标；
  - 从知识库和历史对话中过滤/整理 POC 所需文档和样本数据；
  - 搭建阿里云 Model Studio 工作空间与基本网络连通（出网/专线等）。
- 阶段 2：原型搭建与离线评估（2–3 周）
  - 在 Model Studio 中创建客服应用：配置知识库、系统 Prompt、Agent 拆分与插件；
  - 与工单/订单/CRM 系统打通只读查询接口；
  - 在 Eval 集上跑一轮离线评估，调优 Prompt / 检索参数 / 模型规格；
  - 形成第一版效果报告，确认是否进入线上小流量阶段。
- 阶段 3：小流量线上验证（2–3 周）
  - 以灰度方式接入真实请求（如 5–10% 的新会话，或内部员工测试入口）；
  - 搭建 BI 看板，实时监控自助率、升级率、AHT、转人工率等指标；
  - 收集业务方与客服团队反馈，迭代话术、界面与流程；
  - 形成小结报告，评估上线价值与 ROI。

## 6. 风险与回滚策略

- 内容与合规风险：
  - 高风险用例（如补偿、赔偿、退款）的回答全部标记为「草稿建议」，不直接展示给用户或执行；
  - 对敏感话题（金融、医疗等）设置「强制转人工」规则；
  - 配置敏感词/敏感操作规则，若命中则自动降级为 FAQ 查阅或人工接管。
- 成本与性能风险：
  - 设置调用配额与限流阈值，避免异常流量导致成本失控；
  - 在高并发场景下对模型规格做 AB 测试（高性能 vs 经济型），平衡体验与成本；
  - 如延迟超标，支持退回传统 FAQ/人工流程。
- 工程与集成风险：
  - 工具集成阶段先仅支持只读接口，待 POC 结束且通过评审后再评估写操作接入；
  - 所有 Agent 行为（请求、回复、工具调用）统一记录日志，便于审计与问题排查；
- 回滚预案：
  - 客户端保留「一键关闭 AI 客服」或「只做推荐、不自动回复」的开关；
  - 可随时将灰度流量切回传统流程，保证对生产业务影响可控。
