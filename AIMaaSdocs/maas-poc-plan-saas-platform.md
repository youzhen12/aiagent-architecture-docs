> 用途：面向「多租户 B2B SaaS 平台内嵌 Agent」场景，在 6–8 周内验证基于阿里云 Qwen + 百炼（Model Studio）的 Admin/Dev/Ops 助手机制在缩短上手时间、降低工单量、提升运维洞察方面的价值。

## 1. POC 目标与范围

- 业务目标（可度量）：
  - 将新租户管理员完成核心配置（如组织结构、权限、关键业务流程）的时间缩短 20–30%；
  - 将与配置 / 集成相关的工单量在选定租户或 Pilot 群体中降低 15–25%；
  - 将平台对关键监控/计费指标的理解效率提升（运维/FinOps 团队可以通过自然语言更快获得洞察），用「诊断时间」或「分析覆盖面」来度量；
  - 在不显著增加支持团队工作量的情况下，提升租户对平台「易用性 / 支持体验」的主观评分。
- 技术目标：
  - 验证通过 Model Studio 提供的 Qwen 通用 + 代码 + Embedding 模型，能否较好理解平台配置文档、API 文档与监控数据；
  - 验证在多租户场景下，通过上下文工程（租户 ID、环境、角色）能否做到「只看本租户」的数据隔离和权限控制；
  - 验证基于百炼知识库和插件，是否能快速搭建 AdminAgent / DevAgent / OpsAgent 的原型，并与现有控制台深度集成。
- 范围与不做的事：
  - 本次 POC 仅针对 3–10 个试点租户或一个内部 Sandbox 环境，不追求全量租户覆盖；
  - 不直接在生产环境执行高风险配置写入，所有写操作仅生成「草稿变更」供管理员审核；
  - 暂不覆盖移动端 / 第三方集成工具内嵌，仅针对 Web 控制台与开发者门户。

## 2. 场景与用例选择

- 典型角色与用例：
  - 管理员（AdminAgent）：
    - U1：基于业务描述生成权限角色与菜单配置建议；
    - U2：基于现有配置诊断常见问题（如数据不可见、流程未触发等）；
  - 开发者（DevAgent）：
    - U3：根据错误日志 / 返回码 / 代码片段定位 API 调用问题，并给出示例代码与修改建议；
    - U4：基于业务集成需求生成 SDK 调用样例或接口调用序列；
  - 运维 / FinOps（OpsAgent）：
    - U5：用自然语言查询监控和计费数据（如「最近一周某租户 5xx 波动情况及可能原因」）。
- 对每个用例定义输入 / 输出与成功标准：
  - U1 输入：简要业务描述 + 现有租户配置快照；输出：建议的角色/菜单/流程草案 + 风险提示，成功标准：管理员认为「可直接基于草案微调」的比例 ≥ 70%；
  - U3 输入：错误日志片段 + API 名称/URL；输出：问题原因分析 + 修改建议 + 示例代码，成功标准：开发者按照建议能在合理时间内复现和修复问题；
  - U5 输入：自然语言查询；输出：以图表/表格 + 文本解读的报告，成功标准：运维人员认为「解释到位、可指导行动」的比例 ≥ 70%。

## 3. 技术方案概览（POC 版）

- 模型与 MaaS：
  - 通用对话模型：Qwen3.6-Plus 或同档位模型，用于业务理解与说明文案生成；
  - 代码模型：Qwen-Coder，用于代码/日志理解与示例生成；
  - Embedding：Qwen Embedding，用于产品文档、API 文档、配置说明、变更日志的向量检索。
- 工程链路（针对 POC 精简）：
  - 前端：
    - 在管理控制台与开发者门户中嵌入 Agent 面板（侧边栏或浮窗），通过统一前端 SDK 调用后端 AI 网关；
  - 后端与百炼层：
    - 使用 Model Studio 「知识库」托管平台产品手册、API 文档、错误码说明、最佳实践文章等；
    - 在「应用开发」中配置一个 SaaS Copilot 应用，内部通过角色或会话参数区分 AdminAgent / DevAgent / OpsAgent；
    - 通过插件 / 应用组件 API 连接配置中心、监控系统、计费系统、审计日志服务（以只读查询为主）；
  - 上下文工程：
    - 每次调用时注入租户 ID、环境（dev/stage/prod）、用户角色与最近配置快照，保证响应与当前上下文强相关。
- 环境：
  - 初始阶段建议在内部 Sandbox / 试点租户环境运行；
  - 如需面向真实客户，可采用「仅读 + 草稿变更」策略，确保即便 Agent 建议错误也不会直接影响生产配置。

## 4. Eval 集与评分体系

- Eval 集构建：
  - 从历史工单与支持对话中抽取 100–200 条典型配置/集成类问题；
  - 从监控与计费平台的使用记录中抽取 50–100 条典型分析需求（如排查某段时间性能异常）；
  - 整理对应的标准答案：可为已有内部知识库 / SOP / 专家给出的处理步骤。
- 评分方式：
  - 规则与自动校验：
    - 检查 Agent 生成的配置草案是否满足既定安全约束（如权限不应过宽）；
    - 检查生成的 API 调用示例是否符合最新接口规范；
  - Judge LLM：
    - 使用较小规格 Qwen 模型对回答进行维度打分（准确性、完整性、安全性、可操作性）；
  - 人工评审：
    - 由产品、实施、运维代表对部分样本进行打分和主观评语；
  - 指标示例：
    - 配置建议可用率（无需大修改即可采用的比例）；
    - 排障建议成功率（按建议操作能解决问题的比例）；
    - 监控/计费分析报告的可用率（能否提供 actionable insight）。

## 5. POC 步骤与时间规划

- 阶段 1：目标与试点租户确认（1–2 周）
  - 选择 3–10 个试点租户或内部 Sandbox 环境；
  - 明确各角色的典型任务与期望指标；
  - 梳理已有文档、工单、监控面板，规划需要导入到知识库的数据。
- 阶段 2：原型构建与离线评估（2–3 周）
  - 在 Model Studio 中创建 SaaS Copilot 应用与知识库；
  - 打通配置中心 / 日志 / 监控 / 计费等只读接口；
  - 在 Eval 集上运行一轮离线评估，优化 Prompt / Agent 分工 / 工具接口；
  - 输出第一版评估报告，确认是否进入线上试点阶段。
- 阶段 3：线上试点与反馈收集（2–3 周）
  - 在试点租户的控制台中默认开启 Copilot 面板（可允许用户选择退出）；
  - 收集使用日志（问题类型、Agent 建议是否被采纳、后续结果等）；
  - 组织试点租户管理员 / 开发者 / 运维访谈，收集主观反馈；
  - 基于数据与反馈，给出是否推进全面产品化的建议与路线。

## 6. 风险与回滚策略

- 配置与安全风险：
  - 所有写操作一律生成为「草稿变更」，需通过现有审批流与 RBAC 权限审核后才能落地；
  - 对高风险操作（如删除数据、关闭计费、修改鉴权配置）仅生成说明文档和工单，不生成可执行调用；
  - 对 Agent 调用配置中心接口设置严格的白名单与参数校验。
- 多租户隔离风险：
  - 在上下文和工具接口实现中强制校验租户 ID，避免跨租户数据访问；
  - 在 Model Studio 知识库层面采用租户分片或租户标签，以避免 embedding 混淆带来隐私风险。
- 体验与采纳风险：
  - 在前端清晰标注「AI 建议」「请管理员审核后执行」，避免误解为自动操作；
  - 对失败率较高或反馈较差的用例及时下线或改为仅提示文档链接；
- 回滚预案：
  - 控制台保留「关闭 Copilot」开关，一键切回传统文档中心 + 工单流程；
  - 若发现严重安全/数据风险，可在网关层直接禁止相关 Agent 应用调用。
