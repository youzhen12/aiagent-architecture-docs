# 端到端 MaaS POC / 测试策略模板

> **用途**：针对任意一个场景（如 SaaS 平台、客服、秒杀等），快速设计一个 4–8 周可落地的 POC 计划与评估体系。
> 
> **相关文档**：[README](readme.md) | [场景族谱](scene-landscape.md) | [POC 计划示例](maas-poc-plan-customer-support.md)

## 1. POC 目标与范围

- 业务目标（可度量）：
  - 示例：
    - 自助解决率提升 Xpp；
    - 新租户上手时间缩短 Y%；
    - 大促期间事故数减少 N 次；
    - 开发者完成标准任务的时间缩短 Z%。
- 技术目标：
  - 验证在客户当前基础设施下，能否以可接受的成本和延迟稳定运行；
  - 验证模型候选组合（如 Qwen / GPT / Claude）在该业务场景的效果差异。
- 范围与不做的事：
  - 明确 POC 不涉及哪些功能（如生产环境写操作、计费变更等）。

## 2. 场景与用例选择

- 从场景文档中选出 1–3 个高价值、高频用例：
  - 标记「必须完成」和「可选」用例；
  - 对每个用例定义输入/输出示例和成功标准。

## 3. 技术方案概览（POC 版）

- 模型与 MaaS：
  - POC 阶段采用的模型列表与路由策略；
- 工程链路：
  - 只保留 POC 所需的最小链路（例如：Prompt + RAG + 单 Agent / 双 Agent）；
- 环境：
  - POC 部署环境（测试租户 / 沙箱环境）。

## 4. Eval 集与评分体系

- Eval 集构建：
  - 从历史数据/脚本中抽取典型请求；
  - 人工标注期望输出（或使用已有标准答案）。
- 评分方式：
  - 规则/程序自动评分 + Judge LLM + 抽样人审；
- 指标：
  - 业务效果指标（任务完成率、准确率、时间节省等）；
  - AI 质量指标（正确率、一轮解决率、工具合规率等）；
  - 系统指标（延迟、错误率、成本）。

## 5. POC 步骤与时间规划

- 阶段 1：方案细化与数据准备（1–2 周）
  - 确认用例与指标；
  - 准备 Eval 集与必要的数据接入。
- 阶段 2：原型搭建与离线评估（2–3 周）
  - 搭建 Prompt / RAG / Agent 流程；
  - 在 Eval 集上离线跑一轮评估，调整 Prompt 与模型。
- 阶段 3：小流量线上验证（1–2 周）
  - 接入真实请求的小部分流量或内部用户；
  - 监控线上指标与用户反馈。

## 6. 风险与回滚策略

- 识别 POC 期间可能出现的风险（错误回答、安全问题、成本失控等）；
- 为每类风险定义回滚或降级策略（关闭某些 Agent、切换到只读模式等）。
