# 1. 背景与目标

- 场景简介：
  - 面向个人用户或超小型团队的「个人助手」，核心能力包括：个人知识库问答、待办管理、简单自动化工作流（如整理笔记、生成周报、辅助邮件撰写等）。
- 业务问题：
  - 个人信息来源分散（文档、笔记、邮件、聊天记录），难以统一检索和回顾；
  - 重复性文字工作（总结、整理、格式化）占用较多时间；
  - 不具备搭建复杂基础设施的能力，希望用尽可能少的组件获得 AI 能力。
- AIAgent 目标：
  - 提供一个统一入口，对个人知识进行检索与问答；
  - 通过简单工作流自动化部分重复性任务（如汇总日记、整理阅读摘录、生成周报或报告草案）；
  - 保持使用成本和运维成本足够低，可在本地或轻量云环境运行。
- 文档目标：
  - 说明在「个人助手」场景下的最小可用架构与建议选型；
  - 为后续从「玩具」升级为「长期可维护的个人系统」提供参考路径。

# 2. 业务与使用场景

- 主要用户与角色：
  - 单个自然人用户：作者、程序员、学生、独立开发者等；
  - 可选的小型团队：2–5 人共享部分知识和工作流。

- 关键使用场景（示例）：
  1. 个人知识库问答：
     - 将本地文档/笔记导入知识库，通过对话形式回答「某年某月的学习总结」「某个项目的设计思路」等问题。
  2. 周报与总结生成：
     - 汇总最近的任务记录、日记片段、Git 提交信息等，生成周报或项目进展总结草案。
  3. 文档/邮件草拟：
     - 根据简单要点生成邮件、博客草稿或报告结构，并允许用户反复修改。
  4. 轻量工作流：
     - 例如「每天晚 9 点：整理今日 TODO 完成情况 + 生成简短日记草稿」。

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 「轻量化本地/云端服务 + 直连云厂商模型（或本地开源模型） + 简单编排（LangGraph 或自定义脚本） + 关系型数据库/文件 + 可选向量检索」。

- 核心组件简要：
  - 用户接口：CLI、桌面/移动应用或 Web 前端；
  - AIAgent 服务：负责接收请求、管理工作流和调用模型；
  - 数据存储：本地文件/笔记系统 + SQLite/PostgreSQL + 可选的向量库；
  - 模型接入：直连云厂商（OpenAI/Claude/Qwen）或本地 Llama/Qwen 等开源模型；
  - 最小观测：结构化日志 + 可选的轻量 metrics（用于了解调用次数、成本等）。

# 4. 核心技术选型与原因

> 本节给出「个人助手场景」的推荐选型，
> `aia-agent-architecture/decisions/` 中的文档作为通用对比参考。

## 4.1 模型与推理服务

- 选型结论（推荐）：
  - 入门/轻量使用：
    - 直连一个主力模型（如 OpenAI GPT-5.3 Instant 或 Claude Sonnet），作为统一聊天/生成接口；
  - 需要更强中文能力或国内环境：
    - 选择 Qwen3.5-Plus/Max，或基于阿里云 Model Studio 等平台；
  - 高隐私/离线需求：
    - 在本地运行 Llama 3 8B/Qwen3-8B 等开源模型（通过 Ollama/vLLM）。
- 选择原因：
  - 对个人场景而言，工程复杂度和成本比「极致效果」更重要，优先选择易用/稳定的模型；
  - 可以从「一个模型」开始，未来再根据需要引入更多模型或自建小型网关。
- 相关文档：
  - `aia-agent-architecture/decisions/0003-model-and-inference-service-selection/`
  - `aia-agent-architecture/decisions/0006-foundation-models-selection/`

## 4.2 向量数据库与检索方案

- 选型结论（推荐）：
  - 第一阶段（最小实现）：
    - 不引入专门向量库，仅使用内存/简单文件索引或 SQLite 进行「伪 RAG」，理解流程为主；
  - 第二阶段：
    - 使用 SQLite/PostgreSQL + pgvector 作为统一存储（文本 + 向量）；
  - 对个人知识库非常大的情况：
    - 可引入单节点 Qdrant（Docker/本地），但不必一开始就上复杂集群。
- 选择原因：
  - 大多数个人场景数据量有限，维护专用向量数据库的成本过高；
  - pgvector 与 SQLite/PG 的组合足以支撑几十万级别文档的检索。
- 相关文档：
  - `aia-agent-architecture/decisions/0001-vector-db-selection/`

## 4.3 Agent 编排框架

- 选型结论（推荐）：
  - 入门阶段：
    - 直接在代码中编写简单的 while 循环/函数调用，理解工具调用与多轮对话机制；
  - 需要可视化/更复杂工作流时：
    - 采用 LangGraph 或类似轻量编排框架，编排 2–3 个节点的工作流（如「检索 → 总结 → 格式化输出」）；
  - 小心使用高层框架（CrewAI 等）：
    - 可以用于玩具/实验，但不适合当作长期个人生产力工具的唯一编排内核。
- 选择原因：
  - 个人场景核心目标是学习与效率，尽量避免在一开始投入过多精力到复杂编排基础设施；
  - LangGraph 提供的图式编排和 checkpoint 特性，对「定时汇总/多步生成」这类工作流非常有用。
- 相关文档：
  - `aia-agent-architecture/decisions/0002-agent-orchestration-framework-selection/`

## 4.4 可观测性与日志方案

- 选型结论（推荐）：
  - 以 `logs-only` 为主：
    - 将每次 Agent 调用的输入/输出关键信息（去敏）写入本地文件或 SQLite 表；
  - 可选：
    - 若个人搭建了自用 Prometheus/Grafana，可暴露一个简单 `/metrics`，统计调用次数、Token 数和错误数。
- 选择原因：
  - 个人场景下，复杂观测平台价值有限，日志足以用于调试与简单统计；
  - 关注点主要是「我到底调用了多少次模型」「哪些工作流最常用」。
- 相关文档：
  - `aia-agent-architecture/decisions/0004-observability-and-logging-selection/`

## 4.5 工程设计侧重点（0005）

- 提示词工程：
  - 将不同用途（知识问答/周报生成/文档草拟）的 Prompt 模板存成独立文件，并记录版本与效果；
  - 学会用示例（few-shot）与显式结构化输出（JSON/Markdown）提升可用性。
- 上下文工程：
  - 对个人文档按「项目/年份/主题」打标签，并在 RAG 检索时利用这些标签过滤；
  - 尽量将长期知识写入可版本管理的 Markdown 文件，而不是只留在聊天记录中。
- 驭控工程：
  - 所有具有破坏性的工具（删除文件、执行 shell、git push 等）只输出命令草案，由用户手动执行；
  - 对外部 API/云服务访问设置简单的限额和告警，例如超过每日预算时提醒用户。
- 相关文档：
  - `aia-agent-architecture/decisions/0005-engineering-design/`

# 5. 核心组件设计

- 用户接口：
  - CLI 或桌面应用作为统一入口，支持文本输入/输出，未来可扩展到浏览器插件或移动端；
- AIAgent 服务：
  - 简单 Web 服务或本地进程，负责管理知识库、工作流配置和模型调用；
- 数据存储：
  - 文档/笔记：本地文件系统、Obsidian/Notion 导出的 Markdown 等；
  - 索引/配置：SQLite/PostgreSQL，用于存储文档元数据、嵌入向量和工作流配置；
- 模型接入层：
  - 一个小的 `llm_client` 封装，统一调用厂商 API 或本地模型，便于未来切换实现；
- 工作流与定时任务：
  - 可使用简单的调度器（如 cron/apscheduler）定时触发工作流执行。

# 6. 关键流程说明

## 6.1 个人知识库问答流程

1. 用户在 CLI/应用中输入问题；
2. 系统从 SQLite/向量库中检索相关文档片段（可选）；
3. 将问题 + 检索结果作为上下文发送给 LLM；
4. 返回答案，同时在日志中记录使用的文档片段 ID，以支持未来调试与修正。

## 6.2 周报生成流程

1. 定时任务每周触发一次「周报生成」工作流；
2. 工作流收集过去一周的日记、任务记录和 Git 提交摘要；
3. 调用 LLM 生成周报草案（采用结构化标题/小节）；
4. 用户在编辑器中修改并最终确认，系统保存最终版本与草案版本。

# 7. 数据与存储设计

- 文档/笔记：
  - 建议使用 Markdown + Git 版本管理，定期同步到一个安全位置；
- 向量索引（可选）：
  - 小规模场景直接用 SQLite/pgvector；
- 日志与配置：
  - 简单的 SQLite 表记录调用历史、工作流运行结果与错误信息。

# 8. 非功能性要求

- 延迟：
  - 一般交互可接受 2–5 秒延迟；
- 可用性：
  - 工具可在离线/网络不稳定时退化为「仅本地文档检索」，避免完全不可用；
- 成本：
  - 建议设置每月 Token/费用预算，在接近上限时提醒用户。

# 9. 风险、边界与取舍

- 风险：
  - 模型输出错误信息或过度「自信」可能误导用户；
  - 若直连云厂商 API，需注意隐私与数据出境问题。
- 边界：
  - 不在未经用户确认的情况下执行任何文件/系统修改操作；
  - 不主动采集超出用户指定范围的个人数据。
- 取舍：
  - 更偏向简单、易理解的实现，而不是最优性能或最复杂功能；
  - 在自动化程度和可控性之间优先可控性。

# 10. 实施计划与里程碑（可选）

- 阶段 1：仅实现聊天 + 知识库导入 + 日志记录；
- 阶段 2：加入简单工作流（周报、总结）；
- 阶段 3：视需要引入本地模型或多模型支持。
