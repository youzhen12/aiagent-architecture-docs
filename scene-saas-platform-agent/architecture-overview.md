# 1. 背景与目标

- 场景简介：
  - 面向多租户 SaaS 平台的 AI 助手，覆盖控制台、API、计费、权限、配置、运维等模块。
- 业务问题：
  - 平台功能复杂，新租户上手成本高，配置链路长且容易出错；运维排障依赖专家；需求与反馈分散难以沉淀。
- AIAgent 目标：
  - 提供「产品顾问 + 配置助手 + 运维助手」的一站式能力，降低使用门槛。
  - 帮助用户理解功能、快速完成配置、排查常见问题，并将高价值反馈沉淀给平台团队。
- 文档目标：
  - 描述 SaaS 平台场景下 AIAgent 的整体架构、关键技术选型与组件职责，为后续实现提供参考。
  - 非目标：不详细扩展具体 SaaS 子系统内部实现细节。

# 2. 业务与使用场景

- 主要用户与角色：
  - 管理员（Admin）：负责租户级配置、权限与安全策略。
  - 开发者（Dev）：负责 API 集成、SDK 使用、调试与排错。
  - 运维/值班（Ops）：负责监控、告警处理、扩容与版本发布。
  - 财务/运营：关注计费、用量与结算等。

- 关键使用场景（示例）：
  1. 新租户启用：管理员描述业务目标，Agent 推荐初始配置方案（组织结构、权限策略、模块启用等）。
  2. API 集成向导：开发者粘贴错误日志或现有代码，Agent 帮助定位签名、权限、参数等问题，并生成调用示例。
  3. 告警排查：Ops 粘贴告警信息，Agent 汇总相关监控指标和近期变更，并建议排查步骤。
  4. 计费问题排查：财务/运营通过 Agent 查询账单结构、用量细节和可能异常点。

- 典型业务流程（示例）：
  - 流程 A：新租户配置向导。
  - 流程 B：API 集成调试助手。
  - 流程 C：告警分析与运维建议。

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 「多租户上下文 + 统一模型网关 + 多角色 Agent 编排 + 深度集成平台内部 API 与监控体系」。

- 架构图占位：
  - [ ] 逻辑架构图：控制台/CLI/插件 → AI 网关 → Orchestrator → 多角色 Agent → 工具层 → 平台服务/监控/计费。
  - [ ] 部署架构图：多租户环境下，Agent 组件与平台服务的部署拓扑与隔离方式。

- 核心组件清单：
  - 控制台前端 / CLI / Chat 插件。
  - AI 网关（OpenAI 兼容入口）。
  - Orchestrator（多 Agent 编排）。
  - 多角色 Agent：`AdminAgent`、`DevAgent`、`OpsAgent` 等。
  - 工具层：平台内部 API、监控系统、计费系统、权限系统等。
  - 模型与向量库：统一模型网关 + 文档/配置向量库。
  - 监控与审计：指标、日志、Trace 与操作审计系统。

# 4. 核心技术选型与原因（必填）

> 本节给出「本场景的实际选型结论与原因」，
> `aia-agent-architecture/decisions/` 下的文档仅作为通用技术对比与补充说明。

## 4.1 模型与推理服务

- 选型结论（推荐）：
  - 使用统一模型网关（OpenAI 兼容）对接多家 LLM，并按场景选择不同模型（如高推理质量模型用于复杂配置建议，小模型用于简单问答）。
- 选择原因：
  - 平台场景中模型调用频繁且角色多样，需要在成本与质量之间灵活切换模型。
  - 模型网关统一处理鉴权、限流和审计，简化各前端和 Agent 的集成。
- 相关决策文档：
  - `../aia-agent-architecture/decisions/0003-model-and-inference-service-selection/`

## 4.2 向量数据库与文档检索

- 选型结论（推荐）：
  - 使用 Qdrant 或类似向量库存储产品文档、配置指南、API 参考和 Runbook 等内容。
- 选择原因：
  - SaaS 文档通常体量较大且结构多样，需要优秀的向量检索 + 过滤能力支撑 RAG 问答与配置建议。
- 相关决策文档：
  - `../aia-agent-architecture/decisions/0001-vector-db-selection/`

## 4.3 Agent 编排框架

- 选型结论（推荐）：
  - 使用 LangGraph 进行多角色 Agent 协作建模，并结合自研 Orchestrator 约束调用顺序与权限。
- 选择原因：
  - 不同角色 Agent 之间需要协调（例如 AdminAgent 调用 OpsAgent 或 DevAgent 的子流程），图式编排便于表达和控制。
- 相关决策文档：
  - `../aia-agent-architecture/decisions/0002-agent-orchestration-framework-selection/`

## 4.4 可观测性与日志方案

- 选型结论（推荐）：
  - 沿用平台现有 Prometheus + Grafana + 日志平台体系，并通过 OpenTelemetry 打通 AI 调用链 Trace。
- 相关决策文档：
  - `../aia-agent-architecture/decisions/0004-observability-and-logging-selection/`

## 4.5 其他关键技术

- 会话与配置快照存储：
  - 在平台数据库中记录 AI 建议与最终实际配置，对比评估 AI 建议质量。

# 5. 核心组件设计

- 控制台前端 / CLI / 插件：
  - 为不同角色提供统一入口，在请求中携带租户 ID、环境标识（prod/stage）、角色信息等上下文。
- AI 网关：
  - 将前端请求转换为统一模型调用规范，接入模型网关并返回流式结果。
- Orchestrator：
  - 根据场景（配置向导、API 调试、告警分析等）路由到对应角色 Agent，并在多步骤任务中协调多个 Agent。
- 多角色 Agent：
  - `AdminAgent`：围绕租户配置、权限、策略提供建议与检查清单。
  - `DevAgent`：围绕 API 集成、SDK 使用、错误排查提供示例与建议。
  - `OpsAgent`：围绕监控、告警、扩容与发布流程提供分析与 Runbook。
- 工具层：
  - 将平台内部 API、监控系统、计费系统、权限系统等统一封装成工具，提供结构化参数与严格权限控制。
- 模型网关与向量库：
  - 向上提供统一接口，便于在不同 Agent 与场景间复用。
- 监控与审计：
  - 记录每次 AI 相关操作的上下文、工具调用、建议与用户最终采纳情况。

# 6. 关键流程说明

## 6.1 新租户配置向导

1. 管理员在控制台发起「新租户配置」向导，会话附带租户 ID、环境标识等。
2. Orchestrator 将意图路由到 `AdminAgent`。
3. `AdminAgent` 基于业务描述和默认模板，生成组织结构、权限策略和模块启用建议。
4. 建议以「配置草案 + 解释说明」形式展示，由管理员审阅后选择一键应用或手动微调。

## 6.2 API 集成调试流程

1. 开发者将错误日志、调用代码或报错信息粘贴给 `DevAgent`。
2. `DevAgent` 调用向量库检索相关 API 文档、FAQ 与示例，分析签名、权限、参数等问题。
3. 返回「问题原因推测 + 修复建议 + 示例代码」，开发者按建议修改并重试。

## 6.3 告警排查流程

1. Ops 将告警详情发送给 `OpsAgent`，或从监控系统一键触发。
2. `OpsAgent` 调用监控 API 获取相关指标、日志与近期变更记录。
3. Agent 生成「可能原因 + 影响范围 + 建议操作步骤」，并可生成自动化脚本草案供人工审阅后执行。

# 7. 数据与存储设计

- 数据类型与来源：
  - 文档与配置指南：平台产品文档、API 参考、配置最佳实践等。
  - 监控与告警数据：来自平台现有监控/日志系统。
  - 计费与用量数据：来自计费系统。
  - 会话与 AI 建议记录：用于后续分析和质量评估。

- 存储与索引：
  - 向量库：按模块/版本/环境组织文档嵌入向量，支持按租户/环境过滤。
  - 配置与审计：在平台数据库中记录 AI 建议与最终应用配置，对比分析质量。
  - 日志平台：存储详细调用与 Trace，用于排障与审计。

- 数据生命周期：
  - 会话记录和建议按合规要求设定保留时间，用于质量回顾与模型/提示词迭代。
  - 向量索引定期重建，清理过期或无效文档。

# 8. 非功能性要求

- 性能与延迟目标：
  - 一般交互（问答/建议）P95 延迟控制在 3s 内；涉及多次工具调用的复杂流程可适当放宽但需保持可用体验。
- 可用性与容错：
  - 模型或向量库异常时，降级为基于静态文档搜索或模板化回答，必要时提示转人工支持。
- 安全与合规：
  - 严格隔离不同租户的数据与操作；对任何写入类操作要求显式确认和审计记录。
- 成本与资源：
  - 对不同租户/角色的模型调用设定配额与限流策略，控制整体成本。

# 9. 风险、边界与取舍

- 风险：
  - AI 建议配置不当可能带来安全或成本风险（例如过大配额、错误权限）。
  - 在监控与告警分析中误判原因导致处理滞后。
- 边界：
  - 高风险、不可逆的操作只提供「脚本草案/操作步骤」，不自动执行。
  - 不在无明确信任边界的情况下访问跨租户数据。
- 取舍：
  - 在「自动化程度」与「安全/审计」之间倾向谨慎，优先保持可控性与可追责性。

# 10. 实施计划与里程碑（可选）

- 阶段 1：
  - 实现文档问答 + 控制台引导，不执行任何写操作。
- 阶段 2：
  - 引入只读监控/计费工具，初步支持告警分析与计费解释。
- 阶段 3：
  - 逐步引入低风险自动化操作（测试环境优先），并完善审批与审计机制。
