# 1. 背景与目标

- 场景简介：
  - 典型形态：在线工单系统、网站 IM、App 内嵌客服、语音呼叫中心等多渠道客服体系。
- 业务问题：
  - 人工客服成本高、响应质量不稳定、知识更新滞后、跨渠道体验不一致。
- AIAgent 目标：
  - 在保证合规与安全的前提下，承接 60%+ 标准咨询与问题排查，减少人工重复工作。
  - 对复杂问题进行结构化澄清与归档，辅助人工升级处理，提高问题解决质量。
  - 提升整体响应速度与用户满意度，同时降低人力与培训成本。
- 文档目标：
  - 给出智能客服场景下的整体架构方案与关键技术选型思路。
  - 说明各组件职责、数据与安全边界以及典型业务流程，方便后续实现与评估。
  - 非目标：不详细展开具体业务系统（例如某家 CRM/工单系统）的内部实现细节。

# 2. 业务与使用场景

- 主要用户与角色：
  - 最终用户：通过 Web、App、电话或 IM 渠道提出咨询/投诉/报障等请求。
  - 客服/运营人员：处理升级工单、监控 AI 效果、维护知识库和话术。
  - 运营/产品团队：分析问题类型与用户反馈，优化流程和产品。

- 关键使用场景（示例）：
  1. FAQ 问答：用户咨询账号、套餐、支付、功能说明等常见问题，Agent 基于知识库进行问答。
  2. 工单查询：用户查询当前工单处理进度、责任人或历史处理记录。
  3. 订单/设备查询：查询订单状态、账单、设备运行状态等，与内部系统打通。
  4. 升级与投诉：用户对结果不满，Agent 帮助澄清问题并生成结构化升级单转人工处理。

- 典型业务流程（示例）：
  - 用户通过任一渠道发起请求 → 接入网关统一用户身份与会话 → Orchestrator 调用 `RouterAgent` 识别意图 → 路由到 `FAQAgent` 或 `TaskAgent` 完成问答/查询/简单操作 → 如无法解决则由 `EscalationAgent` 生成升级单 → 人工客服接管处理 → 全链路事件写入监控与审计系统。

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 「多渠道接入 + OpenAI 兼容推理网关 + LangGraph 多 Agent 编排 + Qdrant RAG + 工具层集成业务系统」。

- 架构图占位：
  - [ ] 逻辑架构图：用户渠道 → 接入网关 → Orchestrator → 多 Agent → 工具层/模型网关/向量库 → 监控与审计。
  - [ ] 部署架构图：开发/测试/生产环境中各组件的部署拓扑与伸缩方式。

- 核心组件清单：
  - 渠道入口（Web 客服组件、App IM、呼叫中心等）。
  - 接入网关（认证、限流、基础路由）。
  - Orchestrator / Router（基于 LangGraph 的多 Agent 编排层）。
  - Agent 集合：`RouterAgent`、`FAQAgent`、`TaskAgent`、`EscalationAgent`。
  - 工具层：工单系统、CRM、订单系统、知识库搜索等工具。
  - 模型与推理服务：统一模型网关 + 多家 LLM/Embedding 服务。
  - 向量库 / 检索服务：用于 FAQ/文档/工单摘要的 RAG 检索。
  - 监控、日志与告警：Prometheus + Grafana + 日志平台 + OpenTelemetry Trace。

# 4. 核心技术选型与原因（必填）

> 本节给出「本场景的实际选型结论与原因」，
> `aia-agent-architecture/decisions/` 下的文档仅作为通用技术对比与补充说明。

## 4.1 模型与推理服务

- 选型结论（推荐）：
  - 通过自建 OpenAI 兼容模型网关，对接多家 LLM 提供方；在 FAQ 场景优先使用成本较低的通用模型，在复杂诊断/升级摘要场景可使用推理能力更强的模型。
- 选择原因：
  - 客服场景对对话质量、中文理解与长上下文支持有较高要求，未来可能需要切换或混用不同模型。
  - 模型网关便于集中做鉴权、限流、路由与审计，减少各服务直连多家厂商的复杂度。
- 考虑过的方案与 tradeoff：
  - 直接厂商 API：集成简单，但多厂商支持和统一治理困难。
  - 第三方统一网关：接入快，但可控性与合规性不如自建。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0003-model-and-inference-service-selection.md`
  - `aia-agent-architecture/decisions/0006-foundation-models-selection.md`

## 4.2 向量数据库与检索方案

- 选型结论（推荐）：
  - 采用 Qdrant 或等价向量数据库，存储 FAQ、产品文档和工单摘要的嵌入向量。
- 选择原因：
  - 需要「向量相似度 + 结构化过滤」（按业务线/产品/渠道）来实现高质量 RAG 问答。
  - Qdrant 在 RAG 场景生态成熟、部署较简单，适合客服知识库检索。
- 考虑过的方案与 tradeoff：
  - PostgreSQL + pgvector：结构化数据和向量共存，初期简单，但在高并发检索下性能略弱。
  - Milvus：更适合极大规模向量场景，当前客服规模下可能偏重。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0001-vector-db-selection.md`

## 4.3 Agent 编排框架

- 选型结论（推荐）：
  - 使用 LangGraph 进行多 Agent 流程建模，并在其上封装内部 Orchestrator 抽象。
- 选择原因：
  - 客服流程通常包含澄清、多轮问答、检索、工具调用与升级，图式编排更直观、便于可视化和演进。
- 考虑过的方案与 tradeoff：
  - 自研状态机：完全定制但成本高，易重复造轮子。
  - 高层 Agent 框架（CrewAI/AutoGen）：上手快，但生产可控性和集成度有限。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0002-agent-orchestration-framework-selection.md`

## 4.4 可观测性与日志方案

- 选型结论（推荐）：
  - 使用 Prometheus + Grafana + 现有日志平台 + OpenTelemetry Trace 监控端到端链路。
- 选择原因：
  - 易与现有运维体系对齐，方便统一监控 AI 与非 AI 组件。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0004-observability-and-logging-selection.md`

## 4.5 其他关键技术

- 会话存储：
  - 使用结构化存储（如 Postgres/ClickHouse）保存会话元数据、升级单、评估指标。
- 搜索与报表：
  - 可以结合全文搜索引擎（如 Elasticsearch）用于工单/会话内容离线分析。

# 5. 核心组件设计

- 渠道入口：
  - 多种前端（Web、App、呼叫中心）统一封装为消息接口，向后端发送标准化请求结构（用户 ID、渠道、语言、会话 ID 等）。
- 接入网关：
  - 负责认证、限流、基础路由，并注入 trace_id 与安全上下文。
- Orchestrator / Router：
  - 基于 LangGraph，调用 `RouterAgent` 判断意图，控制 `FAQAgent`、`TaskAgent`、`EscalationAgent` 的执行顺序与重试策略。
- Agents：
  - `RouterAgent`：意图识别与路由。
  - `FAQAgent`：通过向量检索 + 文档回答常见问题。
  - `TaskAgent`：通过工具访问工单/订单/CRM 等系统执行查询与简单操作。
  - `EscalationAgent`：生成结构化升级单和总结，供人工客服使用。
- 工具层：
  - 将业务系统封装为具备清晰参数 Schema 的工具（工单、CRM、订单、知识库检索等），保证幂等和可审计。
- 模型网关与向量库：
  - 对上提供统一模型/检索接口，对下隐藏多模型/多向量库细节。
- 监控与日志：
  - 采集模型调用、工具调用、会话结果和升级比例等指标，日志中记录 request_id/trace_id/agent 名称/工具名/知识片段 ID 等。

# 6. 关键流程说明

## 6.1 FAQ 问答流程

1. 用户通过任一渠道提出问题，接入网关生成会话与 trace_id。
2. Orchestrator 调用 `RouterAgent` 识别为 FAQ 场景，路由至 `FAQAgent`。
3. `FAQAgent` 调用向量库检索相关 FAQ/文档片段，并结合会话上下文生成回答。
4. 将答案返回给用户，同时记录使用的知识片段 ID、模型版本和置信度等信息。

## 6.2 工单查询流程

1. 用户提出「查询工单进度」等诉求。
2. `RouterAgent` 将意图路由至 `TaskAgent`。
3. `TaskAgent` 调用 `ticket_tool` 访问工单系统，获取工单当前状态与历史处理轨迹。
4. 对结果进行归纳整理，以自然语言形式返回给用户，并在日志中记录摘要与调用状态。

## 6.3 升级与总结流程

1. 多轮对话后仍未解决问题，或用户主动要求转人工。
2. Orchestrator 调用 `EscalationAgent`，汇总会话关键信息（用户诉求、排查步骤、当前状态等）。
3. 生成结构化升级单（JSON + 文本摘要），推送到人工客服系统，并打标用于后续分析与训练数据收集。

# 7. 数据与存储设计

- 数据类型与来源：
  - 知识库数据：FAQ、产品文档、公告信息、工单摘要等，来自内部知识库/Wiki/工单系统。
  - 业务数据：工单、订单、账号等数据，来自内部业务系统，通过工具层访问。
  - 会话与日志：用户输入、Agent 决策、模型与工具调用记录等。

- 存储与索引：
  - 向量库：按业务线/产品线/渠道划分 Collection，存储 FAQ/文档/摘要嵌入向量。
  - 关系型数据库：存储会话元数据、升级单、评估指标等结构化信息。
  - 日志平台：存储详细日志与 Trace，支持按 trace_id 和用户维度检索。

- 数据生命周期：
  - 会话与日志按合规要求设定保留期，并支持按 trace_id 回放。
  - 知识库与索引定期更新，清理过期或失效文档。

# 8. 非功能性要求

- 性能与延迟目标：
  - FAQ 问答 P95 延迟 1–2s；涉及外部系统查询的请求 P95 控制在 3–5s。
- 可用性与容错：
  - 模型或向量库故障时，可降级为关键词检索或直接转人工模式，避免中断服务。
- 安全与合规要求：
  - 不在模型上下文中写入完整敏感信息（身份证号、手机号、卡号等），使用脱敏/占位符表示。
  - 外部 API 访问采用白名单策略，对写操作工具增加二次确认和审计日志。
- 可观测性：
  - 监控会话完成率、自助解决率（无需转人工）、升级率、误转率、用户满意度等指标。

# 9. 风险、边界与取舍

- 已知风险：
  - 模型误判意图导致错误路由，影响用户体验或引起错误操作建议。
  - 知识库更新不及时导致回答过期或不准确。
- 边界与不做的事：
  - 不在无明确授权的情况下执行高风险不可逆操作（如删除关键资源、直接补偿等）。
  - 不尝试完全替代人工客服，对高价值/高风险场景保留人工兜底。
- 设计取舍：
  - 在体验与安全之间优先保证安全与可审计性，允许部分复杂问题直接转人工。

# 10. 实施计划与里程碑（可选）

- 阶段 1（MVP）：
  - 打通 FAQ 问答与工单查询两个闭环能力，不执行任何写操作。
  - 建立基础监控与审计链路，验证场景价值与稳定性。
- 阶段 2：
  - 增加只读工具与部分低风险写操作（如测试环境工单创建），完善多 Agent 协作策略。
- 阶段 3：
  - 与更多业务系统集成，引入投诉与升级流程自动化，强化安全隔离与审批机制。
