# 1. 背景与目标

- 场景简介：
  - 说明这是一个什么场景，例如「本地 IDE Coding Agent」「客服问答 Agent」「运营自动化 Agent」等。
- 业务问题：
  - 当前希望通过 AI Agent 解决什么问题、替代或增强哪些人工流程。
- 文档目标：
  - 用 2–3 点说明本架构文档要回答的核心问题（例如：整体方案长什么样、关键技术怎么选、有哪些风险与边界）。
- 非目标（可选）：
  - 明确本设计不打算解决/覆盖的内容，避免误解。

- 关键使用场景（建议列 3–5 个）：
  1. 场景 A：一句话概述 + 简短步骤。
  2. 场景 B：……
  3. 场景 C：……

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 例如：「OpenAI 兼容推理网关 + LangGraph 多 Agent 编排 + Qdrant RAG + 可插拔工具层」。
- 架构图占位（后续在实际场景文档中补图）：
  - [ ] 逻辑架构图：从用户入口 → 编排/Orchestrator → 各 Agent → 工具 / 模型网关 / 向量库 → 结果回传。
  - [ ] 部署架构图：各组件在不同环境中的部署方式（本地开发 / 测试环境 / 生产环境）。
- 核心组件清单（仅列出名称及一句话职责，详细说明放在后文）：
  - 用户入口（CLI / TUI / Web / API 网关 等）。
  - Orchestrator / Router（负责任务拆解与多 Agent 协作）。
  - Agent 集合（代码助手、知识问答助手、自动化流程助手等）。
  - 工具层（文件 / Git / CI / HTTP / 数据库 / 内部系统等）。
  - 模型与推理服务（LLM / Embedding / 语音 / 图像等）。
  - 向量库 / 检索服务。
  - 监控、日志与告警。

# 4. 核心技术选型与原因（必填）

> 本节是本模板的核心：
> - 对每一类关键技术（模型网关、向量库、Agent 编排框架、可观测性等），都需要写清楚：
>   1）最终选择了什么；
>   2）为什么选它（优势、契合的需求和约束）；
>   3）考虑过但放弃的方案，以及放弃原因；
>   4）未来有哪些可替代/演进方向。
> - 如有详细 ADR，请在小节末尾链接到 `docs/aia-agent-architecture/decisions/` 下的对应文档。

## 4.1 模型与推理服务

- 选型结论：
  - 示例：统一通过自建 OpenAI 兼容网关，后端对接多家云厂商 + 可选自托管推理集群。
- 选择原因：
  - 与本场景强相关的理由，例如：需要多模型切换、成本可控、对日志与审计有要求等。
- 考虑过但放弃的方案：
  - 例如：直接调用单一云厂商 API / 完全自托管推理 / 第三方推理网关，逐一说明放弃原因（成本、维护难度、扩展性、安全合规等）。
- 未来可替代/演进方向：
  - 在哪些条件变化时会考虑切换方案（例如调用量极大、私有化要求更强、模型类型增加等）。
- 相关决策文档：
  - 例如：[0003 模型与推理服务选型](../decisions/0003-model-and-inference-service-selection/)（如存在则填写）。

## 4.2 向量数据库与检索方案

- 选型结论：
  - 示例：使用 Qdrant / Postgres + pgvector / Milvus 等。
- 选择原因：
  - 结合本场景的数据量、查询模式（在线检索频率）、团队现有经验、运维成本等进行说明。
- 考虑过但放弃的方案：
  - 例如：其他云托管向量库、自研索引层等，并说明放弃的具体原因。
- 未来可替代/演进方向：
  - 如数据规模扩张或性能瓶颈出现时，可能迁移到的目标方案及迁移大致成本评估。
- 相关决策文档：
  - 例如：[0001 向量数据库选型](../decisions/0001-vector-db-selection/)。

## 4.3 Agent 编排框架

- 选型结论：
  - 示例：选用 LangGraph / 自研编排服务 / CrewAI / AutoGen 等。
- 选择原因：
  - 例如：对可视化、调试能力、复杂流程编排、多 Agent 协作能力的需求，以及与现有技术栈的契合度。
- 考虑过但放弃的方案：
  - 对比其他编排框架或「完全不使用框架，直接写代码」的方案，说明为什么不选。
- 未来可替代/演进方向：
  - 若框架升级或出现更适合的框架，切换时预期需要处理的兼容性问题（例如状态存储、Flow 表达方式等）。
- 相关决策文档：
  - 例如：[0002 Agent 编排框架选型](../decisions/0002-agent-orchestration-framework-selection/)。

## 4.4 可观测性与日志方案

- 选型结论：
  - 示例：使用 Prometheus + Grafana + 现有日志平台 / SaaS Observability 平台 / 仅日志方案 等。
- 选择原因：
  - 与本场景的稳定性要求、合规要求、团队运维能力等相关的理由。
- 考虑过但放弃的方案：
  - 例如：更重型的 APM、全 SaaS 方案、自研等，并说明性价比不足或集成难度过高等原因。
- 未来可替代/演进方向：
  - 在访问量提升或场景扩展后，需要增加哪些监控能力或调整哪些组件。
- 相关决策文档：
  - 例如：[0004 可观测性与日志方案选型](../decisions/0004-observability-and-logging-selection/)。

## 4.5 其他关键技术

- 若本场景还有其他关键技术选型（如：消息队列、中间件、配置中心、缓存、鉴权方案等），可按以下模板继续补充：
- 特别建议在本节中至少考虑以下三个工程维度（可按 4.5.1 / 4.5.2 / 4.5.3 等形式展开）：
  - Prompt & Context 架构：如何拆分和装配 System Prompt，各类 Section（身份与定位、做事哲学、风险规则、工具使用规则、Session 指引等）如何组织，以及上下文来源（RAG / memory / transcript / MCP instructions 等）如何组合与压缩。
  - 工具运行时 + 权限 + Hooks 的驭控层：工具执行链路（输入校验 → PreHook → 权限决策 → 执行 → PostHook/FailureHook → Telemetry）在本场景的设计，以及高危操作分类与权限模型如何落地。

# 5. 核心组件设计

> 本节聚焦「组件职责与边界」，不展开到代码级别。

- 组件清单：
  - 列出本场景涉及的主要组件（入口层、编排层、Agent 层、工具层、模型层、数据与存储层等）。
- 各组件职责与交互：
  - 对每个组件，用 3–5 行说明其职责、输入输出，以及与其他组件的关系。
- 依赖关系与边界：
  - 哪些组件可以独立部署/扩缩容；哪些组件之间是强耦合；哪些能力是通过接口抽象出来便于替换的。

# 6. 关键流程说明

- 选择 2–3 条最重要的业务流程，用文字 + 时序图（如需要）说明：
  - 例如：「用户提问 → RAG 检索 → Agent 规划 → 工具调用 → 最终回答」。
- 对于每条流程，说明：
  - 参与的组件 / Agent / 工具。
  - 是否有人工确认节点（例如执行高风险操作前的二次确认）。
  - 关键的错误处理与重试策略。

# 7. 数据与存储设计

- 数据类型与来源：
  - 代码、文档、日志、运行指标、用户输入等分别来自哪里，以何种形式存储。
- 存储与索引：
  - 使用了哪些数据库 / 向量库 / 对象存储，各自存什么，如何分库分表或分 Collection。
- 数据生命周期：
  - 保留周期、归档策略、删除策略，哪些数据需要长期保存以便审计。

# 8. 非功能性要求

- 性能与延迟目标：
  - 针对本场景设定具体的延迟与吞吐目标（例如：P95 响应时间、每日调用量等）。
- 可用性与容错：
  - 关键组件的可用性目标（如 99.9%），以及故障时的降级策略（例如：关闭写入工具，仅保留问答等）。
- 安全与合规：
  - 对数据安全、访问控制、审计日志、合规（如内部安全规范）的要求与实现思路。
- 成本与资源：
  - 粗略估算模型调用成本、存储成本、运维成本，并给出控制成本的策略（例如缓存、模型混用、小模型优先等）。

# 9. 风险、边界与取舍

- 已知风险：
  - 列出在本架构下已知的风险点（例如：对某单一厂商的过度依赖、某组件的单点瓶颈、对特定开源项目的依赖等）。
- 边界与不做的事：
  - 明确本方案不会做或暂不支持的能力（例如：不支持跨租户共享上下文、不支持离线模式等）。
- 设计取舍：
  - 说明在性能、安全性、复杂度、可维护性之间做过哪些权衡，并给出当前选择的理由。
