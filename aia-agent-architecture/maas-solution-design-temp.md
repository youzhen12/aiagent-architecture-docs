# 1. 客户业务目标与技术约束

- 客户业务背景：
  - 所在行业、主要产品/服务形态、目标用户画像。
- 业务目标（必须是可度量的）：
  - 例如：降低客服人力 30%、提升转化率 5pp、缩短处理时长 20%、减少告警漏报 50% 等。
- 业务边界与不做的事：
  - 明确本方案不会覆盖/不打算引入的能力，避免过度承诺。
- 技术与组织约束：
  - 合规要求（数据出境、隐私、审计）；
  - 基础设施现状（云厂商、K8s、有无向量库/日志平台）；
  - 人员能力（语言栈、MLOps/LLMOps 经验）；
  - 成本与上线时间约束。

# 2. 场景与用例拆解

- 场景定义：
  - 列出本次要覆盖的 1–3 个核心场景（最好能与本仓库 `scene-*/` 对应或复用）。
- 典型用例清单：
  - 对每个场景列出 3–5 个「用户视角」的用例，用一句话 + 简短步骤描述；
  - 标记每个用例的大致频次与业务价值（高/中/低）。
- 优先级与波次划分：
  - 哪些用例进入 v1（MVP），哪些在后续版本实现；
  - 哪些用例只做建议/辅助（人审确认），哪些可以逐步走向自动化。

# 3. 模型与 MaaS 能力选型

> 参考：`0003-model-and-inference-service-selection/`、`0006-foundation-models-selection/`

- 模型访问方式（MaaS 方案）：
  - 直连单一云厂商 / 自建 OpenAI 兼容网关 / 第三方推理网关 / 同时支持云 + 自托管模型；
  - 说明理由（成本、合规、多模型策略、运维复杂度）。
- 模型家族与规格选择：
  - 对话/推理主模型（如 GPT-5.x, Claude Sonnet, Gemini, Llama, Qwen 等）；
  - 低成本模型（用于批量任务或低价值链路）；
  - Embedding 模型（向量检索/RAG）；
  - 如需多模态（图像/语音/视频等）则单独说明。
- 多模型路由策略：
  - 不同场景/用例如何选择模型（按价值/延迟/成本分级）；
  - 是否需要 A/B 测试或灰度路由机制。

# 4. 工程链路设计：推理 / 微调 / RAG / Agent

> 参考：`0001-vector-db-selection/`、`0002-agent-orchestration-framework-selection/`、`0005-engineering-design/`

## 4.1 推理链路（Inference Pipeline）

- 请求路径：
  - 从前端/系统入口到模型网关的调用链（含鉴权、限流、日志）；
  - 同步/异步调用策略，是否使用队列/任务系统。
- Prompt & Context 架构：
  - System Prompt 的模块化设计（身份、做事规则、风险规则、工具规则、输出要求等）；
  - 会话上下文管理（截断/压缩/摘要）；
  - 与租户/用户/会话元数据的绑定方式。

## 4.2 微调 / 指令微调（如适用）

- 是否需要微调：
  - 如果需要，说明目标（领域知识 / 风格 / 工具使用稳定性等）；
  - 评估数据规模与标注方式（人工标注 / 自监督 / 蒙版重建等）。
- 微调流程：
  - 数据清洗与脱敏；
  - 训练资源与平台（自建集群 / 云厂商服务）；
  - 版本管理与回滚策略；
  - 微调模型与基座模型的路由逻辑。

## 4.3 RAG 与知识/数据接入

- 数据源与类型：
  - 文档、FAQ、知识库、业务数据库、日志/事件流等；
- 索引与存储：
  - 选用向量库（Postgres+pgvector / Qdrant / Milvus / 其他），以及 Collection 与分区设计；
  - 分片与多租户（如 SaaS 场景）策略。
- 检索与融合策略：
  - 检索时过滤条件（租户、权限、时间窗口、标签等）；
  - Retrieval + 生成（RAG）如何与 Prompt 结合（段落数量、重写策略、证据引用等）。

## 4.4 Agent 编排与工具层

- Agent 角色划分：
  - 是否采用多 Agent（例如 Explore / Plan / Execute / Verify）、如何拆分职责；
- Orchestration 框架：
  - 使用 LangGraph / Semantic Kernel / LlamaIndex Agents / 自研 orchestrator / OpenAI Assistants / DeerFlow 等何种组合；
  - 在本场景中的推荐用法（顶层编排 vs. 子流程编排）。
- 工具层与驭控（Harness）：
  - 工具清单（内部系统 API、数据库操作、搜索、发送通知等）；
  - 工具执行链路：输入校验 → Hooks → 权限决策 → 执行 → 结果记录；
  - 高危操作分类与约束策略（必须人审 / 不允许自动执行等）。

# 5. 效果评估体系与测试策略

> 参考：`0004-observability-and-logging-selection/`、`0007-agent-testing-strategy-selection/`

## 5.1 指标体系

- 业务指标：
  - 与第 1 章业务目标强绑定，如转化率、AHT、留存率、告警漏报率、用户满意度等；
- AI/模型指标：
  - 正确率、一致性、一轮解决率、工具调用合规率、多轮对话稳定性等；
- 系统指标：
  - 延迟、错误率、吞吐、资源使用、成本（每次调用成本 / 每个用例成本）。

## 5.2 测试与评估策略

- 基准集与离线评估：
  - 构建场景覆盖的 Eval 集（含 golden cases、边界用例、高风险用例）；
  - 使用规则、程序或 Judge LLM 自动打分，并辅以人审抽检。
- 回归与变更管理：
  - Prompt / 模型 / 工具 / 编排变更时如何快速跑回归；
  - 将关键评估嵌入 CI/CD（如有）。
- 在线监控与反馈闭环：
  - 结合日志与 Trace 监控线上表现；
  - 用户反馈（评分/纠错）如何回流到数据与模型迭代中。

# 6. 上线路径与部署策略

- 环境与灰度策略：
  - 区分开发 / 测试 / 预发 / 生产环境；
  - 初期仅在内部或小范围用户中灰度；
  - 明确「旁路 vs. 主链路」：是否允许 AI 直接影响生产操作，还是仅输出建议。
- 安全与合规：
  - 数据传输与存储的加密方案；
  - 日志脱敏与访问控制；
  - 合规审核与报备（如适用）。
- 运维与运营：
  - 监控告警配置；
  - 运维 Runbook 与应急预案；
  - 业务侧运营同学如何使用和优化 MaaS 能力。

# 7. 风险分析与演进路线

- 已知风险：
  - 供应商锁定、模型能力不稳定、数据/知识时效性、工具调用风险等；
- 缓解策略：
  - 多模型与多通道预案、回滚方案、人工接管机制、迭代节奏控制；
- 演进路线：
  - 从 MVP → Beta → 全量上线的大致阶段目标与里程碑。
