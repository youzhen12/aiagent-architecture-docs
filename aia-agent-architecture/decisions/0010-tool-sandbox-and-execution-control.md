# 背景

在 AIAgent 系统中，工具调用（如 shell、数据库、HTTP、内部业务 API 等）是「让模型真正产生影响」的关键环节，也是风险最高的部分：

- 一次错误的 shell 命令可能删除或泄露重要数据。
- 一次错误的业务 API 调用可能触发资金流转或敏感操作。
- 一次错误的 HTTP 请求可能造成安全问题或合规风险。

仅靠 Prompt 约束模型「不要执行危险操作」显然不够，需要在工程上设计明确的工具沙盒与执行控制机制，将模型的执行能力限制在可控范围内，并且做到：

- 可观测：谁在何时、通过何种工具、对哪些资源做了什么动作。
- 可回滚：关键路径上出现错误时有清晰的撤销或补救策略。
- 可扩展：新的工具接入时，有统一的安全/审计/限流标准。

本 ADR 用于给出本项目在「工具沙盒与执行控制」上的设计原则与选型，为具体实现（如 middleware、router、工具包装器等）提供统一约束。

# 范围

本决策主要关注：

1. 工具执行环境的隔离与权限控制：
   - 文件系统访问（读/写路径白名单）。
   - 网络访问（域名/IP 白名单、协议限制）。
   - 进程级隔离（容器 / VM / serverless 任务）。

2. 工具调用生命周期管理：
   - 调用前的参数校验与策略检查。
   - 调用中的超时、重试、并发限制。
   - 调用后的结果审计与日志记录。

3. 高风险操作的防护：
   - 人工确认 / 审批环节（human-in-the-loop）。
   - 预执行模式（dry-run）、脚本草案输出而非直接执行。

4. 与驭控工程（`0005`）的关系：
   - 如何将工具沙盒的策略纳入「驭控」体系，而不仅是零散的 if/else 判断。

不在本决策范围内的内容：

- 具体某个工具的业务协议与字段设计（例如具体 CRM API 的参数），这些应在场景或子系统文档中描述。
- 针对模型输出文本本身的安全过滤（由别的安全/内容政策模块负责）。

# 初步设计原则

1. **最小权限原则（Least Privilege）**：
   - 工具只暴露完成任务所需的最小能力，不给 Agent 任意执行系统命令或任意访问内网的能力。

2. **白名单优先**：
   - 用白名单（允许列表）而不是黑名单（禁止列表）来定义可访问资源。  
   - 如：允许访问的目录、允许访问的外部域名、允许调用的业务 API 列表等。

3. **工具包装而非裸调用**：
   - 所有外部能力通过受控的「工具适配层」暴露给 Agent，而不是直接让 Agent 构造 shell/sql/http。  
   - 适配层负责：参数校验、审计、结果标准化。

4. **高风险路径必须有人类参与**：
   - 如涉及资金、生产环境配置、用户隐私数据的操作，应采用：
     - 生成脚本/变更草案 + 人工审核执行。  
     - 或由专门的「审批 Agent + 人」共同决策。

5. **可观测性内建**：
   - 所有工具调用应产生结构化日志，至少包含：
     - 调用方（哪个 Agent / 会话 / 用户）。  
     - 工具名称与版本。  
     - 关键参数（经过脱敏）。  
     - 调用结果（成功/失败/超时）以及资源影响范围。

6. **易于演进和扩展**：
   - 新工具接入时，只需在统一的配置/策略体系下声明其能力、权限和审计规则，而无需每次在业务代码中散落逻辑。

# 与其他决策的关系

- 与 `0005-engineering-design`（驭控工程）：
  - 工具沙盒是驭控工程在「执行能力」层面的具体落地。  
  - 驭控工程定义策略与反馈回路，工具沙盒负责执行层面的硬边界与审计。

- 与 `0002-agent-orchestration-framework-selection`：
  - 编排框架需支持：
    - 为每个工具配置上下文与权限。  
    - 在流程图/状态机级别插入审批/回滚节点。  
  - 若框架内置的工具机制无法满足安全需求，需要自建一层工具网关。

- 与 `0003-model-and-inference-service-selection`：
  - 部分推理服务支持内置「工具调用」协议（如 function calling / tool calling），在使用这些能力时，需要在外侧仍然有一层沙盒策略控制。

# 后续工作

本 ADR 只是高层设计，后续需要：

1. 在子目录 `0010-tool-sandbox-and-execution-control/` 中补充：
   - 针对 shell/文件系统、HTTP/外网访问、数据库/内部服务的具体沙盒实现案例。  
   - 与现有中间件（如 `internal/pkg/middleware`）和路由器的集成方式。  
2. 在实际项目中，优先为「高风险工具」设计严格的沙盒策略和审批流程，记录为场景级文档的一部分。  
3. 将工具沙盒的策略（白名单、限流、审计规则）抽象为配置或策略语言，避免逻辑散落在代码中难以治理。
