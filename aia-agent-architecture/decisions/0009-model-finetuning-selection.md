# 背景

在 AIAgent 系统中，适配「基础模型」与「具体业务」通常有几条路线：

1. 仅靠 Prompt / 系统指令约束行为。
2. 通过 RAG（检索增强生成）接入外部知识库。
3. 对基础模型进行不同程度的微调（fine-tuning / PEFT）。

随着开源与商业大模型的发展，微调相关技术和产品形态也在快速演进：

- 全量微调（full fine-tuning）：直接优化模型全部或大部分参数。
- 参数高效微调（PEFT）：如 LoRA / QLoRA / Prefix Tuning / Adapter 等，只训练少量附加参数。
- 厂商托管微调服务：如各云厂商提供的 API 级微调能力（多为 LoRA/Adapter 或轻量全参微调封装），屏蔽底层训练细节与基础设施。

本 ADR 用于：

- 定义在本项目中 **何时需要微调、优先选择哪种微调路线**。  
- 说明微调与 RAG、Prompt 工程之间的边界。  
- 指导各具体场景在 `scene-*/architecture-overview.md` 中做微调相关选型。

> 详细技术背景与术语参考：
> - Hugging Face PEFT 文档（介绍 LoRA/QLoRA 等参数高效微调方法）。
> - 各主流模型厂商的微调文档（OpenAI、Anthropic、Qwen 等）。

# 备选方案概览

本项目中，我们将微调相关策略大致拆为四类方案：

1. **仅用 Prompt + RAG，不做微调**  
2. **厂商托管微调（API 级）**  
3. **PEFT 微调（LoRA/QLoRA 等），在自托管或专用集群中进行**  
4. **全量微调（Full FT），仅在极少数高价值闭环场景考虑**

下文从「什么时候用」「用在哪儿」「怎么评估」三个角度展开。

# 决策结论（TL;DR）

1. **默认优先级**：Prompt → RAG → 厂商托管微调 → 自托管 PEFT → 全量微调。  
2. **不将微调作为通用第一选择**：
   - 对大多数企业场景，**高质量 RAG + Harness + Eval** 比「盲目微调」更有收益也更可控。  
   - 只有在明确识别到「基础模型+RAG 无法满足需求」且有足够数据与评估体系时，才考虑微调。
3. **微调的典型触发条件**：
   - 需要模型学习**稳定的风格 / 格式 / 操作规范**（如特定 DSL 输出、固定 API 合约、强结构化 JSON）。
   - 需要在**高频重复任务**中显著降低 Prompt 复杂度与 Token 成本。  
   - 在已有 Eval 集上，经过 Prompt/RAG 优化后依然存在明显「系统性误差」。
4. **选择哪一种微调方式**：
   - 若使用的是云厂商托管模型，**优先考虑厂商托管微调**：
     - 降低基础设施与运维成本。  
     - 适合中等规模数据（千级–万级样本）。
   - 若已有自托管模型和训练基础设施，且需要更强控制：
     - 优先使用 **PEFT（LoRA/QLoRA 等）**，将新增参数与基础模型隔离，便于回滚与多版本管理。  
   - **全量微调仅在以下情况考虑**：
     - 有大量高质量标注数据。  
     - 有稳定的训练/评估/回滚流程与资源。  
     - 业务收益足以覆盖高成本与风险。

# 何时用微调 vs RAG vs Prompt

## 1. 倾向使用 Prompt / RAG 的场景

1. 知识频繁变化、需要可视化运维的场景
   - 如产品 FAQ、政策条款、内部知识库问答等：
     - 知识内容变更较快。  
     - 需要非技术人员也能维护内容。  
   - 这类场景不适合用微调「固化」知识，RAG 更易于更新与审计。

2. 只需要「风格柔性控制」的场景
   - 如文案润色、翻译、一般性客服话术等：
     - 可以通过少量示例和清晰 Prompt 达到可接受效果。  
   - 优先通过 Prompt 模板 + 少量示例（in-context learning）解决。

3. 短期验证或 PoC 阶段
   - 在场景尚未稳定、需求和约束可能频繁变化时：
     - 不建议过早投入微调工作。  
     - 用 Prompt 和 RAG 快速迭代，以验证价值与可行性为主。

## 2. 倾向使用微调的场景

1. 强结构输出 / 严格格式约束
   - 例如：
     - 生成严格遵守特定 Schema 的 JSON（错误会直接导致调用失败）。  
     - 生成 DSL / 配置语言 / 查询语言（如特定业务 DSL、SQL 变种）。
   - 若在充分利用工具函数调用、JSON 模式、系统指令等手段后，仍然出现大量格式错误，可以考虑微调模型以内化这些约束。

2. 高频重复任务中的成本优化
   - 对于某些高 QPS 场景（如大规模客服、批量文档处理），若：
     - 需要用长 Prompt 才能获得理想输出。  
     - 且逻辑相对稳定。  
   - 可以通过微调将部分 Prompt 逻辑「固化」到模型权重中，降低每次调用的 Token 成本与延迟。

3. 领域特定风格与术语统一
   - 如法律、医疗、金融等领域：
     - 需要保持统一术语、格式、语气。  
     - 需要模型更好理解某些领域特定表达（缩略语、行话）。
   - 在有足够领域语料时，可通过轻量微调（PEFT 或厂商托管微调）增强模型在该领域的一致性与稳定性。

4. 已经有稳定 Eval 与数据闭环的场景
   - 当某个 Agent 流程已经有：
     - 明确的输入/输出规范。  
     - 稳定的 Eval 集与线上监控指标。  
   - 可以考虑通过持续微调来「编码」这些反馈，将部分纠偏能力移入模型本身。

# 微调路线与选型

## 1. 厂商托管微调（API 级）

### 特点

- 微调流程由模型提供方托管，通常支持：
  - 上传训练数据集（JSONL 等）。  
  - 配置少量训练参数（epoch、learning rate、训练数据权重等）。  
  - 通过统一 API 调用微调后的模型版本。  
- 内部实现多为 LoRA / Adapter 或轻量全参微调，具体细节对用户半透明。

### 适用场景

- 主要使用云厂商模型（如某家大模型作为主力），不打算自建训练基础设施。  
- 数据量在「千到万」级别，且多为高质量指令-输出样本。  
- 对训练过程的可控性要求不高，主要关注最终表现与稳定性。

### 优点

- 工程门槛低，运维成本小。  
- 与厂商现有 Eval / 监控体系易集成。  
- 部署路径简单：多数情况下只需替换模型名称或版本参数。

### 局限

- 训练细节与权重形态不透明，难以跨厂商迁移。  
- 需要信任厂商的数据处理与隐私策略。  
- 价格模型与调用成本需结合具体厂商评估。

## 2. PEFT 微调（LoRA/QLoRA 等）

### 特点

- 只训练少量附加参数（如若干低秩矩阵），基础模型权重保持冻结。  
- 新参数可以作为独立「适配层」挂载：
  - 便于按场景/租户加载不同适配器。  
  - 便于回滚和多版本管理。  
- QLoRA 等技术可以在低精度（如 4bit）基础上进行训练，提高训练效率。

### 适用场景

- 使用开源模型或自托管模型，且：
  - 有一定 GPU/加速资源。  
  - 希望精细控制训练细节与部署形态。  
- 需要在同一基础模型上支撑多个场景/客户，每个有自己微调权重（多适配器模式）。

### 优点

- 训练与部署成本远低于全量微调。  
- 可组合性强：一个基础模型 + 多个 LoRA 头。  
- 易于 A/B 测试与灰度：按请求/租户动态选择加载哪个适配器。

### 局限

- 仍然需要较完整的训练/评估/监控链路。  
- 对团队的 MLOps 能力有一定要求（数据清洗、训练管理、模型注册等）。

## 3. 全量微调（Full Fine-tuning）

### 特点

- 直接更新模型绝大部分参数：
  - 训练成本高（GPU 资源、时间）。  
  - 对数据与训练配置敏感，容易「毁掉」基础模型原有能力（灾难性遗忘）。

### 适用场景

- 极少数高价值、数据充沛且对模型行为有极强控制需求的场景：
  - 如完全自研、闭环控制系统。  
  - 或需要在开源基础模型上训练出新的领域模型。

### 风险

- 训练失败或数据质量问题可能严重破坏模型能力，需要完善的回滚与版本管理。  
- 需要强 MLOps 能力与大量资源支持，一般不推荐作为企业应用的首选路线。

# 与其他决策的关系

- 与 `0003-model-and-inference-service-selection`：
  - 决定模型托管/推理方式的决策会影响可用的微调路径：
    - 若主要采用云厂商托管模型，则更多考虑厂商微调服务。  
    - 若采用自托管或第三方推理网关，则更适合用 PEFT/全量微调结合。

- 与 `0006-foundation-models-selection`：
  - 不同模型家族对微调的支持程度与策略不同：
    - 有的模型针对微调做了特殊优化（如提供官方 LoRA 支持）。  
    - 有的在厂商侧仅开放有限的微调入口。  
  - 在选择基础模型时，应考虑其微调与部署生态。

- 与 `0008-knowledge-store-and-retrieval-selection`：
  - 微调与 RAG 互补：
    - RAG 适合承载可变知识与可审计信息。  
    - 微调适合内化稳定模式、格式与风格。  
  - 在设计场景时，应先根据 `0008` 决策设计好知识检索策略，再评估是否需要额外微调。

- 与 `0005-engineering-design`（驭控工程）：
  - 微调改变的是模型本身行为，属于「高风险变更」。
  - 在流程上应被视为需要严格审批和回滚策略的变更类型：
    - 引入前必须有 Eval 与监控方案。  
    - 上线后需要密切监控关键指标与错误样本。
