# 描述

Qdrant 是专门面向向量检索场景的开源数据库，支持 HNSW 索引、多租户 Collection、结构化过滤查询，提供 HTTP/gRPC API 以及多语言 SDK，可通过 Docker 单节点或集群模式部署，也有云托管版本。

# 优点

- 针对向量检索深度优化：
  - 内置 HNSW 等 ANN 算法，适合高维向量和大规模数据集的近似最近邻搜索。
  - 同时支持向量相似度 + 结构化过滤条件，适合按「项目/用户/标签」等维度做精细检索。
- 运维与集成成本较低：
  - 提供官方 Docker 镜像，单节点/集群部署都相对简单，适合从本地开发到生产环境逐步演进。
  - HTTP/gRPC API 设计清晰，多语言 SDK（Python、TypeScript、Go 等）完善，易于在多语言微服务体系中复用。
- 社区与生态成熟：
  - 有较活跃的开源社区，文档完整，示例丰富，遇到问题容易找到实践经验。
  - 与 LangChain、LlamaIndex 等主流 LLM 框架有现成集成，可直接复用 RAG 相关组件。
- 可扩展性较好：
  - 支持分片、副本、多节点部署，可以在数据量和 QPS 增长时平滑扩容。

# 缺点

- 引入新的独立服务：
  - 相比「直接用现有关系型数据库」，需要额外运维一个组件，增加部署和监控复杂度。
- 对传统关系型查询支持有限：
  - 适合做「向量相似度检索 + 过滤」，不适合承载业务交易数据；通常需要与其他数据库（如 PostgreSQL/MySQL）配合使用。
- 生态仍在快速迭代：
  - 与更老牌的数据库相比，API 与配置仍在演进，升级时需要关注兼容性与性能变化。

# 使用 Qdrant 的开源项目示例

- LangChain & LlamaIndex 官方/社区示例：
  - 官方文档中提供了使用 Qdrant 作为向量存储的 RAG 示例，涵盖文档索引、检索与问答。
- Qdrant 官方示例仓库：
  - 包含多种语言的集成示例（Python、Node 等），展示了基础 CRUD、过滤检索、分片部署等特性。
- 对比本仓库中的项目：
  - `aiguides` 当前没有使用独立向量数据库，而是依赖上游 Gemini 能力和现有数据库，适合单项目场景。
  - `opencode` / `pi-mono` 更偏向「本地上下文 + 会话存储」，未强依赖某一向量库；Qdrant 则适合我们在多项目、多知识源场景下构建统一 RAG 服务。
