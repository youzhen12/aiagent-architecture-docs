# 设计本功能时需要注意的点

本目录下的文档用于对比不同向量数据库方案（Postgres + pgvector、Qdrant、Milvus 等），在做选型和落地时，建议重点关注以下方面：

1. 数据规模与增长趋势
- 预估当前与 1–2 年内的文档/代码/日志数据量，以及向量条目数量和向量维度。
- 确认候选方案在该规模下的性能与容量上限（单机 / 集群），避免一开始就选到天花板太低的方案。

2. 查询模式与性能目标
- 读写比例、检索 QPS、允许的 P95 延迟、是否需要多租户隔离等。
- 是否需要复杂过滤（按项目/标签/时间范围）与排序组合，检查向量库对「向量相似度 + 结构化过滤」的支持情况。

3. 运维与团队经验
- 团队对 PostgreSQL、Qdrant、Milvus 等的熟悉度，是否有现成的监控与备份方案。
- 是否有托管服务（云厂商 / SaaS），能否减少自建运维成本和故障风险。

4. RAG 生态与集成能力
- 与当前/预期使用的 LLM 框架（如 LangChain、LlamaIndex、LangGraph 等）的集成成熟度。
- 是否有较完善的 SDK/客户端、迁移工具和社区最佳实践案例。

5. 多环境与多项目支持
- 如何在同一个向量库实例中隔离多项目、多环境（dev / staging / prod）数据。
- 是否支持按 Collection/命名空间划分，如何做访问控制与配额管理。

6. 成本与资源占用
- 存储成本（磁盘/内存）、计算成本（CPU/GPU）以及随数据量增长的成本曲线。
- 与「直接用关系型数据库 + pgvector」等方案在总拥有成本（TCO）上的对比。

7. 安全与合规
- 支持的加密机制（静态/传输中）、访问控制（API Key、Token、VPC 内访问等）。
- 是否有审计日志、操作记录，满足内部合规/审计需求。

8. 迁移与退出策略
- 如果未来需要从当前方案迁移到其他向量库，是否有可行的导出/导入路径。
- 索引格式、向量存储结构是否易于抽象，避免形成过深的技术锁定。

在编写本目录下的具体方案文档（如 `qdrant.md`、`postgres-pgvector.md`、`milvus.md`）时，建议针对上述每一点给出：

- 对应方案的现状评估。
- 已知优势与限制。
- 结合 AIAgent 使用场景的推荐程度（适用/勉强可用/不推荐）。

## 近期生态变化（简要）

- 截至近两年，主流向量检索方案大致分为三类：关系型数据库向量扩展（如 PostgreSQL + pgvector）、专用向量数据库（如 Qdrant、Milvus）、以及搜索/分析引擎的向量能力（如 Elasticsearch/OpenSearch 向量检索）。
- 多数厂商都提供托管服务或云原生版本，向量数据库不再是「必须自建」的基础设施，可以根据团队运维能力选择自建 vs 托管。
- RAG 框架（LangChain、LlamaIndex、LangGraph 等）对上述常见向量方案均已有成熟适配，本目录的对比更聚焦在运行成本、规模上限与团队技能栈匹配度，而非「是否能用」。
