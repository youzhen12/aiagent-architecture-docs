# 描述

PostgreSQL 是通用关系型数据库，通过 pgvector 插件可扩展向量类型与相似度搜索功能，实现「结构化数据 + 向量数据」统一存储与查询。近年来，pgvector 已被多家云数据库服务（如 Supabase、各大云厂商托管 PG）原生集成，易用性与生态都有明显提升。

# 优点

- 架构简单、技术栈熟悉：
  - 利用已有的 PostgreSQL 集群即可引入向量能力，无需额外部署新组件。
  - 运维团队通常对 PostgreSQL 更熟悉，上线风险相对较低。
- 数据模型统一：
  - 结构化元数据与向量数据同库存储，部分场景下查询与事务处理更简单。
  - 可以直接使用 SQL 做联表、过滤与权限控制，复用现有的数据访问层。
- 生态与托管支持增强：
  - 多个云服务（Supabase、各大云厂商托管 PG 等）已经将 pgvector 作为一等能力集成，减少了自行安装/升级插件的工作量。
  - 社区对 pgvector 的向量类型、索引选择（IVFFLAT、HNSW 等）以及参数调优有越来越多实践经验和文档可参考。

# 缺点

- 针对向量检索优化有限：
  - 与 Qdrant/Milvus 等专用向量数据库相比，在大规模向量与高并发检索场景的性能与扩展性仍偏弱，尤其是在百亿级向量、极高 QPS 的检索任务中。
  - 对 ANN 算法支持和参数调优能力不如专用向量库灵活。
- 资源与职责叠加：
  - 向量检索负载与业务事务共用同一数据库，若没有清晰的资源隔离（如独立实例/只读节点），可能放大数据库压力，影响核心业务。
  - 在多租户、高并发的云环境中，需要更加谨慎地规划连接池、查询模式和索引，以避免 RAG 负载拖垮主库。

# 使用 PostgreSQL + pgvector 的开源项目示例

- Supabase：
  - 托管 PG 服务 Supabase 内置了 pgvector 支持，并提供向量检索相关文档和示例，是「PG + 向量」方案的重要实践者之一。
- LangChain 示例项目：
  - 官方文档中提供了使用 pgvector 作为向量存储的 RAG 示例，适合小到中等规模的数据集。
- 对比本仓库中的项目：
  - `aiguides` 当前使用关系型数据库承载用户/会话等数据，如未来希望「少组件」地引入向量检索，pgvector 会是自然选项。
  - 对于 AIAgent 这种需要在多项目、多知识源上做高并发检索的场景，pgvector 更适合作为早期或中小规模方案，而非长期唯一方案。
