# HTTP / 网络访问沙盒设计

## 1. 威胁模型与目标

在 AIAgent 具备 HTTP/网络访问能力时，需要防范：

- 对不受信任目标的大规模请求（如被用作 DDoS 工具）。
- 访问不合规或危险站点（恶意网站、违规内容等）。
- 非预期的数据泄露（将内部信息发到外部服务）。

目标：

- 将 Agent 的网络访问限制在一组受控目标（域名/IP 列表）内。  
- 控制请求速率、并发和数据量。  
- 对关键请求和响应做审计与必要的内容过滤。

## 2. 域名/IP 白名单

- 域名级白名单：
  - 在配置中声明允许访问的域名（如公司自有 API、特定 SaaS 服务）。  
  - 默认禁止访问其他域名，除非显式加入白名单。

- IP 白名单（可选）：
  - 对内网服务或高敏感服务，可进一步使用 IP 白名单限制访问范围。  

- 解析与校验：
  - 工具层需负责域名解析与校验，防止「通过重定向」绕过白名单（例如只允许初始域名，但重定向到未知站点时拒绝）。

## 3. 请求过滤与限流

### 3.1 方法与头部限制

- 限制 HTTP 方法：
  - 默认只开放 GET/HEAD，用于只读访问。  
  - 对 POST/PUT/PATCH/DELETE 等修改性操作，仅在特定 API 上、在严格审批下允许。

- 限制请求头：
  - 对 Authorization、Cookie 等敏感头进行严格管理（通常由系统注入，而不是由模型自由构造）。  
  - 禁止模型直接控制某些安全相关头部（如 Host、X-Forwarded-For）。

### 3.2 限流与并发控制

- 基于调用方（用户/Agent/会话）进行速率限制：
  - 限制单位时间内的请求数。  
  - 限制并发连接数和单请求最大 body 大小。

- 针对特定目标的限流：
  - 避免对同一外部服务发起突发流量，影响其可用性。

## 4. 请求与响应审计

### 4.1 请求审计

记录信息包括：

- 调用方标识（Agent/用户/会话）。  
- 目标 URL（域名、路径、查询参数）。  
- 方法、关键头部（敏感信息脱敏）。  
- 请求体摘要（如截断后的前 N 字节或 hash）。

### 4.2 响应处理

- 对响应做基本大小限制：
  - 防止极大响应占用带宽和内存。  
- 对响应内容可选做：
  - MIME 类型检查，避免意外下载二进制/可执行文件。  
  - 安全扫描或敏感信息检测（视场景而定）。

## 5. 网络访问模式的分层

### 5.1 只读开放互联网（低风险信息）

- 用于公开信息查询（天气、汇率、新闻摘要等）。  
- 策略：
  - 严格限制目标域名（仅知名公共 API）。  
  - 只允许 GET/HEAD。  
  - 对返回内容进行大小和类型限制。

### 5.2 企业内部 API 与服务

- 用于访问内部业务系统。  
- 策略：
  - 只允许访问通过 API 网关暴露的接口。  
  - 凭证管理由平台负责（例如通过注入的短期 token），模型看不到原始凭证。  
  - 对所有请求做强审计，并与权限体系集成（谁可以访问哪个 API）。

## 6. 在 AIAgent 中的使用建议

- 工具定义层面：
  - 将 HTTP 访问封装为具体的业务 API 工具，而不是裸露「任意 URL 请求」。  
  - 在工具 schema 中声明：需要哪些参数、哪些是由系统注入的（如认证）。

- 编排与策略层：
  - 将不同类别的 HTTP 工具（公开 API、内部 API）映射到不同的策略（白名单、限流、审计强度）。  
  - 对涉及敏感数据/操作的 HTTP 调用增加审批或「dry-run + 人工确认」流程。
