# Shell / 文件系统沙盒设计

## 1. 威胁模型与目标

在 AIAgent 中暴露 shell 和文件系统能力时，需要重点防范：

- 删除或篡改关键文件（如源码、配置、数据库文件）。
- 访问不该访问的目录（如主机敏感路径、用户隐私目录）。
- 通过 shell 间接发起网络攻击或越权访问（如安装恶意软件、扫描内网）。

目标：

- 将 Agent 对文件系统的影响限制在「指定工作区」内。  
- 将可执行命令限制在一小部分白名单工具和参数上。  
- 对每一次写入/删除建立可审计记录，必要时可以回滚或重建环境。

## 2. 隔离级别与实现模式

### 2.1 最小隔离：受限工作目录 + 命令白名单

适用：早期原型、本地开发环境、低风险任务（如只读浏览代码、运行 linters）。

基本做法：

- 限定工作目录：
  - 在工具实现中，将所有路径解析为相对于某个根目录（如 `/workspace`），禁止访问该目录之外的路径。  
  - 对诸如 `..` 路径进行规范化和检查，避免目录穿越。
- 命令白名单：
  - 只允许执行有限集合的命令，如 `ls`、`cat`、`rg`、`go test`、`npm test` 等。  
  - 对命令参数做基本校验（禁止 `rm -rf`、禁止 `> /etc/...` 等）。

优点：

- 实现成本低，无需额外基础设施。  
- 对纯文档/代码仓库类场景基本足够。

缺点：

- 一旦应用进程本身被攻破，仍可能突破限制。  
- 适合集成在开发/测试环境，不宜作为生产级唯一防线。

### 2.2 容器级隔离（推荐默认方案）

适用：中高风险场景，需要防止 Agent 对宿主机造成任何实质性影响。

基本做法：

- 为工具调用使用专用容器镜像：
  - 镜像中只包含必要工具（编译器、测试工具、少量系统命令）。  
  - 默认用户非 root，禁用特权模式。  
- 通过挂载卷暴露工作目录：
  - 只挂载某个只读或可控的工作目录（如 Git 仓库的一个副本）。  
  - 对「写入目录」单独挂载，并在任务结束后根据策略清理或打包归档。
- 通过容器运行时限制资源：
  - CPU、内存上限。  
  - 网络访问（可完全关闭或仅允许访问特定内网服务）。  
  - 进程数量、文件描述符数量等。

优点：

- 对宿主机提供较强隔离，便于多租户使用。  
- 容器栈成熟，易于与现有 CI/CD 体系集成。

缺点：

- 需要运维能力和容器编排基础（Kubernetes/Docker 等）。  
- 冷启动开销，需要通过池化等手段优化延迟。

### 2.3 更强隔离：VM / Serverless 任务

适用：对安全要求极高的场景，或需要在客户隔离环境中运行的任务。

基本做法：

- 使用轻量 VM（如 Firecracker 类技术）或 serverless 运行环境：
  - 每次工具调用在独立沙箱中执行，执行完销毁环境。  
- 文件系统通过对象存储或只读镜像提供，输出结果通过制品（artifacts）或日志回传。

优点：

- 最强隔离，适合处理高度敏感数据或执行实验代码。  

缺点：

- 实现复杂，成本高。  
- 仅推荐在少数高风险场景中使用。

## 3. 命令与路径控制

### 3.1 命令白名单与参数校验

- 白名单：
  - 在配置中列出允许的命令（及路径），例如：
    - `rg`、`ls`、`cat`、`go test`、`npm test`、`python -m pytest` 等。  
- 参数校验：
  - 禁止危险参数组合，如：
    - `rm -rf /` 或包含根路径的删除。  
    - 对 `chmod`、`chown` 一类命令默认禁用。  
- 仅暴露高层封装命令：
  - 将复杂脚本封装为内部命令（例如 `run_tests`、`format_code`），由运维/开发团队控制脚本内容，而不是让 Agent 直接织命令。

### 3.2 路径白名单与只读区域

- 路径白名单：
  - 限制只允许访问某些子目录，如：
    - 代码仓库工作区。  
    - 临时输出目录。  
  - 禁止访问 `/etc`、`/var`、用户家目录等系统路径。

- 只读/读写分区：
  - 将代码、依赖等挂载为只读。  
  - 仅在临时目录中允许写入，必要时任务结束后删除。

## 4. 审计与回滚

### 4.1 审计

每次 shell / 文件操作应记录：

- 调用时间、调用方（用户/Agent/会话 ID）。  
- 执行的命令和关键参数（敏感信息需脱敏）。  
- 受影响的路径（读/写/删除）。  
- 执行结果与退出码。

这些日志应与可观测性决策（`0004`）联动，支持：

- 按会话回放一次任务中所有命令执行情况。  
- 在出现事故时快速定位是哪个工具调用导致问题。

### 4.2 回滚策略

- 对纯临时环境：
  - 可以采用「每次调用独立环境 + 用完销毁」的方式，天然具备回滚能力。  
- 对持久工作区：
  - 结合版本控制（Git）、快照或备份，在高风险操作前创建快照。  
  - 将 Agent 生成的修改限制为 Git 分支/补丁，由人工 review 后再合并。

## 5. 在 AIAgent 中的使用建议

- 在工具定义层面：
  - 将 shell 类操作封装为有限的高层动作（如「运行测试」「格式化代码」），而不是开放自由 shell。  
- 在编排层面：
  - 对涉及写操作的工具调用增加审批节点或双确认（由另一个 Agent 或人工）。  
- 在驭控工程中：
  - 将命令白名单、路径白名单、资源限制等策略集中管理，而不是散落在各处代码里。
