# 数据库 / 内部服务沙盒设计

## 1. 威胁模型与目标

数据库和内部微服务往往直接承载核心业务数据和能力，错误使用可能导致：

- 数据泄露（查询敏感字段、导出大批量数据）。
- 数据破坏（错误更新/删除重要记录）。
- 越权操作（调用本不属于当前用户权限的内部接口）。

目标：

- 将 Agent 对数据库/内部服务的访问限制在严格定义的「能力边界」内。  
- 对生产数据的写操作加上审批与回滚机制。  
- 对所有访问建立审计线索，并与权限体系集成。

## 2. 访问模式分层

### 2.1 只读查询（推荐默认）

- 允许执行只读查询：
  - SELECT 查询、只读视图访问。  
- 禁止任何 DML（INSERT/UPDATE/DELETE）和 DDL 操作。  
- 使用专门的只读账号或只读连接池。

### 2.2 受控写入（高风险场景）

- 在确有需要自动写入的场景：
  - 将写操作封装成业务级 API（如「创建订单」「更新工单状态」），而不是让模型直接写 SQL。  
  - API 层实现所有校验与权限检查。

- 对写入执行的策略：
  - 小步写入：一次只修改极少量记录。  
  - 幂等设计：确保重复调用不会造成额外副作用。  
  - 审批机制：对特定 API（如「删除用户」「发起退款」）要求人工或审批 Agent 确认。

## 3. 技术实现要点

### 3.1 连接与凭证管理

- 永远不要让模型直接构造连接字符串或凭证。  
- 在工具实现中：
  - 使用固定的连接配置和最小权限账号。  
  - 区分只读和读写连接，默认只使用只读连接。

### 3.2 查询模板与参数绑定

- 对于 SQL 查询：
  - 优先使用预定义查询模板 + 参数绑定，而不是让模型输出完整 SQL。  
  - 对参数做类型与范围校验，防止注入和逻辑错误。

- 对于内部服务调用：
  - 使用严格定义的请求结构（如 JSON schema），并对字段值做校验。

### 3.3 零信任访问与网络边界

- 零信任思路：
  - 默认不信任任何网络请求，即使来自内部网络或同一集群。  
  - 所有访问数据库/内部服务的调用都应：
    - 通过受控网关（API Gateway、Service Mesh）进入。  
    - 携带可验证的身份和最小权限的访问令牌。

- 对 Agent 工具层的约束：
  - 工具只调用网关地址，而不是直接访问数据库或内部服务地址。  
  - 凭证由平台注入（短期 token），模型看不到原始凭证细节。  
  - 网关根据调用方身份和策略决定是否放行特定操作。

## 4. 审批与 dry-run 机制

### 4.1 dry-run

- 对写操作，优先实现 dry-run 模式：
  - 仅生成将要执行的变更计划（SQL/请求体 diff），不直接写入。  
  - 将计划展示给人工/审批 Agent 确认后，再执行真实操作。

### 4.2 审批流

- 在编排层中：
  - 对某些工具（如「执行数据库变更」「调用高危内部 API」）添加审批节点。  
  - 审批可以是人，也可以是另一个具有更严格策略的 Agent。

### 4.3 CQRS / 读写分离在 Agent 场景中的应用

- 将读写模型分离：
  - 对读操作和写操作使用不同的访问路径和权限集：
    - 读路径：面向 Agent 的只读视图或 Reporting DB，仅包含必要字段。  
    - 写路径：严格封装的命令 API，仅暴露业务上允许的变更动作。

- 对 Agent 工具设计：
  - 为查询类工具和变更类工具设计不同的接口和权限：
    - 查询工具：只读、无副作用、允许较高 QPS。  
    - 变更工具：低 QPS、必须通过审批或 dry-run。

- 好处：
  - 降低误用写接口的概率。  
  - 便于在读侧使用更激进的缓存和索引，而写侧保持简洁安全。

## 5. 审计与回滚

### 5.1 审计

记录信息包括：

- 调用方（用户/Agent/会话）。  
- 目标资源（数据库/服务名称、表/接口）。  
- 操作类型（只读/写入）。  
- 关键参数摘要（脱敏）。  
- 结果状态（成功/失败）以及影响记录数。

### 5.2 回滚策略

- 对数据库：
  - 在高风险写入前创建备份或使用事务，确保在短期内可回滚。  
  - 在数据层或业务层设计「逻辑删除」与恢复机制。  
- 对内部服务：
  - 在业务设计中提供对关键操作的「反向操作」（如取消订单、撤销变更），并将其纳入工具集。

## 6. 多租户与审计隔离

- 按租户/业务线划分访问边界：
  - 为不同租户配置不同的数据库 schema、逻辑库或服务命名空间。  
  - 工具层根据调用上下文（租户 ID/业务线 ID）路由到对应资源，避免跨租户访问。

- 审计隔离与可视化：
  - 审计日志中显式包含租户/业务线信息。  
  - 在监控与审计界面中支持按租户/业务线过滤，便于排查问题与合规审计。

- 配合权限系统：
  - 将 Agent 工具调用绑定到具体的应用身份和用户身份，确保：
    - 即使 Agent 能够访问某个工具，也只能在授权的租户/业务范围内生效。  
  - 在策略层中对「跨租户写操作」做硬性禁止。

## 7. 在 AIAgent 中的使用建议

- 将数据库/服务访问封装为高层工具，避免模型直接构造 SQL 或内部协议。  
- 在 `0010` 顶层 ADR 的策略下：
  - 将所有「写入类」工具标记为高风险，默认要求审批或 dry-run。  
- 在场景文档中明确：
  - 该场景允许哪些数据库/内部服务操作。  
  - 哪些是只读，哪些是写入，以及对应的安全措施。
