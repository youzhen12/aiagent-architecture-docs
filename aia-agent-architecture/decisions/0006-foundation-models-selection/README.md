# 设计本功能时需要注意的点

本目录用于对比主流大模型厂商及其模型版本（OpenAI、Anthropic、Google Gemini、Meta Llama、Mistral、Qwen 等），为具体场景中的模型选型提供参考。

这里**不直接给出最终推荐**，而是提供一套对比维度与各厂商优劣分析，具体场景的最终选型应写在对应的 `scene-*/architecture-overview.md` 的「核心技术选型」章节中。

## 1. 主要对比维度

在为某个场景选择大模型时，建议至少从以下维度进行评估：

1. **能力与适配度**
   - 语言能力：中英文 / 多语种表现、长文档理解能力、复杂指令遵从度。
   - 代码能力：代码生成、重构、调试、单元测试生成的质量。
   - 推理能力：复杂推理、多步骤任务、工具/函数调用时的规划与自洽性。
   - 多模态能力：图像理解、语音、视频等是否为必需能力。

2. **成本与计费模型**
   - Token 单价 / 请求单价与团队预算的匹配情况（这里只做相对高/中/低的定性判断）。
   - 不同版本之间的性价比（例如：小模型是否足够覆盖 80% 场景，大模型只用于关键路径）。

3. **性能与可用性**
   - 延迟与吞吐：是否能满足在线服务的 P95 延迟目标，以及批处理/离线任务的吞吐需求。
   - 服务稳定性：SLA、历史稳定性、是否有成熟的多区域容灾能力。

4. **生态与集成**
   - SDK / API 设计是否清晰，是否支持流式输出、工具调用、系统指令等关键能力。
   - 与常用框架（LangChain、LlamaIndex、LangGraph 等）的集成成熟度。
   - 评估工具、监控工具、代理框架等生态是否完善。

5. **数据安全与合规**
   - 数据存储与传输策略：是否提供「不用于训练」选项、数据驻留区域、加密机制等。
   - 是否具备满足所在行业/地区合规要求的证书与条款。

6. **部署与运维模式**
   - 完全托管云服务 vs 自托管开源模型 vs 第三方推理网关。
   - 对 GPU/CPU 资源的依赖程度与团队的运维能力匹配程度。

7. **厂商锁定与迁移策略**
   - 接入方式是否深度绑定某家厂商的 SDK/特性，未来更换模型的成本多大。
   - 是否存在较为统一的抽象层（例如自建 Model Gateway），便于后期切换供应商。

## 2. 本目录中文档结构

本目录按「**每个厂商一个文件**」进行组织，例如：

- `openai.md`
- `anthropic.md`
- `google-gemini.md`
- `meta-llama.md`
- `mistral.md`
- `qwen-aliyun.md`

每个文件包含：

1. **厂商与模型族简介**：简要说明该厂商主要提供哪些模型族和能力方向。
2. **版本对比**：用表格或小节形式对比该厂商不同模型/版本的特点（能力、上下文长度、多模态、适用场景等）。
3. **优点**：在工程实践中表现突出的优势。
4. **缺点 / 风险点**：已知的限制、坑位、风险（例如价格、合规、区域受限、API 变化频繁等）。
5. **适用/不适用场景**：结合 AIAgent 典型场景给出适配性评估。
6. **与其他厂商的相对位置**：简要说明在典型对比维度上的相对强弱（如：代码能力、中文能力、多模态能力等）。

## 3. 使用方式

- 做通用认知时：
  - 从 `README.md` 开始，快速浏览不同厂商的优势和限制。
  - 针对有兴趣或候选的厂商，阅读对应的 `*.md` 文件。

- 为具体场景选型时：
  - 在场景文档中列出能力需求（例如：是否必须强代码能力、是否必须支持中文多轮对话、是否需要多模态）。
  - 根据这些需求在本目录下各厂商文档中做筛选，并在场景文档中记录最终选型及原因。

- 随时间演进：
  - 大模型迭代较快，建议定期更新本目录中的版本信息与评估结论。
  - 新增模型时，可在对应厂商文件中新开小节进行补充，而不必另建目录。
