# 设计本功能时需要注意的点

本目录下的文档用于对比不同的 Agent 编排框架（LangGraph、自研 Orchestrator、CrewAI/AutoGen 等）。在选择和设计编排层时，建议关注：

1. 流程表达能力
- 是否支持复杂流程：条件分支、循环、并行执行、补偿/回滚等。
- 对多 Agent 协作、分阶段规划与执行是否有一等支持，还是需要大量自定义代码实现。

2. 状态管理与持久化
- 会话状态如何存储：内存、本地文件、外部数据库，是否支持跨进程/多实例共享。
- 是否支持长流程、可恢复的工作流（失败之后从中间节点恢复）。

3. 调试与可观测性
- 是否提供可视化调试工具、执行图/日志视图，能否方便地复现和分析一次编排执行。
- 与现有日志、Tracing、Metrics 系统接入的难易度。

4. 与现有技术栈的兼容性
- 与现有语言/框架（Python/TypeScript/Go 等）的融合程度，是否会引入额外运行时或部署复杂度。
- 是否会与现有服务框架（如 FastAPI、gRPC、微服务框架）产生冲突或重复造轮子。

5. 扩展性与可替换性
- 工具、Agent、Node 等概念是否通过接口/抽象定义，未来更换框架时能否较容易迁移。
- 编排逻辑是否与业务逻辑强耦合，是否可以通过 DSL/配置/声明式方式减少耦合度。

6. 性能与资源消耗
- 在高并发、多并行节点的情况下的开销（协程/线程/进程模型）。
- 是否容易出现瓶颈（单点 Orchestrator）、是否有水平扩展方案。

7. 社区活跃度与维护情况
- 项目是否持续维护，有无安全/稳定性方面的已知问题与更新节奏。
- 文档、示例、Issue 质量如何，遇到问题是否容易找到解决方案。

8. 安全与权限边界
- 是否支持对不同 Agent/工具的权限隔离与审计。
- 是否方便实现「高危操作需要人工确认」等安全策略。

在编写本目录下具体方案文档（如 `langgraph.md`、`custom-orchestrator.md`、`crewai-autogen.md`）时，建议围绕以上条目给出：

- 该方案在每个维度上的表现与限制。
- 与其他方案的对比结论。
- 在 AIAgent 场景中的推荐使用方式与注意事项。
