# 描述

自研轻量编排层通常基于现有通用组件（消息队列、数据库、任务调度器等），实现一套以消息/事件驱动的状态机，将各个 Agent 视为处理节点，通过事件流来完成编排。有的团队会在此基础上再引入 LangGraph 等库作为「局部编排引擎」，自研层更多负责跨系统治理与集成。

# 优点

- 完全按自身需求定制：
  - 不受第三方框架约束，可以紧密贴合现有基础设施和业务特点。
  - 可以只实现少量通用原语（如「发送消息」「等待工具结果」），保持内核简单。
- 与现有系统深度融合：
  - 容易将编排逻辑与现有监控、告警、审计系统打通。
  - 可直接复用现有基础设施（Kafka、Redis、PostgreSQL 等），减少新组件数量。
- 更易承载企业级治理：
  - 可以在自研层统一实现鉴权、限流、审计、AB 实验等策略，与公司既有平台规范保持一致。

# 缺点

- 开发与维护成本高：
  - 需要自行解决状态持久化、错误重试、幂等性、扩展性等一系列问题，容易重复造轮子。
  - 文档、最佳实践全部需要团队自己沉淀，新人上手成本高。
- 可移植性较差：
  - 编排逻辑高度绑定在当前技术栈和基础设施上，迁移到其他平台代价较大。
- 容易忽视「LLM/Agent 特有需求」：
  - 如果只是照搬传统业务工作流引擎思路，容易缺少对 Prompt/上下文/模型调用这些特有实体的结构化支持，后续再接入专用框架或观测平台时需要较大改造。

# 使用自研编排层的开源项目示例

- OpenCode：
  - 在 Go 代码中自建了针对 Coding Agent 的状态管理与工具调用逻辑，结合 Bubble Tea 提供事件驱动的交互体验，本质上是一套轻量 Agent 编排层。
- pi-mono：
  - `@mariozechner/pi-agent-core` 提供基于事件流的 Agent 状态机，实现了模型调用、工具执行、思考内容（thinking）等过程的编排，是一个库级的自研编排实现。
- 其它 agent 框架雏形：
  - 很多早期的 CLI Agent/Demo（包括类似 helloagent 这样的项目）都是以自研循环和状态管理实现的，编排逻辑清晰但复用度有限。
