# 1. 背景

在 AIAgent 中，绝大部分有价值的场景都需要访问「外部知识」：

- 产品和业务文档
- 代码仓库
- 运维和监控数据
- 行业/合规规则

这些知识既可以存在于内部系统（Git、Wiki、文档中心、数据库），也可以来自开放网络（搜索引擎、开放 API）。

随着知识规模变大、结构越来越复杂，仅靠「一次性把文档塞进 Prompt」已经不现实，需要系统化地设计：

- 知识存储方式（文件系统、全文检索、向量库、树状索引等）
- 检索方式（关键词检索、语义检索、分层检索、网络搜索）
- 与 RAG / Agent 编排的集成方式

本决策文档给出在本项目中对「知识存储与检索方式」的总体选型原则，并在子文档中展开各类方案。

# 2. 决策概览

我们采用**分层、组合式**的知识检索策略，而不是单一技术路线：

1. 小规模、结构清晰的代码/配置仓库：
   - 首选 **直接 grep / ripgrep / 传统全文检索**，辅以简单的文件级索引。
   - 对关键路径（架构说明、决策文档）可以通过「显式链接 + 目录结构」来保证可发现性。

2. 中等规模、相对稳定的业务文档和产品知识：
   - 采用 **向量检索 + RAG** 作为主通路。
   - 根据场景需要，结合 BM25 / TF-IDF 等关键词检索做 **混合检索（hybrid search）**，提升可控性。

3. 大规模、层次化很强的知识库（大量长文档、标准、接口手册）：
   - 在向量检索基础上，补充 **树状索引 / 分层摘要结构**：
     - 先在高层节点上进行粗筛和规划。
     - 再下钻到具体章节 / 片段进行精读与 RAG。

4. 对「强时效性 / 开放域」信息（新闻、行业动态、网络资料）：
   - 通过 **网络搜索（Search API）** 作为补充通路。
   - 所有网络搜索结果在使用前都经过最小的「清洗 / 重排 / 去重」和安全检查。

总体原则：

- 对**小规模、结构清晰、变更频率高**的内部仓库，以 grep / 传统搜索为主，保持简单可控。
- 对**中大型文档库**，以 RAG 为基础能力，再根据场景叠加树状索引和混合检索。
- 对**时效性强 / 开放域知识**，以网络搜索为补充，严格限制其在决策链路中的权重和使用边界。

# 3. 选型维度

在为具体场景选择知识存储与检索方式时，我们主要考虑以下维度：

1. 数据规模与结构
   - 小规模（几十到几百个文件/文档），且结构清晰、有明确目录 → 倾向 grep / 传统检索。
   - 中等规模（几千到十万级文档），结构相对稳定 → 倾向 RAG + 混合检索。
   - 超大规模或层次结构复杂（规范、法规、手册） → 倾向树状索引 + RAG。

2. 变更频率与时效性
   - 高频更新（代码、配置、部分运维文档）：
     - 优先选择更新成本低、索引构建简单的方案（grep、轻量全文检索）。
   - 中低频更新（产品文档、知识库）：
     - 可以接受定期增量/全量重建向量索引。
   - 强时效性（新闻、网络资料）：
     - 使用网络搜索，或在采集后进入专门的时效性索引。

3. 可解释性与可控性
   - 如果场景对「为什么检索到这段内容」非常敏感（合规、审计场景）：
     - 优先选择可解释性更强的检索方式（关键词检索、结构化查询），并对 RAG 做充分的 logging 与 trace。
   - 如果偏重召回率与语义召回能力：
     - 可以用 RAG + 混合检索，同时在 UI 上显式展示来源片段。

4. 延迟与吞吐
   - 对实时性要求极高的在线场景（例如高 QPS 对话）：
     - 优先选择延迟可控的检索方案（内存缓存、局部索引、尽量减少多跳树状检索）。
   - 对延迟容忍度较高的离线分析 / 研讨类 Agent：
     - 可以采用多阶段检索（粗筛 + 精排 + 抽象），提升质量。

5. 安全与合规
   - 涉及敏感数据时：
     - 向量库和全文检索系统都必须部署在合规环境中。
     - 网络搜索返回的数据需要经过脱敏、过滤和白名单策略。

# 4. 与其他决策的关系

- 与 `0001-vector-db-selection`：
  - 本决策关注「是否以及如何使用向量检索 / 其他检索方式」。
  - `0001` 决策聚焦在「如果使用向量检索，选哪种向量数据库」。

- 与工程设计（驭控 / 上下文 / 提示词工程）：
  - 知识检索是「上下文工程」的重要组成部分。
  - 对检索结果的使用边界和检查规则，属于「驭控工程」的范畴。
  - 如何把检索到的内容以结构化方式注入 Prompt，则与提示词工程紧密相关。

- 与 Agent 编排框架选型：
  - 不同编排框架（LangGraph、LlamaIndex Agents、自研调度等）对检索方式的支持程度不同。
  - 实际落地时，需要在框架支持能力和本决策的检索策略之间做折中。
