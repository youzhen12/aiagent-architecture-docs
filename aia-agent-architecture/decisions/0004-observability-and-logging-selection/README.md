# 设计本功能时需要注意的点

本目录下的文档用于对比不同的可观测性与日志方案（仅日志方案、Prometheus + Grafana、自建或 SaaS Observability 平台等）。在为 AIAgent 设计观测与日志能力时，建议重点思考：

1. 观测目标与指标体系
- 需要观测哪些关键指标：LLM 调用量与失败率、Token 消耗、工具调用成功率、向量检索命中率、延迟分布等。
- 指标的粒度（按场景、Agent、模型、用户等维度）以及保留周期。

2. 日志策略
- 哪些事件需要结构化日志（如模型调用、工具执行、文件变更、外部 API 调用等）。
- 日志中应包含的关键信息（trace_id、request_id、场景、Agent 名称、模型名称等）以及必须避免记录的敏感数据。

3. Tracing 与调用链
- 是否需要端到端调用链（从入口到模型网关、向量库、工具层）。
- 如何与现有的 OpenTelemetry/Tracing 系统接入，Span 的命名与分层策略。

4. 告警与 SLO/SLA
- 针对哪些指标设置告警（例如：错误率、延迟、成本异常飙升等）。
- 如何为关键场景定义可用性与性能目标（SLO），并通过观测系统进行验证。

5. 集成与部署成本
- 仅日志 vs 完整观测方案的实施成本与维护成本对比。
- 与现有监控/日志平台的集成方式，是否会引入重复组件或观测孤岛。

6. 隐私与合规
- 日志与 Trace 中如何处理敏感字段（脱敏/屏蔽/哈希）。
- 是否有访问控制与审计机制，确保只有授权人员可以查看敏感链路与内容。

7. 可视化与自助分析能力
- 是否为开发/运维/业务提供可配置的 Dashboard 与查询能力。
- 是否方便对特定 Session/请求做深度排查和回放。

在编写本目录下的具体方案文档（如 `logs-only.md`、`prometheus-grafana.md`、`saas-observability.md`）时，建议：

- 从上述注意点出发，说明该方案能覆盖到哪些点、在哪些方面有明显短板。
- 给出适合采用该方案的环境/阶段（本地开发、PoC、生产等）。
- 明确未来如果要升级/替换观测方案，需要迁移哪些数据与配置。

## 近期生态变化（简要）

- OpenTelemetry 已成为事实上的观测标准，越来越多的日志、指标、Trace 系统都支持 OTLP 协议，这使得在不改动应用代码的前提下更换后端（自建 vs SaaS）变得更容易。
- 针对 LLM/Agent 系统的专用观测产品兴起（如用于跟踪 Prompt、上下文、模型输出、工具调用链路的 APM/日志平台），可以与传统的基础设施观测结合使用：前者关注「智能体行为」，后者关注「系统性能」。
- 在高吞吐 Agent 系统中，「批处理日志分析 + 实时错误告警」成为常见组合：不必把所有细节实时可视化，但要确保关键错误、成本异常、模型退化等信号能及时触达。
