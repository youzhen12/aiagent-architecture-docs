# 描述

只保证所有服务的日志可以集中收集和查询，不单独建设系统化的指标与分布式 Trace 体系，依赖日志排查问题和监控运行状况。对于早期原型、CLI 小工具或个人项目，这是最常见且实现成本最低的方案。

# 优点

- 实现简单：
  - 只需统一日志格式与收集管道，短期见效快。
- 额外运维负担小：
  - 不需要部署额外的指标和 Trace 组件，适合小团队或早期原型阶段。

# 缺点

- 对复杂问题排查能力不足：
  - 缺少时序指标和 Trace，很难快速定位性能瓶颈和跨服务调用链问题。
- 难以做容量规划与成本优化：
  - 没有系统化的指标支持，很难量化系统负载与资源使用情况。
- 不适合中大型 LLM/Agent 系统长期使用：
  - 在涉及多 Agent 编排、模型成本控制和质量评估的场景中，仅靠日志很难回答「哪个场景最贵」「哪个 Agent 最不稳定」「哪条链路经常超时」这类问题。

# 使用「仅日志」方案的开源项目示例

- 小型 CLI / Demo 项目：
  - 类似 helloagent 这类示例项目通常只输出简单日志，不构建完善的监控体系，适合作为教学或演示用途。
- 早期阶段的 Web/后端项目：
  - 很多项目在早期仅通过日志排障，随着复杂度增加才逐步引入指标与 Trace。
