## 知识检索选型决策指南

在为 AIAgent 场景选择知识存储与检索方案时，建议按以下维度进行评估：

### 核心评估维度

| 维度 | 关键问题 |
|------|----------|
| 数据规模 | 文档数量、总 token 量、更新频率 |
| 查询特征 | 关键词精确匹配 vs 自然语言模糊查询 |
| 时效性要求 | 知识是否需要实时更新，允许的延迟 |
| 权限复杂度 | 是否需要细粒度的租户/角色/文档级权限控制 |
| 可解释性 | 是否需要明确说明「为何检索到这条结果」 |
| 运维成本 | 团队是否具备向量数据库/图谱的运维能力 |
| 容错要求 | 检索错误对业务的影响程度 |

### 何时选择 RAG

✅ **推荐使用 RAG 的场景**：

1. **自然语言问答为主**：用户或 Agent 使用多样化的自然语言表达，同义说法多
2. **跨文档知识聚合**：单个问题需要综合多份文档、多个章节的信息
3. **语义相似度高**：文档间内容相似但业务含义不同，需要语义区分
4. **知识更新适中**：文档变更频率中等（天/周级），可接受增量更新延迟
5. **权限可分层**：可以在检索层按租户/角色做硬过滤

**典型用例**：企业知识库问答、智能客服 FAQ、产品文档智能检索、复杂业务规则解释

### 何时选择其他方案

| 场景特征 | 推荐方案 | 理由 |
|----------|----------|------|
| 代码/配置查找，函数名、错误码精确匹配 | `grep/rg` + 传统全文检索 | 语义检索对精确标识符不敏感，关键词匹配更高效准确 |
| 文档层级清晰，按目录/分类导航即可 | 树状索引 + 路径检索 | 利用已有结构，避免向量检索的语义漂移 |
| 需要最新公开信息，内部知识不足 | 网络搜索（受控） | 补充知识缺口，获取时效性内容 |
| 知识关联复杂，需要关系推理 | LLM Wiki + 图谱 | 向量无法显式表达实体关系 |
| 超小规模（<100 文档），手动可管理 | 直接查阅 + 简单搜索 | 避免过度工程化 |

### 混合策略（推荐实践）

实际项目中很少「非此即彼」，常见组合策略：

```
用户查询
    ↓
┌───────────────────────────────┐
│ 1. 查询理解                   │
│    - 意图识别（精确/语义/时效）│
│    - 实体提取（错误码/产品名）│
└───────────────┬───────────────┘
                ↓
┌───────────────────────────────┐
│ 2. 路由决策                   │
│    - 精确标识 → grep/全文检索 │
│    - 自然语言 → RAG           │
│    - 关系查询 → 图谱检索      │
│    - 最新信息 → 网络搜索      │
└───────────────┬───────────────┘
                ↓
┌───────────────────────────────┐
│ 3. 结果融合与重排             │
│    - 多路召回合并             │
│    - Cross-Encoder 精排       │
│    - 权限过滤                 │
└───────────────────────────────┘
```

### 选型决策树

```
需要检索的知识是什么类型？
├── 代码/配置/标识符 → grep/全文检索
├── 文档/手册/FAQ
│   ├── 文档 < 100 份 → 简单搜索 + 目录导航
│   └── 文档 > 100 份 → RAG
├── 需要关系推理 → 图谱 + LLM Wiki
└── 需要最新公开信息 → 网络搜索（受控）

是否需要细粒度权限？
├── 是 → 检索层硬过滤（向量索引按权限隔离）
└── 否 → 可在生成层做软约束

知识更新频率？
├── 高（小时级）→ 传统检索 + 缓存
├── 中（天/周级）→ RAG + 增量更新
└── 低（月级）→ 全量重建可接受
```
