# LLM Wiki（大模型维基百科）设计方案

## 1. 概念

LLM Wiki 是指基于大语言模型构建的可检索、可维护、可溯源的结构化知识管理系统。与传统 Wiki（如 MediaWiki、Confluence）不同，LLM Wiki 将向量检索、语义理解与结构化知识组织相结合，使知识具备以下特征：

1. **语义检索增强**：不仅支持关键词搜索，还支持自然语言提问式的语义检索
2. **动态知识聚合**：能够跨条目自动关联相似或相关的知识片段
3. **智能摘要与生成**：对长文档自动生成结构化摘要，支持多视角知识呈现
4. **增量更新与版本管理**：知识条目支持增量更新，保留历史版本与变更溯源

从架构上看，LLM Wiki 通常包含以下核心组件：

- **知识摄入层**：文档解析、分片、向量化、元数据提取
- **存储层**：向量数据库 + 图数据库/关系型数据库（存储结构化元数据与关系）
- **检索层**：混合检索（向量 + 关键词 + 图谱关系）
- **生成层**：基于检索结果的摘要生成、问答生成、关系推理
- **管理层**：版本控制、权限管理、知识质量评估

## 2. 适用场景

1. **企业内部知识库**
   - 技术文档、产品手册、SOP 流程的统一管理
   - 支持新人快速上手、跨部门知识共享

2. **研发团队知识沉淀**
   - 代码库文档、架构决策记录（ADR）、故障复盘报告
   - 支持开发者通过自然语言查询历史技术方案

3. **行业研究与竞品分析**
   - 行业动态、竞品功能对比、市场趋势分析
   - 支持多维度知识聚合与趋势洞察

4. **产品文档中心**
   - API 文档、用户指南、FAQ 的统一管理
   - 对外提供智能问答入口，对内支持文档质量评估

## 3. 与传统 Wiki / RAG 的对比

| 维度 | 传统 Wiki | 纯 RAG 系统 | LLM Wiki |
|------|-----------|-------------|----------|
| 检索方式 | 关键词匹配 | 向量语义检索 | 混合检索（向量+关键词+关系） |
| 知识组织 | 人工分类/标签 | 扁平化向量 | 结构化 + 层级 + 关系图谱 |
| 更新频率 | 手动编辑 | 重建索引 | 增量更新 + 版本管理 |
| 问答能力 | 无 | 一次性问答 | 多轮追问 + 溯源引用 |
| 知识关联 | 人工链接 | 无显式关联 | 自动关联 + 关系推理 |
| 适用用户 | 内容编辑者 | 终端用户 | 编辑者 + 终端用户 + Agent |

## 4. 架构设计要点

### 4.1 知识摄入与分片策略

```
原始文档 → 解析器 → 分片器 → 向量化 → 元数据提取 → 写入存储
```

- **解析器**：支持 Markdown、PDF、HTML、Word 等多种格式
- **分片策略**：
  - 按语义边界分片（段落、章节、代码块）
  - 保留层级结构信息（H1/H2/H3 路径）
  - 控制分片大小（通常 500-1000 tokens）
- **元数据提取**：
  - 文档来源、作者、创建/更新时间
  - 关键词、标签、分类层级
  - 权限信息（谁可以访问/编辑）

### 4.2 存储架构

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  向量数据库     │     │  图数据库/关系库 │     │  对象存储       │
│  ( embeddings ) │     │  ( 元数据/关系 )│     │  ( 原始文档 )   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

- **向量数据库**：存储文本 embedding，支持语义检索
  - 可选方案：Milvus、Qdrant、pgvector、Weaviate
- **图数据库/关系库**：存储知识条目间的关系与元数据
  - 可选方案：Neo4j（图谱）、PostgreSQL（关系）
- **对象存储**：保留原始文档，用于溯源与全文查阅

### 4.3 检索层设计

```
用户查询 → 查询理解 → 多路召回 → 重排 → 结果融合 → 返回
```

- **查询理解**：
  - 意图识别（问答/搜索/导航）
  - 实体提取（产品名、错误码、版本号）
- **多路召回**：
  - 向量检索（语义相关）
  - 关键词检索（BM25/全文索引）
  - 图谱关系检索（相关条目/上下游依赖）
- **重排**：
  - 使用 Cross-Encoder 或 LLM 进行精排
  - 考虑时效性、权威度、用户权限等因素

### 4.4 版本管理与溯源

- **版本控制**：
  - 每次更新保留历史版本
  - 支持 Diff 查看与回滚
- **溯源机制**：
  - 回答中显式标注来源（文档名、章节、行号）
  - 支持一键跳转到原始文档位置
- **变更通知**：
  - 订阅者接收知识更新通知
  - 关联条目自动触发重新评估

## 5. 技术选型建议

| 组件 | 推荐方案 | 备选方案 |
|------|----------|----------|
| 向量数据库 | Milvus / Qdrant | pgvector / Weaviate |
| Embedding 模型 | text-embedding-3-large / Qwen Embedding | BGE / E5 |
| 重排模型 | BGE-Reranker / Cohere Rerank | Cross-Encoder |
| 图数据库 | Neo4j | NebulaGraph |
| 文档解析 | Unstructured / Docling | PyPDF2 / Mammoth |
| 前端界面 | Docusaurus / VitePress | 自研 Wiki UI |
| Agent 集成 | LlamaIndex / LangChain | 自研 Orchestrator |

## 6. 与本项目现有方案的集成

在 `aia-agent-architecture` 上下文中，LLM Wiki 可作为以下场景的知识底座：

1. **RAG 增强**：`rag.md` 中提到的向量检索方案可与 LLM Wiki 的元数据层结合，实现更精确的权限过滤与结果重排
2. **树状索引补充**：`tree-index.md` 的层级结构可作为 LLM Wiki 的分类体系基础
3. **Agent 知识工具**：将 LLM Wiki 封装为 Agent 可调用的 `knowledge_search` 工具，支持多轮知识探索
4. **场景适配**：
   - `scene-saas-platform-agent`：内嵌产品知识库，支持 Admin/Dev/Ops 角色差异化检索
   - `scene-customer-support`：FAQ 与工单知识库的统一管理
   - `scene-claude-code-assistant`：代码库文档与 ADR 的语义检索

## 7. 实践建议与陷阱

### 7.1 推荐实践

1. **从小规模试点开始**：先选择一个垂直领域（如某产品线的 API 文档），验证效果后再扩展
2. **保留原始文档**：向量只是索引，原始文档是 truth source，务必保留可追溯性
3. **建立知识质量评估机制**：定期对检索结果进行人工审核，评估覆盖率与准确率
4. **权限前置**：在检索层而非生成层做权限过滤，避免模型越权泄露
5. **混合检索优先**：纯向量检索容易遗漏关键词精确匹配，建议向量+BM25+关系图谱组合

### 7.2 常见陷阱

| 陷阱 | 表现 | 应对策略 |
|------|------|----------|
| 分片语义丢失 | chunk 割裂上下文 | 保留层级路径，使用滑动窗口 overlap |
| 更新延迟 | 知识已过期但索引未更新 | 建立增量更新管道，设置 TTL 策略 |
| 检索漂移 | 语义相近但业务不相关 | 加元数据过滤，使用重排模型 |
| 权限泄露 | 非授权用户看到敏感内容 | 检索层硬过滤，向量索引按权限隔离 |
| 评估缺失 | 无法衡量知识质量 | 建立测试集，定期跑 Recall@k 与人工审核 |
