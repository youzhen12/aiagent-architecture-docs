# 直接 grep / 传统全文检索

## 1. 定义与典型形态

- 直接基于文件系统的文本搜索工具：
  - 如 `grep`、`ripgrep (rg)`、`ag` 等。
- 传统全文检索引擎：
  - 如基于倒排索引的搜索（Lucene、Elasticsearch、OpenSearch 中的基础 BM25 等）。

它们通常基于关键词或短语匹配，可能包含分词、词干化、简单的相关性打分，但本质上仍然是「词项匹配」而非语义匹配。

## 2. 适用场景

1. 代码仓库与配置库
   - 查找函数、类、配置项、环境变量名等。
   - 开发者和 Agent 都可以把仓库视为「系统事实来源」，通过 `rg` 快速定位定义与调用位置。

2. 结构清晰的小中型文档库
   - 企业内部的技术文档、架构说明、ADR、Runbook 等。
   - 通过合理的目录结构和命名规则，配合关键词检索即可较好地发现信息。

3. 对可解释性要求极高的场景
   - 合规审计、安全排查等，需要明确说明「为何检索到这条结果」。
   - 关键词匹配 + 高亮命中词，解释成本低。

## 3. 优点

- 简单可靠，可预测性强。
- 不需要复杂的建模和 embedding 训练或选择。
- 更新成本低：文件更新即生效，或仅需轻量级重建索引。
- 对短文本、结构化命名（函数名、字段名、接口名）非常友好。

## 4. 局限

- 对同义表达、自然语言问题理解能力弱，需要用户/Agent「说人话像机器」。
- 在长文档中，返回位置可能很多，Agent 需要再做一层筛选和摘要。
- 对没有明显关键词的语义问题（例如「这段逻辑是否有竞态风险？」）帮助有限。

## 5. 在 AIAgent 中的使用建议

1. 把 `rg` 视为基础工具
   - 在代码相关场景中，默认优先通过 `rg` 定位，再视情况调用更重的 RAG。 

2. 与 RAG 组合
   - 可以在 RAG 前后使用关键词检索：
     - 先用关键词预筛，再对筛选出的文档做向量检索。
     - 或者在 RAG 返回结果后，以关键词为辅助进行重排与过滤。

3. 加强命名与文档结构
   - 为了让 grep 型检索更有效，需要在工程层面鼓励：
     - 清晰一致的命名规则。
     - 合理的目录结构与文件拆分。
