# 向量检索 + RAG

## 1. 概念

RAG（Retrieval-Augmented Generation）通过「检索 + 生成」的方式，让大模型可以结合外部知识回答问题：

1. 先将文档分片（chunk），为每个片段计算向量 embedding，并写入向量数据库或向量索引。
2. 对用户问题计算 embedding，在向量空间中检索相似片段。
3. 将检索到的片段作为上下文注入 Prompt，让模型基于这些外部知识生成回答。

从工程视角看，一个完整的 RAG 系统通常包含四个核心组件（参考 Pinecone、AWS、IBM 等实践的抽象）：

1. 数据摄入（Ingestion）
   - 数据清洗与预处理：去除噪声、模板内容、导航等无用文本。
   - 文档分片（Chunking）：按语义或结构切分为更小单元。
   - 向量化（Embedding）：调用 embedding 模型生成向量表示。
   - 写入索引：将向量和元数据写入向量数据库或检索引擎。

2. 检索（Retrieval）
   - 基础语义检索：基于向量相似度（余弦 / 内积 / L2）。
   - 混合检索（Hybrid Search）：结合向量检索和关键字检索（BM25、稀疏向量）。
   - 重排（Rerank）：用更强的交叉编码器对候选结果重排序。

3. 增强（Augmentation）
   - 将检索结果（context）与用户问题组合成结构化 Prompt：
     - QUESTION：用户问题。
     - CONTEXT：检索到的相关片段。
     - INSTRUCTIONS：约束模型「只依据 CONTEXT 回答」。

4. 生成（Generation）
   - 模型基于增强后的 Prompt 生成回答。
   - 在高风险场景中，生成结果需要额外的规则检查或人工审核。

在 Agent 场景下，RAG 不只是一次性「查一次、答一次」，而是：

- Agent 根据任务动态构造查询（query rewriting / decomposition）。
- 反复调用检索工具，评估当前上下文是否足够，再决定是否继续补充。
- 对比多轮检索结果进行交叉验证（agentic RAG）。

## 2. 适用场景

1. 企业知识库 / 产品文档问答
   - FAQ、操作指南、API 文档等。
   - 用户问题表达多样、同义说法多，传统关键词检索容易漏召回。

2. 复杂业务问答 / 配置解释
   - 需要跨多份文档拼接信息才能回答的问题。

3. 面向外部用户的智能客服
   - 需要在保证安全的前提下尽量覆盖各种自然表达。

4. Agent 复杂任务的「知识底座」
   - 如代码重构、复杂排障、行业报告撰写等需要大量阅读和综合外部资料的任务。

## 3. 优点

- 更强的语义召回能力，对同义词、口语化表述更鲁棒。
- 可以跨文档、跨章节聚合上下文。
- 易于与现有向量数据库产品集成（参考 `0001-vector-db-selection`）。
- 相比直接微调模型：
  - 更经济：不需要频繁重训或精调模型本体。
  - 更新更灵活：只需重建或增量更新索引即可反映新知识。
  - 更可控：可以精确指定哪些数据源可被检索、可被展示。

## 4. 挑战与常见坑

### 4.1 分片与窗口设计

- chunk 太小：语义不完整；太大：检索不精确、上下文窗口浪费。  
- 需要根据文档结构（标题、段落、章节）设计智能切分，而不是盲目按字数。  
- 常见策略：
  - 句子/段落级切分 + 滑动窗口（overlap）。
  - 利用标题层级（H1/H2/H3）构建层次化 chunk。

### 4.2 检索阶段的工程细节

- 仅使用向量检索容易：
  - 漏掉重要关键词（如内部缩写、产品代号、错误码）。
  - 召回大量语义相近但业务不相关的内容。
- 实践中常见做法：
  - 混合检索：向量 + 关键词（稠密 + 稀疏）。
  - 对结果使用重排模型（reranker）提升前几条的相关性。

### 4.3 索引更新成本

- 文档频繁变更时：
  - 全量重建向量索引成本较高（时间 + 费用）。
  - 需要设计增量更新策略（按文档级或目录级更新）。
- 实践建议：
  - 把原始文档和向量索引看作「双写」系统，确保有源头可重建。
  - 对高价值知识库建立定期重建计划（如每日/每周）。

### 4.4 召回错误与幻觉

- 向量检索无法完全避免「拿错文档」的问题，尤其在知识高度相似的场景。  
- 模型在不充分或错误上下文的基础上仍然会产生自信但错误的回答。  
- 需要：
  - 显式展示来源（文档名、章节、链接）。
  - 配合关键词过滤 / 规则过滤（黑名单、领域约束）。
  - 对高风险场景加入额外的校验或人工审批。

### 4.5 评估难度

- RAG 系统需要分别评估：
  - 检索质量（是否找到了正确的片段）。
  - 生成质量（是否基于这些片段给出了正确、清晰的答案）。
- 实践中通常通过：
  - 构建一批带「标准答案/参考文段」的 ground truth 测试集。
  - 对检索阶段使用指标（Recall@k、MRR 等）。
  - 对生成阶段结合人工标注和自动指标（如是否引用了正确片段、是否出现禁用模式）。

### 4.6 安全与权限

- 如果把权限控制交给模型本身，会存在严重风险。  
- 需要在检索层或索引层做硬约束：
  - 按用户/租户/角色划分索引或过滤条件。
  - 对敏感字段（PII、机密数据）做脱敏或不入库处理。

## 5. 在 AIAgent 中的使用建议

1. 以 RAG 作为默认文档问答方案，但不过度使用
   - 对简单查找类问题（找函数、看某个配置）仍优先使用 grep / 精确查询。

2. 与混合检索结合
   - 使用向量检索 + BM25 或其他稀疏检索做混合排序，在同一召回集上平衡语义相关性与关键词命中。

3. 明确失败模式与兜底策略
   - 检索不到高相关内容时，应该：
     - 显式告诉用户「当前知识库中没有找到相关信息」。
     - 而不是强行编造答案。

4. 与 Agent 编排结合（Agentic RAG）
   - 允许 Agent：
     - 对原始问题进行改写或分解，再分别检索。
     - 在多轮对话中记住先前检索结果，避免重复查询。
     - 对多个数据源（向量库、SQL、日志）结果进行聚合和交叉验证。

5. 与驭控工程联动
   - 把 RAG 的「数据源、检索参数、可用字段」视为策略配置，而不是散落在 Prompt 里的隐性约定。
   - 对每次检索和生成过程做日志与 Trace 记录，便于事后审计和调优。
