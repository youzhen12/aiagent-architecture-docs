# 树状索引 / 分层检索结构

## 1. 概念

树状索引指的是将大规模文档或知识库构建成一个多层级结构，用「节点 → 子节点」的方式组织信息：

- 叶子节点：原始文档分片（chunks），通常保持较细粒度，用于最终 RAG 检索与阅读。
- 中间节点：对若干子节点内容的摘要、聚合或分类，类似于「章节概览」「目录项」。
- 根节点：对整个语料库或某一大类内容的总览，帮助在最顶层缩小范围。

检索时不再直接在全量叶子上做一次性搜索，而是：

1. 先在高层节点（根节点或第一层）做粗筛或规划，选出与当前查询最相关的子树。  
2. 在选定子树的下一层节点继续检索或下钻。  
3. 一直下沉到叶子节点，再对叶子节点进行更精细的向量检索或关键词检索，并将结果用于 RAG。

在一些框架（如 LlamaIndex 的 TreeIndex、LangChain 的 Hierarchical Retrieval）中，树状索引已经有较成熟的实现，可以理解为「在普通向量检索之上再加一层/多层路由与摘要结构」。

## 2. 适用场景

1. 超长文档与规范
   - 如国家标准、行业规范、复杂 API 文档、长篇设计文档等。  
   - 用户问题通常只涉及某个章节或主题，而不是整篇文档。

2. 强层次结构的知识库
   - 如：
     - 产品手册按功能模块组织的知识库。  
     - 知识图谱的某一文本化投影（例如按领域 → 子领域 → 实体分层整理的说明文档）。

3. 需要「先规划再阅读」的 Agent 任务
   - Agent 需要先形成一个「阅读计划」，再逐步执行：
     - 先浏览高层摘要，确定要深入的章节。  
     - 然后在子章节内做细粒度检索与阅读。  
   - 典型例子：
     - 让 Agent 阅读一份长达数百页的标准/白皮书，并产出结构化综述。  
     - 让 Agent 理解一套大型系统的架构文档和 ADR 决策历史。

4. 多 Agent 协作需要共享「知识地图」的场景
   - 不同 Agent（如规划 Agent、执行 Agent、审核 Agent）需要对同一知识库有一个共同的「全局地图」，而不仅是各自零散检索到的局部片段。

## 3. 设计与构建方式

### 3.1 自底向上构建

1. 对原始文档进行分片，得到叶子节点（chunks）。  
2. 将相邻或语义/结构相关的叶子节点分组，对每组内容生成摘要或主题描述，形成上一层节点。  
3. 对新生成的节点继续分组和摘要，逐层向上折叠，直到形成 1–N 个根节点。

特点：

- 可以在没有明确目录结构的情况下构建层级。  
- 摘要质量对整个树的可用性影响非常大。

### 3.2 自顶向下构建

1. 利用已有的目录/章节结构作为骨架：
   - 如文档中的 H1/H2/H3 标题、API 手册的章节、知识库系统中的分类层级。  
2. 对每个目录节点生成对应的「节点摘要」，并将其下属内容（子标题或叶子 chunk）作为子节点。

特点：

- 层级结构更贴近人类理解（目录即索引）。  
- 对已有结构较好的文档成本更低，但对「结构混乱」的知识库效果一般。

### 3.3 节点信息设计

无论自顶向下还是自底向上，每个节点通常需要包含：

- 摘要文本：对该节点所覆盖内容的简明描述。  
- 元数据：所属文档、章节号、标签等。  
- 子节点引用：子节点 ID 列表或指向叶子 chunk 的引用。

在检索阶段，可以：

- 对节点摘要本身做向量检索，找到最相关的节点。  
- 或者以节点摘要为条件，对节点覆盖范围内的叶子做二次检索。

## 4. 优点

1. 提升检索效率
   - 对大规模知识库，如果直接在所有叶子节点上做向量检索，成本高且易受噪声影响。  
   - 树状索引先在高层缩小范围，再下钻，减少全库扫描压力。

2. 更符合人类阅读与规划习惯
   - 人类阅读长文档通常会先看目录与摘要，再定位到章节阅读细节。  
   - 树状索引让 Agent 可以模仿这种模式，在高层「规划」，在低层「执行」。

3. 有利于长链路 Agent 任务的分解
   - Agent 可以用「节点 → 子节点」的方式逐层拆解任务：
     - 先决定看哪些部分。  
     - 再在这些部分中检索具体证据。  
   - 有助于减少「漫无目的地扫遍全库」带来的成本与噪声。

4. 提升可解释性
   - 可通过「检索路径」解释模型是如何定位到某段内容的：
     - 根节点 A → 子节点 A.2（某章节）→ 叶子 chunk X。  
   - 对需要审计和复现的场景（合规、法律、医疗）尤其有价值。

## 5. 挑战与风险

1. 构建与维护成本
   - 需要为每层节点生成摘要或分类信息，构建成本比单层向量索引高。  
   - 文档更新时，要考虑从叶子到根节点的「多层同步」：
     - 某个叶子内容变化，可能需要更新其所有祖先节点的摘要。  
   - 需要设计增量更新策略，避免每次变更都全量重建。

2. 信息丢失与偏差
   - 摘要是有损过程：如果摘要质量不佳，可能导致高层节点对下层内容的表示失真。  
   - 查询只在高层运行时，可能因为摘要偏差而漏掉相关子树。

3. 复杂度与调试难度
   - 相比「平铺 RAG」，树状索引引入了额外的检索路径和策略，错误源头可能在：
     - 摘要质量问题。  
     - 节点划分/层级设计不合理。  
     - 检索策略（如只看前几层、不做回溯）过于激进。  
   - 调试需要能看到「路径」和每层的候选节点，工程实现门槛更高。

4. 框架依赖
   - 目前成熟的树状索引实现多绑定在特定框架内（如部分 RAG/Agent 框架的 Tree Index / Hierarchical Retriever）。  
   - 自研实现时需要考虑：
     - 与现有向量库、检索引擎的集成方式。  
     - 与 Agent 编排框架的接口对接。

## 6. 在 AIAgent 中的使用建议

1. 不作为默认方案，只在确有价值的场景启用
   - 对于中小规模知识库，简单的「向量检索 + 合理 chunk」通常已经足够。  
   - 在以下信号出现时再考虑树状索引：
     - 长文档阅读任务频繁，且 Agent 经常迷失在大文档中。  
     - RAG 对超长规范/手册的检索经常召回大量无关片段。

2. 重点用于「文档型层次知识」
   - 如规范、标准、手册、API 文档，而非结构化业务数据（交易记录、时序日志等）。  
   - 对强结构化数据，应优先考虑 SQL / 向量检索 / 图查询等更适配的方式。

3. 控制层数与扇出
   - 层数过深会增加延迟和实现复杂度；过浅则难以真正起到「规划与缩小范围」的作用。  
   - 经验值（可按项目调整）：
     - 3–5 层深度。  
     - 每层 10–50 个节点（取决于文档总规模）。

4. 与 RAG 结合的推荐模式
   - 两阶段检索：
     1. 在高层节点（摘要）上做向量检索，选出若干候选章节/子树。  
     2. 在这些子树内，对叶子 chunk 做更精细的向量检索或混合检索，并将结果用于 RAG。  
   - 对关键问题，可以将「路径」也注入到 Prompt 中，帮助模型理解上下文层次关系。

5. 与驭控工程联动
   - 将以下内容显式写入配置/策略，而不是散落在 Prompt 中：
     - 哪些文档/知识库使用树状索引。  
     - 每层的 chunk 策略与最大深度。  
     - 在哪些业务路径上必须优先通过树状索引进行缩小范围。  
   - 在日志与可观测性中记录「检索路径」（访问了哪些节点、如何决策下钻），便于事后审计和调优。

## 7. 评估与运维建议

1. 评估树状索引是否带来收益
   - 与基线方案对比：
     - 基线：单层向量检索 + RAG。  
     - 实验：在相同评估集上引入树状索引，比较：
       - 检索相关性（Recall@k、MRR 等）。  
       - 人工打分的答案质量。  
       - 检索与生成的延迟、成本。

2. Ground truth 与路径分析
   - 为一部分评估样本记录「理想路径」：
     - 理想应命中的章节/节点。  
     - 理想应命中的叶子片段。  
   - 对比真实运行时的检索路径，识别：
     - 哪一层的节点选择出现偏差。  
     - 是否需要调整节点摘要或层级划分。

3. 运维与重建策略
   - 为树状索引定义「重建阈值」：
     - 如某文档集合有超过 X% 发生较大改动时，全量重建该集合的树。  
   - 对于频繁变更的文档：
     - 设计增量更新流程：更新叶子 → 局部更新对应子树摘要。  
   - 定期巡检：
     - 抽样人工检查高层节点摘要是否仍然准确代表子节点内容。  
     - 将明显失真的节点标记为需要重建。
