# 目标

- 让提示词从「一次性手写指令」升级为「可版本化、可复用、可测试」的工程资产。
- 在不同模型、不同场景之间，尽量共享结构化的提示模板，减少重复劳动和行为不一致。

# 核心关注点

- 结构化：
  - 将提示拆成 System / Developer / User / Tool 等层次，而不是长篇大段自由文本。
  - 为不同职责（代码生成、评审、解释、总结）设计独立的角色与模板。
- 约束与风格：
  - 明确输出格式（JSON / Markdown / 纯文本）、语气、长度等要求，减少模型随意发挥。
  - 将安全要求（避免输出敏感信息、避免执行高风险操作）写入系统级提示，并与驭控策略呼应。
- 可测试性：
  - 为关键提示设计回归样例（输入 / 期望要点），方便在模型或提示变更时快速回归。
  - 针对重要流程（如代码修改、审阅）做基准对比，评估不同提示版本的稳定性与错误率。
- 可演进性：
  - 引入版本号与变更记录（Changelog），避免不同模块「各自 fork 一份模板改动」。
  - 将通用指令（代码风格、日志风格、安全约束）抽取为共享片段，在项目级集中维护。

# 常见实践与开源项目参考

- pi-mono（coding-agent）：
  - 将多种「角色提示」「技能提示」封装在 prompts 目录与扩展机制中，支持重用与组合。
  - 启示：AIAgent 也应将提示拆分为「基础规范 + 任务模板 + 场景扩展」，并提供配置化加载能力。
- OpenCode：
  - 在配置/说明文件（如 AGENTS.md、配置 JSON）中定义统一的行为约束和交互风格，作为系统级提示的一部分。
- aiguides：
  - 针对特定场景（旅行规划、邮件摘要等）有专门的 Prompt 文档和实现说明，示范了「功能级提示文档化」的做法。
- 各类开源 Prompt 集合（Awesome Prompt / Prompt 工程示例）：
  - 为我们提供了丰富的模板和格式约定，可在此基础上做内化和自定义。
