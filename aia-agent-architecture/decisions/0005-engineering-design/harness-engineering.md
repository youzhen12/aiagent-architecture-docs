# 1. 目标

- 在保证模型发挥能力的同时，通过「环境 + 约束 + 反馈回路」对其行为进行可控的约束与引导，确保安全、可靠、可解释。  
- 将「控制」从单纯依赖 Prompt 的软约束，升级为结合权限、架构规则、工具白名单、结构化测试和运行时监控的综合工程体系（即 harness）。
- 让 AIAgent 在长期演进中仍能维持内部质量和可维护性，而不是随着自动生成代码/配置的增加逐渐失控。

# 2. 核心概念：Harness vs Prompt

- Prompt Engineering：
  - 主要通过自然语言指令约束模型行为，强调「说清楚你要什么」。
  - 优点是快速灵活，但缺点是缺乏强约束、难以验证、容易「说一套、做一套」。

- Harness Engineering（驭控工程）：
  - 将 Prompt 看作输入的一部分，把更多的约束与结构固化到「运行环境和工具链」中。
  - 重点不在于写出一条完美的 Prompt，而在于：
    - 设计限制模型自由度的运行时环境。  
    - 提供高质量、结构化的上下文与工具。  
    - 使用确定性的检测机制（linters、结构化测试、约束检查）来校验模型输出。  
    - 持续从失败中收集信号，迭代完善 harness 本身。

- 在 AIAgent 中，Prompt 工程、上下文工程和驭控工程是三条互补主线：
  - Prompt 工程：定义「语言上的期望」。
  - 上下文工程：提供「模型需要知道的事实」。
  - 驭控工程：设计「模型允许做什么、如何被检查和纠偏」。

# 3. 三大驭控维度（借鉴 Harness Engineering & Codex 实践）

结合 Harness Engineering 一文和 OpenAI Codex 团队的实践，我们在 AIAgent 中关注三类驭控维度：

## 3.1 上下文与环境控制（Context & Environment）

- 目标：让模型始终在「可理解、可预期」的环境中工作，而不是面对一个混乱代码库或系统。

- 关键措施：
  - 代码仓库即记录系统 & 渐进式暴露：
    - 借鉴 OpenAI Codex 的做法，将代码仓库视为智能体的单一事实来源（system of record），而不是把重要决策散落在 Slack、文档或口头约定中。
    - 使用 `AGENTS.md` / 设计文档索引等作为「地图」，只注入一小段高层导航到上下文中，引导 Agent 再去按需读取更细节的文档（渐进式暴露），而不是一次性塞入超大提示词。
    - 在仓库中对设计文档、执行计划、技术债务等进行结构化编排，让 Agent 可以像新同事 onboarding 一样，从清晰的目录中理解系统。
  - 持续增强的知识库：
    - 将核心知识（架构说明、约束规则、关键流程、领域字典）显式写入仓库，而不是只存在于人脑或聊天记录里。
    - 为关键约束（例如「不得直接改生产资源」「写操作必须走审批」）提供机器可读的描述，方便在工具和检查器中使用。
  - 动态上下文接入：
    - 为 Agent 提供实时可用的上下文，如监控数据、变更记录、测试结果、代码结构摘要等。  
    - 与「上下文工程」文档联动，确保这些上下文既准确又不会泄露敏感信息。
  - 运行时限制：
    - 对执行环境做硬约束，例如：只允许在给定路径下写文件、只允许访问白名单 API、对资源操作做沙箱隔离，避免「随意生成/修改任何东西」。

## 3.2 架构与行为约束（Architectural & Behavioral Constraints）

- 目标：通过明确的架构规则和自动化检查，控制代码/配置的形状和边界，让 Agent 在有限而清晰的空间中工作。

- 关键措施：
  - 架构约束描述：
    - 将模块边界、依赖方向、数据流规则等，用「机器可检查的方式」描述出来（例如 ArchUnit 风格的结构化测试、配置驱动的依赖规则）。
  - 自定义 linters 和结构化测试：
    - 构建 determinisitic 的检查器，验证模型生成/修改后的产物是否符合架构约束和编码规范。  
    - 典型例子：禁止跨层依赖（UI 直连 DB）、限制某些包的使用、规定资源命名规则等。  
    - 这些检查不依赖 LLM，而是普通的测试/脚本，确保判断标准稳定一致。
  - 行为约束与审批：
    - 对高风险操作（生产环境变更、计费相关配置）强制要求人工审批或多模型交叉验证。  
    - 将「不可逆操作」封装为脚本草案，由人类决定是否执行，而非由 Agent 直接执行。

## 3.3 「垃圾回收」与熵控制（Quality Garbage Collection）

- 目标：长期运行的 Agent 系统会积累「熵」——过期文档、失效约束、互相矛盾的配置、风格漂移等，需要专门机制持续清理和修复。

- 关键措施：
  - 巡检型 Agent：
    - 设计专门的「质量巡检 Agent」，定期扫描代码、配置和文档，查找：
      - 违反架构约束的地方。
      - 文档与代码不一致之处。
      - 已废弃的接口/配置仍在使用的情况。
    - 可类比 OpenAI 内部的 "doc-gardening" / 质量巡检智能体，持续修剪陈旧文档和不良模式。
  - 自动化修复建议：
    - 巡检 Agent 在发现问题时，不直接修改，而是生成包含「问题定位 + 风险描述 + 修复建议」的报告，供人工或其他 Agent 审阅。
  - 黄金原则编码：
    - 将团队的「品味」和「不希望再看到的坏模式」提炼为一组机械化规则（黄金原则），编码到 linter、结构化测试和守护进程里，让 AI 残渣（AI residue）可以被持续清理，而不是靠人工每周集中大扫除。
  - 将失败视为信号：
    - 每当主流程 Agent 出现重复性失败（难以完成任务、频繁产生错误修改），视为 harness 不足的信号：
      - 是否缺少工具？
      - 是否缺少更明确的约束/文档？
      - 是否需要新增 linter 或结构化测试？
    - 通过这样的反馈循环，不断完善驭控体系，而不只是期待「换个更强的模型」。

# 4. 驭控工程的设计原则

结合上述维度，我们在 AIAgent 项目中的驭控工程遵循以下原则：

1. **运行时有硬边界，而不是完全靠模型自觉**：
   - 对执行能力做白名单控制（哪些命令/工具允许，在哪些环境允许）。
   - 对关键资源（仓库、数据库、生产环境）的访问必须经过显式授权和审计。

2. **确定性检查优先，LLM 辅助**：
   - 对「是否违反架构规则」「是否使用了禁用 API」这类问题，尽量用确定性检查解决。  
   - LLM 主要用于生成修复建议或解释，而不是判定是否合规。

3. **控制拆层：策略 → 编排 → 工具 → 模型**：
   - 策略层：定义业务/安全/合规策略（哪些操作允许，哪些需要审批）。
   - 编排层：将策略落实到具体流程（分支、重试、回滚、审批节点）。
   - 工具层：封装实际执行能力，保证幂等与可审计。  
   - 模型层：在上述框架内生成文本/代码/配置，但不拥有无限制执行权。

4. **可观测性是驭控的一部分，而非附加物**：
   - 全链路 Trace、结构化日志与指标是事后审计和改进驭控策略的基础。
   - 对关键事件（模型调用、工具调用、策略触发、审批决策）必须记录足够上下文（但不泄露敏感数据）。

5. **为「后悔」预留空间**：
   - 在重要决策链路中保留回滚路径：例如将生产变更设计为「脚本草案」而非直接执行。  
   - 对自动决策的影响范围有清晰边界，避免一次错误扩散到系统整体。

6. **以智能体可读性为第一目标**：
   - 和 OpenAI Codex 实践类似，代码库与文档的首要读者是智能体本身，其次才是人类。架构说明、设计原则、质量评分、执行计划等都应以「LLM 易于检索和推理」为目标组织。
   - 避免将关键信息锁在非结构化渠道（长邮件、聊天记录）中，尽量转写为仓库中的 Markdown / 配置，以便模型在本地上下文中直接使用。

# 5. 在现有项目中的落地建议

## 5.1 新建项目 vs 既有项目

- 新建项目（greenfield）：
  - 更容易在一开始就以 harness 为核心设计系统：统一技术栈、明确架构边界、规范目录结构。  
  - 可以在项目模板中预置：
    - 基础 lint/结构化测试。  
    - 最小集工具集与权限模型。  
    - 文档与配置的组织方式。

- 既有项目（brownfield）：
  - 往往结构不统一、历史债务较多，「一次性加满全部驭控」成本很高。  
  - 推荐渐进式落地：
    1. 先选一两个模块/子系统试点，增加基础 linters 与结构化测试。  
    2. 为高风险操作增加手动审批与脚本草案机制。  
    3. 逐步引入巡检 Agent，修复最明显的熵源（重复代码、无用配置等）。

## 5.2 与本项目其他工程设计的关系

- 与提示词工程：
  - 驭控工程不替代 Prompt，而是为 Prompt 提供更稳定的运行环境。  
  - 例如：在 Prompt 中要求「不要修改生产配置」，同时在工具层物理上禁止对生产环境执行写操作，两者配合才能真正可靠。

- 与上下文工程：
  - 驭控工程依赖高质量上下文（如架构说明、约束规则）才能发挥作用。  
  - 上下文工程负责「提供正确的信息」，驭控工程负责「限制可执行的动作」。

## 5.3 高吞吐 Agent 团队的额外考量

- 参考 OpenAI Codex 的经验，当 Agent 能以远高于人类的吞吐量创建和合并变更时，传统的「慢审批、强阻塞」流程会成为瓶颈：
  - 更适合的策略是：减小单次变更的影响范围，缩短 PR 生命周期，允许在低风险场景中「先合并、后纠正」，把纠错成本控制在可接受范围内。
  - 在高风险路径（安全、计费、生产变更）上仍保持严格审批和多重校验，将人工注意力集中在「需要判断力」的地方。
- 驭控工程需要显式设计这类权衡：在哪些路径上追求「高吞吐 + 易回滚」，在哪些路径上追求「低吞吐 + 强保证」，并通过策略层 + 编排层落地到实际流程里。

# 6. 开源与业界实践参考

- pi-mono：
  - `@mariozechner/pi-agent-core` 通过事件流暴露 Agent 工作过程，便于在 TUI 与日志中实现细粒度控制与审计。
  - coding-agent 将模型调用、工具调用、思考内容（thinking）拆分为可订阅事件，是一种很好地「把过程结构化」的方式。

- OpenCode：
  - 在 AGENTS.md 中明确限制可执行命令、禁止泄露敏感信息、要求工具幂等与可重试，体现了对「驭控」的工程化约束。

- aiguides：
  - 对 OAuth、JWT、CORS 等安全配置有详细说明，并通过中间件控制鉴权与权限，属于传统 Web 安全对 LLM 系统的驭控延伸。

- Guardrails / 安全框架：
  - 各类 Guardrails 项目（包含 OpenAI Policy、LLM Guardrails 等）提供内容过滤、敏感信息检测、输入输出验证等能力，可与本项目的驭控工程结合使用。

- Harness Engineering 一文的启发：
  - 驭控并非依赖某个工具或框架，而是一个持续演进的工程过程。  
  - 失败不是简单的错误，而是提示我们：哪里少了工具、约束或文档，需要更新 harness 本身。  
  - 真正的「AI 自主」建立在「强约束运行时 + 丰富上下文 + 稳定反馈回路」之上，而不是纯粹依赖更大的模型和更复杂的 Prompt。

- OpenAI Codex Harness 实践的启发：
  - 将代码仓库设计为智能体的记录系统，AGENTS.md 等文件只是地图，真正的知识分布在结构化文档和配置中。
  - 通过高吞吐但细粒度的 PR、定制化 lints 和后台「doc-gardening」任务，持续维持代码库的一致性和可读性。
  - 把团队的品味、边界与反馈回路编码进系统，使「驭控」能力随时间增强，而不是依赖一次性的提示词优化。
