# 目标

- 在有限的上下文窗口内，尽可能为模型提供「正确、必要且精简」的信息。 
- 建立一套可维护的「上下文管线」，涵盖采集、切分、索引、检索、组装与注入的全流程。

# 核心关注点

- 数据源管理：
  - 明确哪些仓库属于长期知识（代码库、设计文档、运维手册），哪些属于短期/会话知识。
  - 为不同数据源定义清晰的同步/更新策略（全量、增量、按需）。
- 切分与索引：
  - 结合内容结构（文件、模块、章节）与语义信息设计 Chunk 策略，避免「句子切断」或 Chunk 过大。
  - 针对不同类型内容（代码、文档、日志）分别设计切分与索引参数。
- 检索与排序：
  - 结合向量检索、关键词检索和规则过滤（项目、目录、标签、时间等）做多阶段召回与重排。
  - 为不同任务设定不同的检索配置（TopK、过滤条件），避免盲目「一刀切」。
- 注入策略：
  - 控制上下文注入位置与形式（系统提示、工具结果、用户附加说明），尽量避免污染用户意图。
  - 对长对话场景引入摘要节点与「树型会话 + 历史压缩」机制，兼顾可追溯性与窗口限制。

# 常见实践与开源项目参考

- Qdrant / Milvus / pgvector 等向量库：
  - 提供高效相似度检索能力，是上下文工程中「索引与检索」阶段的基础设施（对应 0001 中各方案）。
- pi-mono：
  - 在 `@mariozechner/pi-ai` 中强调上下文的序列化与跨模型交接，coding-agent 通过会话树和压缩机制管理历史上下文。
- OpenCode：
  - 通过 autoCompact 自动压缩长会话，并跟踪文件改动，为 Coding 场景提供稳定的上下文体验。
- aiguides：
  - 综合使用外部搜索（Google）、文档内容和会话状态，为特定 Assistant 构建上下文，示范了跨数据源上下文构建路径。
