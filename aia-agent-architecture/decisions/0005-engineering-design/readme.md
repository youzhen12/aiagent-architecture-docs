# 设计本功能时需要注意的点

本目录聚焦三条工程化主线：提示词工程（Prompt Engineering）、上下文工程（Context Engineering）、驭控工程（Control / Steering Engineering）。在为 AIAgent 设计相关能力时，建议重点关注以下方面：

1. 场景划分与职责边界
- 明确哪些场景主要靠「提示词」约束行为（如回答风格、输出格式），哪些场景必须依赖工具和编排（如修改代码、执行命令）。
- 为不同 Agent（代码助手、知识助手、运维助手等）划清职责边界，避免一个提示词模板包揽所有场景。

2. 提示词的结构化与版本管理
- 将提示拆分为 System / Developer / User 等层次，并为不同任务设计独立模板。
- 为重要提示引入版本号和变更记录，避免在不同模块中出现不可追踪的「复制粘贴变种」。
- 结合回归样例（输入/期望要点）对关键提示做定期回归测试。

3. 上下文来源与更新策略
- 列清楚上下文来源：代码库、设计文档、运行手册、工单/Issue、外部搜索结果等，并为每类来源定义更新/同步策略。
- 明确「长期知识」与「会话短期上下文」的分别存储与管理方式，避免所有信息都堆在对话历史里。

4. 上下文切分、索引与注入策略
- 针对代码、文档、日志等不同类型内容分别设计 Chunk 策略，避免切分过细或过粗。
- 为不同任务（问答、代码补全、诊断）设计不同的检索和注入策略，而不是统一一套 TopK/过滤参数。
- 控制上下文注入位置（系统提示、工具结果、用户附加说明），避免干扰用户意图或造成提示冲突。

5. 工具设计与权限控制
- 工具职责要单一、幂等性要好，避免一个工具隐式做多件事，难以审计和重试。
- 对高风险工具（写文件、大规模重构、CI/CD 变更、生产操作）设置明确的权限和交互确认（如「只建议、不直接执行」模式）。
- 区分只读/读写/运维级 Agent，严格控制每个 Agent 可见与可调用的工具集合。

6. 编排与错误恢复策略
- 将复杂流程拆解为可视化/结构化的步骤（图/状态机），避免「全交给模型自由发挥」。
- 为关键步骤设计错误恢复策略（重试、跳过、回滚），在编排层统一管理，而不是依赖模型本身的「自我修正」。
- 结合 LangGraph 或自研状态机时，保持「薄编排、厚工具」原则，保证每个节点背后都是可测试的业务逻辑。

7. 运行时资源与安全约束
- 明确每个场景可接受的 Token 上限、调用次数、成本预算，并在运行时做限制与告警。
- 对输入输出做基本的内容检查（敏感信息泄漏、危险命令等），必要时结合 Guardrails 类库。
- 日志中记录必要的上下文 ID（request_id、user_id、agent_name 等），同时避免记录密钥、完整 Token 等敏感数据。

8. 可观测性与审计
- 为「提示词版本 / 上下文快照 / 工具调用链 / 模型输出」建立基本的审计记录，便于事后复盘。
- 将关键事件（模型调用、工具执行、失败重试、人工干预）打入指标和 Trace，配合 0004 中的监控方案使用。

在编写本目录下的具体工程设计文档（如 `prompt-engineering.md`、`context-engineering.md`、`harness-engineering.md`）时，建议针对上述每一点给出：

- 当前工程实践的设计思路。
- 已知优势与风险点。
- 结合 AIAgent 实际场景的优先级建议（必须考虑/建议考虑/可视情况简化）。
