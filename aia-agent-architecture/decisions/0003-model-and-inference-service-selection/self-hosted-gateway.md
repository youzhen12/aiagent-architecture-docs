# 描述

在内部部署一个统一的模型网关服务，对外暴露 OpenAI 兼容或类似的 API 接口，对内根据模型名称/路由规则转发到不同的真实模型服务（云厂商 API、本地 vLLM/Ollama 实例等）。

# 优点

- 降低上层耦合度：
  - 上层应用统一使用网关提供的 API 与 SDK，切换模型或厂商时只需调整网关配置。
- 统一治理能力：
  - 可以在网关集中实现鉴权、限流、审计、路由策略、AB 测试等功能。
- 易于接入本地/自托管模型：
  - 将本地模型（vLLM、Ollama 等）包装成与云厂商一致的接口，应用侧无需感知差异。

# 缺点

- 需要额外开发与运维：
  - 网关本身是一个新的服务，需要设计、实现和持续维护。
- 网关成为关键路径：
  - 网关性能和稳定性直接影响所有模型调用，需要良好的高可用设计。

# 使用自建模型网关的开源项目示例

- litellm 等 OpenAI 兼容网关项目：
  - 提供统一的 OpenAI 兼容接口，内部可对接多种模型后端，很多团队基于这些项目自建网关。
- 部分企业内部平台（非开源，多数有公开架构介绍）：
  - 常见模式是「统一 OpenAI 兼容入口 + 多模型后端」，本文件延续这一思路。
