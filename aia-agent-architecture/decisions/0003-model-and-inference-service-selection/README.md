# 设计本功能时需要注意的点

本目录下的文档用于对比不同的模型接入与推理服务方案（直连厂商 API、自建或第三方推理网关、自托管模型等）。在设计时需要关注：

1. 多模型与多厂商支持
- 当前和未来预期使用的模型类型（聊天、函数调用、Embedding、语音、图像等）。
- 是否需要同时接入多家云厂商、不同 Region 或专有云环境。

2. 抽象与解耦程度
- 业务层是否通过统一的 `ModelClient`/`ModelGateway` 抽象调用模型，以便切换底层 Provider。
- 是否避免在业务代码中散落具体厂商的 SDK/Endpoint/参数配置。

3. 成本与配额控制
- 不同方案在调用成本、配额限制、免费额度等方面的差异。
- 是否支持按项目/团队/用户维度做配额与预算控制，以及异常消耗告警。

4. 性能与稳定性
- 延迟、吞吐量、可用性目标（如 99.9%）以及在高峰期的表现。
- 是否具备重试、熔断、回退策略（例如主 Provider 故障时自动切换到备选模型）。

5. 安全与合规
- 请求/响应中敏感数据的脱敏策略，是否满足数据出境、隐私保护等合规要求。
- 访问控制与审计能力（每次调用的来源、用途、成本记录）。

6. 部署与运维复杂度
- 自建网关或自托管模型的部署成本（GPU 资源、伸缩策略、升级流程等）。
- 与现有基础设施（Kubernetes、服务网格、API Gateway 等）的集成方式。

7. 观测与调优能力
- 是否能收集到粒度足够的调用指标（模型类型、Token 数、延迟、错误类型等）。
- 是否支持在线 A/B 测试、多模型对比与灰度发布。

8. 厂商锁定与迁移策略
- 当前方案是否对某一厂商/协议耦合过深，未来迁移成本如何。
- 是否有中间抽象层或网关，帮助减轻锁定程度。

在编写本目录下的具体方案文档（如 `direct-vendor-api.md`、`third-party-gateway.md`、`self-hosted-gateway.md`）时，建议：

- 对每个方案逐条评估上述注意点。
- 给出适用场景与不适用场景。
- 明确如果未来要更换该方案，需要做哪些改动。

## 近期生态变化（简要）

- 主流 LLM 厂商（OpenAI、Anthropic、Google、阿里云 Qwen 等）在 2025–2026 年纷纷推出统一的「Responses / Chat / Tools API」，同时支持多模态、函数调用、流式输出，使「模型能力」与「接入方式」更趋一致。
- 多厂商网关（如一些开源或商用 Model Gateway）逐渐成熟，支持在同一网关下路由多家模型、做配额控制和 A/B 实验，有利于降低厂商锁定风险。
- 自托管开源模型（Llama、Mistral、Qwen 等）在能力上不断接近闭源模型，使得「部分工作负载走自托管，部分走托管闭源」成为现实可行的混合方案，本目录的对比可以帮助你决定哪些链路需要高 SLA 托管，哪些可以放在自托管集群。
