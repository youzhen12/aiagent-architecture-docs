# 描述

在服务内部直接使用各云厂商提供的 SDK 或 HTTP API（如 OpenAI、Anthropic、Google Gemini、Azure OpenAI 等），每个服务自行管理对应的鉴权、配置和调用逻辑。近年来，各家厂商的接口形态（如 Responses/Chat/Tools API、多模态输入等）逐渐收敛，相比早期「每家完全不同」的局面，直连方式在多厂商场景下的心智负担有所降低，但高级特性仍然存在明显差异。

# 优点

- 上手简单：
  - 参考官方文档与示例即可快速完成集成，适合原型和小型项目。
- 生态与支持丰富：
  - 针对主流厂商，有大量社区示例、调试工具与 SDK 更新支持，遇到问题容易查资料。
- 能充分利用厂商高级能力：
  - 可以第一时间使用厂商新推出的模型、工具调用特性、思考/推理模式、多模态接口等，而不必等待中间网关或第三方平台跟进。

# 缺点

- 上层代码强耦合：
  - 业务代码直接依赖各家 SDK/HTTP 协议，切换厂商或模型时需要改动大量调用代码与配置。
- 多环境配置复杂：
  - 不同环境（开发/测试/生产）需要分别配置密钥、Endpoint、版本等，容易出错。
- 统一治理能力有限：
  - 难以从全局统一做限流、审计、路由与成本控制，需要在各服务中重复实现。
- 易形成「隐性厂商锁定」：
  - 虽然基础接口趋于统一，但围绕重试策略、错误处理、流式处理、多模态细节等仍存在大量厂商特性，如果不通过统一抽象层封装，后续迁移到自建网关或第三方平台的成本会很高。

# 使用直接厂商 API 的开源项目示例

- aiguides：
  - 直接集成 Google Gemini 与 Imagen，通过官方 SDK/HTTP 接口调用，没有中间网关层，是典型的「单 Provider 集成」方式。
- 各类 Demo/教程项目：
  - 大量入门性质的 ChatGPT/Gemini/Claude Demo 都采用这种方式，适合教学和验证概念。
