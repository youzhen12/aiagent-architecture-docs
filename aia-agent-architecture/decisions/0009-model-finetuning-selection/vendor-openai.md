# OpenAI 微调相关实践要点

> 本文不追踪具体价格和版本号，而是总结工程上在使用 OpenAI 微调时需要注意的几个关键点。具体配置以官方文档为准。

## 1. 能力边界与适用场景

OpenAI 提供的微调通常具备以下特点：

- 支持在特定模型系列上进行指令微调（instruction tuning）：
  - 用结构化的 prompt/response 样本增强模型在特定任务上的表现。  
- 适合作为：
  - 稳定结构化输出的手段（例如固定 JSON schema、固定回复模板）。  
  - 提升特定领域任务表现（在已有 Eval 集上测得基础模型表现不佳）。

不适合作为：

- 快速迭代替代品：
  - 每次微调都存在成本和延迟，不适合高频「试错式」使用。  
- 承载频繁变化的知识：
  - 知识更新应以 RAG 为主，微调更适合承载稳定规则与模式。

## 2. 数据准备

### 2.1 样本格式

- 通常需要以结构化格式（JSONL 等）提供样本，包含：
  - 用户输入（prompt）。  
  - 期望输出（completion）。  
  - 可能还包括系统指令/元数据等。

### 2.2 数据质量

- 高质量样本远比数量更重要：
  - 保证输出风格、格式的一致性。  
  - 避免混入错误或不一致的示例。  
- 在送入微调前，尽量用基础模型 + Prompt 自动生成候选，再经人工筛选，提高效率。

### 2.3 隐私与合规

- 根据 OpenAI 的数据使用政策，注意：
  - 是否勾选「不用于训练」类选项（不同产品形态策略不同）。  
  - 不要上传不应离开客户环境的敏感数据（可考虑脱敏/抽象化）。

## 3. 训练与评估流程

### 3.1 训练参数

- 多数情况下仅需调整少量参数（epoch 数、learning rate 等）。  
- 不宜将 epoch 设得过高，以免过拟合少量样本，削弱基础能力。

### 3.2 Eval 与对比

- 建议在微调前后使用同一 Eval 集对比：
  - 准确率、格式错误率。  
  - 关键任务上的成功率/失败率。  
- 保留基础模型作为对照组：
  - 在关键路径可以配置「双轨推理」，将部分请求在基础模型和微调模型上都跑一遍用于离线评估。

### 3.3 回滚策略

- 微调后的模型版本应有清晰的版本号/别名。  
- 在 AIAgent 系统中：
  - 将模型版本视为配置项，并纳入驭控工程的变更管理流程。  
- 出现异常时：
  - 直接切换回基础模型，或切换到上一个稳定微调版本。

## 4. 与 RAG 与 Prompt 工程的配合

1. Prompt 仍然是第一层约束：
   - 微调后的模型仍需要清晰的系统指令和任务描述。  
   - 微调主要是让模型「更容易遵守」这些约束，而不是取代 Prompt。

2. RAG 负责知识，微调负责模式：
   - 对于知识更新与扩展，优先通过 RAG 接入外部知识。  
   - 微调更适合学习：
     - 如何使用这些知识。  
     - 在给定知识下以何种格式/风格输出。

3. 在 AIAgent 中的建议流程：
   - 先用基础模型 + Prompt + RAG 完成场景端到端打通。  
   - 基于实际错误样本构建微调数据集。  
   - 做一次小规模微调并评估收益，确认正向收益明显后再扩大范围。

## 5. 本项目中的集成建议

- 在 `0003` 模型服务选型中，如选择 OpenAI 作为主要提供方：
  - 在对应场景的 `architecture-overview.md` 中，增加「是否启用 OpenAI 微调」的字段。  
- 将微调使用情况记录在场景文档中：
  - 微调的目标（例如：减少 JSON 解析错误、统一回复风格）。  
  - 微调数据来源（例如：某业务线的历史工单）。  
  - Eval 指标与回滚方案。

## 6. 模型与版本选择的额外考虑

> 具体可用哪些模型支持微调，请以 OpenAI 官方文档为准。本节聚焦于「如何选」而不是「选哪个名字」。

1. 先用小模型验证方向，再上大模型微调
   - 对大多数任务，可以先用成本更低的模型进行试验性微调：
     - 验证任务是否适合通过微调获益。  
     - 找到合适的数据格式和标签定义。  
   - 在明确看到明显收益后，再评估是否值得对更强模型进行微调。

2. 区分「对话助手」与「工具执行者」
   - 有的模型更适合作为通用对话助手，有的模型更偏向「工具/函数调用」。  
   - 在需要大量函数调用/结构化输出的场景，应优先选择对函数调用支持更好的模型系列进行微调，并在数据集中强化函数调用模式。

3. 单轮任务 vs 多轮任务
   - 单轮任务（如分类、单次生成）通常对微调比较友好：
     - 训练样本结构简单，评估也更直观。  
   - 多轮对话任务：
     - 需要在训练数据中显式表现完整对话上下文。  
     - 训练和评估复杂度更高，建议先用 Prompt + RAG 打通，再微调某些关键子任务（如意图识别、槽位抽取、回复模板）。

4. 与 Prompt Caching 等能力的组合
   - 在部分平台形态中，Prompt Caching（提示缓存）可以降低长上下文调用的成本。  
   - 典型策略：
     - 对那些「不常变」且较长的系统指令/说明文档，优先通过缓存能力优化成本。  
     - 微调重点仍放在「如何响应」而不是「重复塞同一堆提示」。

## 7. 常见问题与排查建议

### 7.1 微调后效果不如基础模型

可能原因：

- 训练数据质量欠佳：标签不一致、风格混乱、包含错误示例。  
- 训练参数不合适：epoch 过多导致过拟合，或学习率过大导致模型行为发散。  
- 任务本身更适合用 RAG 或 Prompt，而非微调。

排查思路：

- 构建小而精的 Eval 集，对比基础模型和微调模型在同一集合上的表现。  
- 回顾训练数据集，抽样检查样本质量，必要时清洗并重新训练。  
- 尝试减少训练步数或降低学习率，观察是否改善。

### 7.2 输出仍然经常「跑偏」或格式不稳

可能原因：

- 数据集中没有足够强调目标格式，或格式多样。  
- 推理时的系统指令与训练数据中的假设不一致。

排查思路：

- 在训练数据中显式强化「正确格式」和「错误格式」的对比示例。  
- 确保推理时的系统指令与训练时的假设一致，例如：
  - 训练时总是让模型输出 JSON，但线上系统指令没有强调这一点。  
- 在关键路径增加格式校验与自动修复逻辑，把不合规输出挡在业务逻辑之外。

### 7.3 线上出现意料之外的「语气变化」或「安全问题」

可能原因：

- 训练数据集中混入了不希望出现的语气/内容。  
- 微调数据缺乏针对安全/合规场景的「拒绝示例」。

排查思路：

- 明确区分：
  - 任务样本（如何回答业务问题）。  
  - 安全样本（何时拒答、何时转人工）。  
- 在训练数据中加入安全约束样本，强化拒绝模式。  
- 保留基础模型 + 通用安全策略作为后备方案，在高风险路径上优先使用未微调版本 + 更严格的策略链路。

### 7.4 成本与吞吐意外升高

可能原因：

- 对不必要的场景也使用了微调模型，而非按需选择。  
- Eval 和监控体系缺失，导致未及时发现微调模型在部分路径上的成本过高。

排查思路：

- 将模型版本与业务路径关联，在日志中记录：
  - 每条路径使用的模型版本。  
  - 每条路径的 Token 消耗与延迟。  
- 对「不敏感」路径回退到基础模型，只在确实需要的路径使用微调模型。  
- 在 AIAgent 的策略层中引入「按业务优先级选择模型版本」的能力。
