# PEFT / LoRA / QLoRA 实践要点

> 本文基于公开实践经验，总结在使用 LoRA/QLoRA 等参数高效微调技术时的工程注意事项。

## 1. 何时选择 PEFT

- 使用自托管或开源模型，希望进行轻量微调时。  
- 需要在同一基础模型上支持多个场景/租户，每个有自己小的行为差异。  
- 资源有限，希望在少量 GPU 上完成训练。

不推荐使用 PEFT 的情况：

- 任务规模极小、用 Prompt 就足够解决的场景。  
- 需求变化非常频繁，微调模型存在「刚调好就废弃」的风险。  
- 缺乏任何稳定 Eval 与监控体系的早期探索阶段。

## 2. 数据与任务设计

### 2.1 任务类型

PEFT 在以下任务上普遍效果较好：

- 指令跟随（Instruction Following）。  
- 文本分类与打标签。  
- 文本生成风格/语气定制。  
- 代码补全/生成中的风格或项目特定约定。

对需要深度改变模型结构或多模态能力的任务，则需要更谨慎评估。

### 2.2 数据分布

- 与全量微调一样，需要保证：
  - 训练数据能代表目标任务的输入分布。  
  - 输出一致、干净、格式稳定。  
- 对于结构输出任务，建议使用专门的 Schema/JSON 格式。

## 3. 模型与训练配置

### 3.1 选择基础模型

- 优先选择已针对指令微调过的基础模型（Instruction-tuned base），在此基础上做 LoRA 微调：
  - 可以在较少数据下获得更好效果。  
- 确保基础模型版本固定：
  - LoRA/QLoRA 适配器通常与特定模型版本绑定。

### 3.2 LoRA 配置

- 典型超参数：
  - rank（r）：决定低秩矩阵的大小，常见取值如 4、8、16。  
  - alpha（缩放系数）：控制 LoRA 更新的幅度。  
  - target modules：指定在哪些层/权重上应用 LoRA（如注意力层、FFN 等）。

实践经验：

- 从较小的 rank 开始（如 8），观察效益与过拟合情况。  
- 不必在所有层上应用 LoRA，可只在对任务最敏感的层上启用，以平衡效果与效率。

### 3.3 QLoRA 与量化

- QLoRA 通过对基础模型进行 4bit 量化，再在其上训练 LoRA 适配器：
  - 显著降低显存需求，便于在单卡上训练较大模型。  
- 需要注意：
  - 量化会带来一定精度损失，需在 Eval 中验证是否可接受。  
  - 对训练稳定性有一定影响，需选择成熟实现和配置。

## 4. 部署与运行时

### 4.1 Runtime 组合

- 推理时需要：
  - 加载基础模型权重。  
  - 加载对应 LoRA/QLoRA 适配器权重。  
- 多场景/多租户场景：
  - 可以将适配器权重按场景/租户进行分组，按需加载。  
  - 需要考虑加载/卸载开销与并发量。

### 4.2 模型注册与版本管理

- 建议在模型注册中心（或自建配置）中记录：
  - 基础模型标识与版本。  
  - LoRA 适配器标识、版本、所属场景。  
  - 训练数据集 ID 与训练配置摘要。  
- 对每个适配器版本，记录：
  - 上线时间与负责人。  
  - 对应的 Eval 结果（准确率、格式错误率等）。

## 5. 与 AIAgent 架构的结合

1. 作为 `0003` 模型服务层的一部分：
   - 在模型网关或推理层中增加「基础模型 + 适配器」的组合抽象。  
   - 对外暴露为一个逻辑模型 ID，内部负责加载具体的 base + LoRA。

2. 作为驭控工程（`0005`）中的一类高风险变更：
    - 将「启用/切换/回滚 LoRA 适配器」纳入变更管理与审批链路。  
    - 对适配器的使用进行日志与监控：哪些请求使用了哪个版本。

3. 与场景文档联动：
    - 在 `scene-*/architecture-overview.md` 中：
      - 明确基础模型选择（参考 `0006`）。  
      - 明确是否使用 LoRA/QLoRA 及其目标（如格式稳定、领域风格等）。

## 6. 高级实践与新趋势

> 本节覆盖一些在最近社区实践中比较常见的高级用法和新方法，适合在基本路线跑通后再考虑。

### 6.1 多适配器组合与路由

- 多适配器并存：
  - 为不同业务域/租户训练各自的 LoRA 适配器，在推理时按「场景 ID / 租户 ID」选择加载。  
  - 注意控制同时加载的适配器数量，避免显存占用过高。

- 适配器路由（Router）：
  - 通过轻量规则或小模型，对请求进行分类，然后路由到不同适配器。  
  - 适合「任务类型差异较大」但仍共享同一基础模型的场景。

- 适配器合并（Merge）：
  - 对于一些不再需要独立治理的适配器，可以在离线阶段将权重合并到新的基础模型分支中（需谨慎评估）。

### 6.2 自适应/结构化 LoRA 变体

- AdaLoRA / DoRA 等变体：
  - 一些新方法会在训练过程中自适应调整 rank，或者对权重进行更细粒度分解。  
  - 工程上通常以开源库封装的形式出现，可在对 PEFT 有一定经验后引入，但应保留基础 LoRA 作为 fallback。

- 分层 LoRA：
  - 只在高层（接近输出端）的若干层启用 LoRA，以减少计算和显存开销。  
  - 对于浅层对任务影响较小的情况，这种方式通常足够。

### 6.3 与量化与推理优化结合

- 与更激进的量化结合：
  - 在 QLoRA 之外，业界也在尝试更激进的量化方案（如 3bit/混合精度）。  
  - 工程上引入前需在 Eval 中严格验证稳定性与精度损失，并做好回滚路径。

- 与推理优化库结合：
  - 如 FlashAttention、张量并行/流水线并行等，通常可以与 LoRA 共存。  
  - 部署时要验证 LoRA/QLoRA 与推理框架（如 vLLM、TensorRT-LLM 等）的兼容性。

## 7. 常见问题与排查思路

### 7.1 训练后效果不升反降

可能原因：

- 训练数据分布与真实线上请求差异过大。  
- 样本数量过少或标签噪声大，导致模型学到错误模式。  
- epoch 设置过高或 learning rate 过大，出现过拟合或破坏基础能力。

排查建议：

- 构建一小批手工精标的 Eval 集，分别在：
  - 纯基础模型。  
  - 不同 LoRA 配置下的模型。  
  上比较表现。  
- 逐步减小 epoch/学习率，观察是否缓解问题。  
- 检查数据集是否存在明显错误或风格不一致。

### 7.2 上线后部分请求异常放大

现象：

- 大部分请求表现正常，但在某类输入上表现明显变差，甚至出现严重错误。

可能原因：

- 训练数据在某些模式上高度偏向，强化了模型的某种「偏见」。  
- PEFT 只在有限层上调整参数，对某些复杂模式的覆盖不充分。

排查建议：

- 对异常请求样本进行聚类和人工分析，看是否集中于某些子任务/领域。  
- 针对这类子任务单独扩充训练数据，并考虑为其训练独立适配器。  
- 在线设置「双轨对比」：在一段时间内同时用基础模型和 LoRA 模型处理异常类别请求，用于离线分析。

### 7.3 多适配器共存引发资源或行为问题

- 显存/延迟问题：
  - 同时加载过多适配器会增加显存占用和推理延迟。  
  - 建议为每个进程限制同时加载的适配器数量，或按需按会话维度懒加载并设置 LRU 卸载策略。

- 行为冲突问题：
  - 若误将多个适配器叠加在同一推理路径上，可能导致不可预测行为。  
  - 应在模型网关层强制一条请求仅绑定一个 LoRA 适配器（或明确的组合策略），并在日志中记录使用情况。

### 7.4 与 RAG / Prompt 的协同问题

- 如果仍然频繁出现「答非所问」或幻觉：
  - 优先检查检索质量和 Prompt 设计，而不是盲目继续微调。  
  - 将 LoRA 看作在稳固 Prompt/RAG 的前提下的「加分项」，而不是万能解。
