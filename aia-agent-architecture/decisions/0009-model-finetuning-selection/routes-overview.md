# 微调路线对比：全量 / PEFT / 厂商托管

本节从工程视角对三类主流微调路线做对比：

- 全量微调（Full FT）
- 参数高效微调（PEFT：LoRA、QLoRA 等）
- 厂商托管微调（云厂商提供的 API 级微调）

## 1. 维度对比总览

| 维度 | 全量微调 | PEFT（LoRA/QLoRA 等） | 厂商托管微调 |
| --- | --- | --- | --- |
| 训练成本 | 非常高：GPU + 时间 | 中等：参数少、可低精度 | 中等：按调用/训练计费，无自备算力 |
| 部署复杂度 | 高：新模型需单独部署 | 中：基础模型 + 适配器组合 | 低：通过厂商 API 替换模型名/版本 |
| 可控性 | 最高：完全控制权重 | 高：控制适配器权重与加载策略 | 中：训练过程黑盒，接口有限 |
| 迁移性 | 低：权重与特定模型强绑定 | 中：依赖基础模型，但适配器可独立管理 | 低：通常只能在同厂商生态内使用 |
| 多场景/多租户支持 | 差：每种变体需新模型 | 好：单模型 + 多适配器 | 一般：厂商提供有限版本管理 |
| 回滚难度 | 高：需切换整体模型 | 中：卸载适配器即可回滚 | 低：切换到基线模型或旧版本 |

工程上，一般推荐的优先顺序是：

1. 优先尝试 Prompt + RAG。  
2. 若需微调，优先厂商托管微调或 PEFT。  
3. 只有在明确需要时才考虑全量微调。

从近期业界实践看，一个相对稳妥的决策流程通常是：

1. 用基础模型 + Prompt + RAG 打出可用 baseline，并建立小而精的 Eval 集。  
2. 分析错误样本，判断问题主要来源于：
   - 知识缺失/错误（→ 优先改进 RAG 与知识库）。  
   - 输出格式/模式不稳定（→ 倾向微调）。  
3. 若倾向微调：
   - 对云厂商模型：优先尝试厂商托管微调，降低基础设施成本。  
   - 对开源/自托管模型：优先尝试 PEFT（LoRA/QLoRA），控制风险和成本。  
4. 只有在数据量巨大、长期收益显著且团队具备强 MLOps 能力时，才评估是否上全量微调或更激进路径。

## 2. 全量微调（Full Fine-tuning）

### 2.1 优点

- 最强的表达能力定制：可以针对特定任务/领域重塑模型能力。  
- 不受「附加层参数量」限制，理论上可以获得最佳任务适配效果。  
- 对某些需要深度结构调整的任务（如多模态扩展、自定义 tokenization 等）更灵活。

### 2.2 缺点与风险

- 训练成本极高：
  - 需要大量 GPU / 加速资源。  
  - 训练迭代复杂，调参空间大。  
- 容易破坏基础能力：
  - 数据分布不当或训练配置不佳，可能导致灾难性遗忘。  
  - 需要精心设计混合训练（base 数据 + 新数据），增加复杂度。  
- 版本管理与回滚复杂：
  - 每次全量微调产出一个新「大模型」，部署与回滚成本都很高。  
- 高门槛：
  - 需要成熟的 MLOps 团队与实验管理体系。

### 2.3 对本项目的定位

- 仅作为极少数场景的特殊选项：
  - 有海量高质量标注数据。  
  - 已经过 Proof-of-Concept 阶段，并验证了微调的价值。  
  - 团队有足够资源和工程能力维护独立模型分支。

## 3. 参数高效微调（PEFT：LoRA/QLoRA 等）

### 3.1 核心思想

- 冻结基础模型的绝大部分参数，只在部分层上添加可训练的「低秩矩阵」或适配层：
  - 如 LoRA：通过低秩分解分别学习两个小矩阵，对原权重做增量更新。  
  - QLoRA：在量化后的基础模型上进行 LoRA 微调，降低显存需求。  
  - Prefix/Prompt Tuning：训练一段可学习的前缀向量，引导模型行为。

### 3.2 工程优点

- 训练开销小：
  - 参数量远小于全量微调，训练速度快，对资源要求低。  
  - 可以在单机/少量 GPU 上完成中等规模微调。  
- 多场景复用：
  - 同一基础模型可以挂载多个适配器，按场景/租户动态选择。  
  - 易于 A/B 测试和灰度发布。  
- 回滚简单：
  - 卸载适配器即可回到基础模型。  
  - 可以随时切换到不同版本的适配器权重。

### 3.3 局限

- 对基础模型依赖强：
  - 适配器通常只能与特定版本的基础模型匹配。  
  - 模型更新时，需要验证适配器在新模型上的兼容性。  
- 适合「增量调整」：
  - 若需要大幅改变模型行为，PEFT 可能能力有限。  
- 仍需训练与评估基础设施：
  - 数据清洗、训练脚本、模型注册、Eval 与监控等依然必不可少。

### 3.4 对本项目的推荐用法

- 对使用开源模型/自托管模型的场景：
  - 优先采用 PEFT 作为微调的主路径。  
- 对多租户、多场景系统：
  - 将「不同租户/业务」映射到不同 LoRA 适配器版本，基础模型共享。  
- 与 Harness 工程结合：
  - 将适配器版本、加载规则作为可配置策略，纳入驭控体系。

## 4. 厂商托管微调（云厂商 API 级）

### 4.1 特点

- 由模型提供方负责：
  - 数据上传与存储。  
  - 训练资源与过程管理。  
  - 微调模型的托管与部署。  
- 开发者侧工作：
  - 准备高质量示例数据集。  
  - 调用厂商 API 创建微调任务。  
  - 用新模型 ID/版本进行推理调用。

### 4.2 优点

- 工程成本低：
  - 无需管理训练基础设施与低层工具链。  
- 集成简单：
  - 与厂商现有 Eval/监控、RAG、Agent 产品天然兼容。  
- 适合作为「从 Prompt/RAG 向微调演进」的第一步。

### 4.3 局限

- 黑盒程度高：
  - 无法深度控制训练细节、模型结构等。  
  - 迁移到其他厂商时重用难度较大。  
- 成本结构依赖厂商：
  - 可能按训练 Token/时长 + 推理 Token 收费，需要单独评估。  
- 数据与隐私需仔细评估：
  - 必须确认厂商的数据使用政策（是否用于自家继续训练等）。

### 4.4 对本项目的推荐用法

- 若项目本身以某家云厂商模型为主力（例如阿里云千问、OpenAI 等）：
  - 在需要微调时，优先考虑厂商托管微调。  
- 将厂商侧微调视为「托管版 PEFT/FT」，在 ADR 中明确：
  - 使用哪些训练数据。  
  - 如何评估与回滚。  
  - 与 RAG、Prompt 的分工边界。

## 5. 与偏好对齐方法（RLHF / DPO 等）的关系

> 本文重点在「参数更新路径」的选型，对 RLHF、DPO 等偏好对齐方法不做完整展开，这里只讨论与路线选择相关的边界。

1. 偏好对齐是另一条维度，而非全量/PEFT/厂商托管的替代
   - RLHF（基于人类反馈的强化学习）、DPO（直接偏好优化）更关注输出的偏好与安全策略：
     - 比如：更礼貌、避免有害内容、符合品牌语气。  
   - 它们可以与任意参数更新路线结合：
     - 全量微调 + RLHF。  
     - PEFT + DPO。  
     - 厂商托管微调内部也可能使用类似技术，只是对用户透明。

2. 在企业场景中的推荐顺序
   - 先确保功能正确性：
     - 通过 Prompt + RAG + 基础微调（指令/格式微调）解决「能不能做对」。  
   - 再考虑偏好与安全对齐：
     - 在已有 Eval 与安全策略基础上，用 RLHF/DPO 等方法让模型「做得更好、更安全」。

3. 工程约束
   - 偏好对齐通常需要：
     - 更多高质量的「成对比较」或「评分」数据。  
     - 更复杂的训练流程（奖励模型、策略更新等）。  
   - 在资源有限的情况下，更现实的做法是：
     - 基于规则与 Prompt 先实现安全与语气控制。  
     - 仅在最关键路径上引入 RLHF/DPO 级别的对齐。
