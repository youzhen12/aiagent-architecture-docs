# 1. 场景背景

Hermes Agent 是由 Nous Research 推出的一个开源、自托管的智能体系统，代表了一类「Agent Operating System」的工程实践：

- 不只是对话机器人，而是一个能够持续学习和成长的「成长型智能体」。  
- 通过对话与执行构成闭环，不断积累经验，并将经验固化为可复用的技能。  
- 具备多 Agent 协作、工具调用、长期记忆、自动化任务调度等能力。

从源码结构看（参考官方仓库与社区文章）：

- `agent/`：核心 Agent 运行时与任务调度逻辑。  
- `hermes_cli/`：CLI 交互层，负责本地配置和命令行操作。  
- `tools/`：工具注册与执行后端，支持本地/Docker/SSH 等环境。  
- `gateway/`：多平台消息网关（Telegram、Slack 等）。  
- `cron/`：定时任务调度器。  
- `optional-skills/`：可选技能实现样例与模板。  

本场景的目标不是复刻 Hermes 的功能，而是通过源码与架构解读，提炼其中对 AIAgent 项目有价值的设计模式，作为「对标对象」和「架构参考」。

# 2. 业务/使用场景

从使用者视角，Hermes Agent 适用于：

- 为开发者/运维/业务人员构建多 Agent 工作流：
  - 例如：需求分析 → 方案生成 → 代码实现 → 测试验证 → 部署审核。  
- 将各类工具（CLI、HTTP API、数据库、外部 SaaS、本地脚本、远程环境）统一为可被 Agent 使用的「能力」。  
- 通过 `gateway/` 对接 Telegram、Slack 等多平台，实现多通道对话与任务触发。  
- 通过 `cron/` 定时调度，实现周期性任务（报表生成、监控巡检等）的自动化执行。  
- 在复杂长链路任务中，让多个专职 Agent 协同工作，而不是单个大模型承担所有职责。

从本场景视角，我们关心：

1. Hermes 如何组织 Agent 与工具？  
2. 如何实现多 Agent 协作与调度（包括 CLI、消息网关与 cron 触发）？  
3. 记忆、上下文、执行沙盒是如何设计的？  
4. 自动技能生成（`SKILL.md`）、记忆索引（FTS5 + 向量检索）是如何落地的？  
5. 有哪些工程实践值得在本项目中借鉴？

# 3. 研究目标

通过源码解读与架构梳理，我们希望：

- 提炼 Hermes Agent 在以下方面的设计：
  - Agent 抽象与职责划分。  
  - 任务调度与多 Agent 协作模式。  
  - 工具注册、执行环境（本地/Docker/SSH）与安全策略。  
  - 记忆系统（FTS5 + 向量检索）与自动技能生成（`SKILL.md`）。  
  - CLI 与多平台消息网关集成。  
  - 日志与可观测性。  
- 将这些设计映射到 AIAgent 的决策体系中：
  - 对标 `0002` Agent 编排框架选型。  
  - 对标 `0005` Prompt/Context/Memory/Harness/Skill 工程。  
  - 对标 `0008` 知识存储与检索选型（记忆索引）。  
  - 对标 `0010` 工具沙盒与执行控制。
