# 1. 背景与目标

- 场景简介：
  - 典型形态：电商/游戏/票务等有「秒杀、大促、抢购」活动的高并发交易系统。AIAgent 不参与核心扣减库存/下单链路，而是作为「旁路智能层」，用于风控分析、运营决策支持、客服与运维助手。
- 业务问题：
  - 传统系统在大促期间需要大量运营、风控和客服人力，且决策依赖经验，难以及时复盘与沉淀；
  - 临时活动配置多、变更频繁，容易因配置错误、阈值设置不合理导致风险或损失；
  - 活动过程中产生大量日志和指标，人工分析成本高。
- AIAgent 目标：
  - 为风控/运营/运维团队提供高维度数据的分析与建议，例如活动实时健康分析、异常流量分析、配置合理性评估等；
  - 为用户和客服提供活动相关的智能问答与进度查询；
  - 在任何情况下不直接执行扣库存/转账等不可逆操作，仅输出「建议」「脚本草案」「预案方案」，由人或既有系统最终决策。
- 文档目标：
  - 给出在秒杀/大促场景下构建旁路 AIAgent 的整体架构方案与关键技术选型；
  - 明确哪些链路可以引入自动化，哪些必须保留人工审查和刚性边界；
  - 便于后续实现团队按模块推进（风控分析助手、运营助手、客服助手等）。

# 2. 业务与使用场景

- 主要用户与角色：
  - 风控团队：关注刷单/黄牛/异常行为，使用 Agent 分析行为模式与风险指标；
  - 运营团队：关注活动配置合理性、转化率、库存消耗节奏，使用 Agent 生成实时运营报表与调优建议；
  - 运维团队：关注系统健康度、容量瓶颈和异常告警，使用 Agent 辅助排查与生成 Runbook 草案；
  - 一线客服/售后：使用 Agent 回答活动规则、订单状态、补偿策略等问题。

- 关键使用场景（示例）：
  1. 实时活动健康分析：
     - Agent 聚合指标（QPS、错误率、库存消耗速度、下单成功率等）、日志与配置快照，给出「当前活动健康评分」「潜在风险点」。
  2. 异常流量与风险审查：
     - 当风控系统检测到异常流量段，Agent 拉取相关日志/特征，帮助风控人员分析是否为恶意行为，并生成规则调整建议草案。
  3. 活动配置复盘与建议：
     - 活动结束后，Agent 汇总各波次配置、库存消耗与转化率，生成复盘报告和下一次活动的参数建议。
  4. 客服问答：
     - 用户/客服询问「为什么没抢到」「补偿策略是什么」「订单状态如何」等，Agent 结合知识库与订单系统给出解释性回答，并在高风险问题上提醒升级人工处理。

- 典型业务流程（示例）：
  - 流程 A：实时健康分析 → 风控/运营决策：
    - 定时或事件触发 → Orchestrator 触发 `HealthCheckAgent` → 拉取监控指标/日志/库存数据 → 生成健康评分与风险摘要 → 推送到运营/风控看板或 IM 通知 → 人工根据建议调整配置（由传统系统执行）。
  - 流程 B：异常流量调研 → 风控规则草案：
    - 风控告警 → Orchestrator 触发 `RiskAnalysisAgent` → 对异常时间段日志/请求特征做聚合与相似度分析 → 生成「疑似风险模式 + 规则调整草案」 → 风控人员审阅/修改后生效。

# 3. 整体架构概览

- 一句话描述整体架构风格：
  - 「既有高并发交易系统 + 自建 OpenAI 兼容模型网关 + LangGraph 多 Agent 编排 + Qdrant/Milvus 向量检索 + 工具层深度集成监控/日志/配置中心 + 强驭控与审计」。

- 架构图占位：
  - [ ] 逻辑架构图：交易主链路（现有系统）与旁路 AIAgent 子系统的关系；
  - [ ] 部署架构图：AIAgent 组件在 dev/stage/prod 环境的部署方式与隔离边界（特别是只读 vs 高危写操作路径）。

- 核心组件清单：
  - 交易核心系统：现有的下单、扣库存、支付、风控引擎等，不在本架构中改动实现，只通过只读接口向 Agent 暴露必要数据。
  - AIAgent Orchestrator：基于 LangGraph + 自研 orchestrator 的编排层，负责触发各类分析 Agent、协调工具调用与状态管理。
  - 分析类 Agents：`HealthCheckAgent`、`RiskAnalysisAgent`、`OpsSugestionAgent`、`ReportAgent` 等。
  - 客服类 Agents：`FAQAgent`、`OrderExplainAgent` 等，与客服渠道集成。
  - 工具层：监控系统 API、日志平台 API、配置中心、活动管理系统、订单/库存查询接口等，均以只读或「脚本草案」形式暴露给 Agent。
  - 模型与向量库：统一模型网关（OpenAI 兼容） + Qdrant/Milvus 向量库 + 关系型数据库（交易/配置数据）。
  - 观测与审计：Prometheus + Grafana + 日志平台 + OpenTelemetry Trace，用于监控所有 AIAgent 调用与关键路径事件。

# 4. 核心技术选型与原因（必填）

> 本节给出「本场景的实际选型结论与原因」，
> `aia-agent-architecture/decisions/` 下的文档仅作为通用技术对比材料。

## 4.1 模型与推理服务

- 选型结论（推荐）：
  - 通过自建 OpenAI 兼容模型网关（参见 0003），对接多家 LLM：
    - 核心分析链路：采用能力和稳定性较高的模型（如 OpenAI GPT-5.x、Claude Sonnet/Opus）；
    - 普通问答链路：采用性价比更高的小模型（如 GPT-5.3 Instant、Qwen3.5-Plus）。
- 选择原因：
  - 允许在不同链路间灵活切换模型，实现「高价值链路高成本模型、低价值链路低成本模型」的混合策略；
  - 模型网关统一处理鉴权、限流、日志和审计，降低业务服务与具体厂商的耦合。
- 考虑过的方案与 tradeoff：
  - 直接厂商 API：接入简单，但多厂商时治理困难、隐性锁定严重；
  - 第三方推理网关（OpenRouter等）：上手快，但合规与控制力不如自建网关。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0003-model-and-inference-service-selection.md`
  - `aia-agent-architecture/decisions/0006-foundation-models-selection.md`

## 4.2 向量数据库与检索方案

- 选型结论（推荐）：
  - 活动/运营/风控分析相关文本与特征：使用 Qdrant 或 Milvus 作为专用向量库；
  - 小规模/POC 阶段：可以先用 PostgreSQL + pgvector，等待数据规模和并发上来后再迁移。
- 选择原因：
  - 需要支持高维嵌入 + 复杂过滤（按活动、用户分群、时间窗口等） + 高并发离线/准实时检索；
  - Qdrant/Milvus 在 RAG + 大规模检索方面生态成熟，更容易扩展到后续更多分析场景。
- 考虑过的方案与 tradeoff：
  - 仅用关系型数据库 + pgvector：简单易用，但在峰值 QPS 和数据规模上存在天花板；
  - 搜索引擎内置向量检索（Elasticsearch/OpenSearch）：可以一站式使用，但向量能力偏弱、调优成本高。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0001-vector-db-selection.md`

## 4.3 Agent 编排框架

- 选型结论（推荐）：
  - 组合方案：
    - 以现有业务平台为基础，自研轻量 orchestrator（基于消息队列/任务系统），负责事件订阅、任务触发和权限治理；
    - 在复杂分析任务内部（如健康检查、风险分析）使用 LangGraph 建模 Agent 流程和状态机。
- 选择原因：
  - 自研 orchestrator 方便与现有监控、风控、配置中心和审批系统深度集成，统一治理高危操作；
  - LangGraph 提供图式编排、checkpoint 与事件流能力，适合表达复杂多步骤分析任务，利于调试与可观测性。
- 考虑过的方案与 tradeoff：
  - 完全自研：自由度最大，但容易重复造轮子、维护成本高；
  - 单纯使用高层框架（CrewAI/AutoGen）：适合 Demo，不适合作为长期编排内核。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0002-agent-orchestration-framework-selection.md`

## 4.4 可观测性与日志方案

- 选型结论（推荐）：
  - 延续现有 Prometheus + Grafana + 日志平台 + OpenTelemetry 的观测体系，单独为 AIAgent 增加：
    - 模型调用、工具调用、关键决策节点的指标与 Trace span；
    - 风控/运营建议的采纳率、成功率、误报率等业务指标。
- 选择原因：
  - 秒杀场景对性能和稳定性要求高，需要端到端观测能力，且要与现有系统统一；
  - LLM/Agent 的行为需要被视为可观测的组件，纳入统一监控体系。
- 相关决策文档：
  - `aia-agent-architecture/decisions/0004-observability-and-logging-selection.md`

## 4.5 其他关键技术

- 会话与分析数据存储：
  - 使用关系型数据库/OLAP（如 PostgreSQL/ClickHouse）记录会话元数据、分析任务结果与复盘报告，支持后续离线分析与模型评估。
- 配置与功能开关：
  - 所有 Agent 相关「自动化程度」「高危操作开关」必须通过集中配置中心管理，支持按环境/活动/用户分群控制。

# 5. 核心组件设计

- 交易核心系统：
  - 负责所有下单/扣库存/支付等强一致性操作，不接受来自 Agent 的直接写操作；
  - 对 AIAgent 暴露只读查询接口（订单/库存/活动配置快照）。
- AIAgent Orchestrator：
  - 基于 LangGraph + 自研 orchestrator，接收来自风控/运营/运维/客服渠道的触发事件，分发至对应 Agent 工作流；
  - 负责记录每次分析任务的上下文、调用链路与结果，以便审计和复盘。
- 分析类 Agents：
  - `HealthCheckAgent`：聚合监控指标、库存与错误日志，输出「健康评分 + 风险点列表」。
  - `RiskAnalysisAgent`：对异常流量/订单段做聚合分析与特征提取，输出风险模式与规则草案。
  - `OpsSuggestionAgent`：结合监控/Release 信息，输出扩容/限流/降级操作建议草案。
  - `ReportAgent`：定期生成活动复盘报告与下次活动配置建议。
- 客服类 Agents：
  - `FAQAgent`：回答活动规则、常见问题等，基于知识库 RAG；
  - `OrderExplainAgent`：基于订单/库存信息，给出用户角度可理解的解释（为什么没抢到、是否有补偿等）。
- 工具层：
  - 监控 API、日志平台 API、配置中心、活动管理系统、风控引擎 API、订单/库存查询 API 等；
  - 对高危工具（配置修改脚本、限流规则变更等）仅允许生成草案，不自动执行。
- 模型网关与向量库：
  - 统一模型网关用于代理所有 LLM 调用，记录调用参数/结果摘要/成本；
  - Qdrant/Milvus 存储活动配置快照、日志段落和历史复盘文档的向量表示，支撑 RAG 与相似分析。
- 观测与审计：
  - 对所有 Agent 任务埋点：请求来源、使用模型、工具调用列表、建议结果与人工采纳情况；
  - 支持按活动、时间窗口、角色查询和回放关键分析任务链路。

# 6. 关键流程说明

## 6.1 实时健康分析流程

1. 定时任务或阈值告警触发 AIAgent Orchestrator；
2. Orchestrator 调用 `HealthCheckAgent`：
   - 通过工具层拉取 Prometheus 指标（QPS、错误率、延迟等）、库存/订单数据和关键日志样本；
3. Agent 基于预设 Prompt 和上下文生成：
   - 健康评分；
   - 风险点列表（如错误率异常、库存消耗过快、某 Region 延迟飙升）；
   - 建议关注的指标或后续调查任务；
4. 结果推送到运营/风控看板和 IM 通知，记录在审计与日志系统中。

## 6.2 异常流量分析流程

1. 风控系统触发异常告警（例如某 IP 段订单异常集中）；
2. Orchestrator 调用 `RiskAnalysisAgent`，传入告警上下文（时间窗口、用户/IP 片段、相关指标等）；
3. Agent 调用日志平台/向量库，对相关时间段请求日志做聚类/相似度分析，识别可能的攻击模式或自动脚本行为；
4. Agent 输出「风险模式描述 + 备选规则草案」，并生成适用于规则引擎的配置草案；
5. 风控人员在控制台审阅/修改后，由传统规则发布流程下发，Agent 不直接执行规则变更。

## 6.3 客服问答与解释流程

1. 用户或客服通过 Web/IM 接入层发起问题（例如「为什么没有抢到这次秒杀？」）；
2. Orchestrator 调用 `OrderExplainAgent`：
   - 调用订单/库存/风控决策日志工具，获取相关信息；
   - 调用知识库向量检索活动规则与系统行为说明；
3. Agent 生成解释性回答（例如「库存 X 秒内售罄、订单排队规则、风控判定」），并标记是否建议人工审核；
4. 回答和使用的证据片段 ID 记录在日志和审计系统中，用于后续投诉追踪。

# 7. 数据与存储设计

- 数据类型与来源：
  - 监控指标：来自现有监控系统（Prometheus 等）；
  - 日志与决策记录：来自日志平台与风控系统；
  - 订单与库存数据：来自交易系统，只读访问；
  - 知识库文档：活动规则、运营手册、历史复盘文档等。
- 存储与索引：
  - 向量库：按活动/时间/类型划分 Collection，存储文档片段、日志片段和复盘摘要的嵌入向量；
  - 关系型数据库：存储分析任务、健康检查结果、风险分析报告和建议的结构化记录；
  - 日志平台：存储详细请求/响应/Trace 日志，支持按 trace_id、活动 ID、用户维度检索。
- 数据生命周期：
  - 监控/日志按公司合规要求保留，支持按活动维度归档；
  - 向量库定期重建索引、清理过期活动数据；
  - 分析报告与复盘文档长期保留，作为运营知识库的一部分。

# 8. 非功能性要求

- 性能与延迟目标：
  - 分析类任务通常是异步/旁路任务，对秒级延迟容忍度较高，但在大促期间应控制在数秒内给出结果；
  - 客服问答在 3 秒内返回结果；
  - 不允许对核心交易路径增加任何同步依赖。
- 可用性与容错：
  - AIAgent 相关服务失败时，交易主链路必须不受影响；
  - 支持关闭 Agent 功能时系统仍可正常运行。
- 安全与合规：
  - Agent 不得接触完整敏感数据（如完整身份证号、完整卡号等），必须通过脱敏接口获取；
  - 所有高危建议（规则变更、配置修改等）必须有审计记录和人工确认步骤。
- 成本控制：
  - 通过模型混用、Prompt 缓存、结果缓存等手段控制大促期间的模型调用成本；
  - 对不同 Agent 和场景配置调用配额与限流策略。

# 9. 风险、边界与取舍

- 已知风险：
  - 模型误判健康/风险程度，可能导致不恰当的运营/风控建议；
  - 对日志/指标依赖不全面时，分析结论可能偏差。
- 边界与不做的事：
  - 不在 AIAgent 中直接做扣库存、转账或修改账户余额等不可逆操作；
  - 不绕过现有合规审查与发布流程直接修改风控规则或系统配置。
- 设计取舍：
  - 更倾向于「提供高质量建议 + 明确证据」，由人执行决策；
  - 在体验与安全之间优先安全与可审计性。

# 10. 实施计划与里程碑（可选）

- 阶段 1：
  - 实现健康检查和活动复盘 Agent，只读访问监控与日志，输出报告与建议。
- 阶段 2：
  - 引入异常流量分析 Agent，与风控系统集成，生成规则草案并接入审批流程。
- 阶段 3：
  - 引入客服解释 Agent，与客服系统集成，开始在部分活动中试运行。
